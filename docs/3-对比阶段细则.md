# 3 - 对比阶段细则

本文档记录"匹配完成后"进入差异比对阶段的全流程，包括调用链、字段分发、每个属性的具体规则、当前实现与设计意图之间的偏差。

来源代码：
- [`server/src/comparators/styleComparator.js`](../server/src/comparators/styleComparator.js)
- [`server/src/comparators/spatialComparator.js`](../server/src/comparators/spatialComparator.js)
- [`server/src/routes/check.js`](../server/src/routes/check.js)（调用方）
- [`client/src/components/DiffReport.vue`](../client/src/components/DiffReport.vue)（前端展示）

## 1. 总流程

`runCheck` 在拿到 `pairs`（已匹配节点对）后，执行两次比对，再汇总差异：

```
matchNodes(...) → { pairs, unmatchedDesign, unmatchedArkui, ... }
        │
        ├─→ compareAll(pairs)                 // 节点内部样式对比 → diffs[]
        │       └─ for each pair: compareStyles(pair) → StyleDiff[]
        │
        └─→ compareSpatialRelations(pairs)    // 已禁用，固定返回 []（间距/对齐偏差不参与报告）
```

最终 `diffs = [...compareAll, ...compareSpatialRelations]`，每条 diff 会被挂上 `designRect / arkuiRect`（供前端高亮），按 `severity` 累加为 `errorCount / warningCount / infoCount`，再参与还原度评分。

> **重要**：不在 `pairs` 里的节点（`unmatchedDesign / unmatchedArkui`）**不会进入比对阶段**，只在末尾以"未匹配"汇总形式出现，不会生成属性级 diff。

## 2. `compareStyles(pair)`：节点内部样式对比

### 2.1 字段分发

`compareStyles` 按 `pair.design.type` / `pair.arkui.type` 决定走哪一组对比：

| 条件 | 调用的 diff 函数 |
|---|---|
| `dn.type === 'text' && an.type === 'text'` | `diffNumber('fontSize')`、`diffFontWeight`、`diffColor('fontColor')`、`diffFontFamily`、`diffBlur`、`diffShadow`、`diffOpacity` |
| `dn.type !== 'text' && an.type !== 'text'` | `diffBackgroundColor`、`diffBorderRadius`、`diffBorder`（内部派生 `borderWidth` + `borderColor`）、`diffOpacity`、`diffBlur`、`diffShadow` |
| 一侧 text、一侧 container | **不对比任何属性**（不会生成 diff） |

> 文本节点 `fontSize` 在 ArkUI 侧 `rawType === 'titlebar'` 时跳过（标题栏字号由系统主题接管）。

### 2.2 容差表

```js
TOLERANCE = {
  fontSize:      0,     // 零容差；分级：≤2 → warning，>2 → error
  lineHeight:    0.5,   // 当前未接入 compareStyles
  letterSpacing: 0.5,   // 当前未接入 compareStyles
  borderRadius:  0,     // 零容差：任一角差 > 0 即报 warning
  padding:       1.0,   // 当前未接入 compareStyles
  opacity:       0.02,
  blur:          2.0,
  shadowRadius:  2.0,
  shadowOffset:  1.0,   // 当前未参与 diff
  colorDelta:    0,     // ΔE 阈值（0 表示严格相等才不报；> 40 升级为 error）
}
```

### 2.3 各对比项详细规则

下面每节都标明**实际产生的 `property` 名称**、触发条件、阈值、严重级别。所有 diff 都通过 `makeDiff` 工厂统一打上 `nodeType / matchType / confidence / iou / topologyScore / regionScore` 等元数据。

---

#### 2.3.1 `fontSize`（文本节点）

- 调用：`diffNumber(diffs, ctx, 'fontSize', ds.fontSize, as_.fontSize, 'dp/vp', 0, '字号', 2)`
- 跳过条件：
  - 任一侧 `fontSize` 为 `null / undefined`
  - ArkUI 侧 `rawType === 'titlebar'`
- 触发：`|d - a| > 0`（任何偏差都报）
- 严重级别：`≤ 2`：`warning`；`> 2`：`error`
- 描述："字号偏差 ±X.Xdp/vp"

#### 2.3.2 `fontWeight`（文本节点）

- 调用：`diffFontWeight(diffs, ctx, ds.fontWeight, as_.fontWeight)`
- 跳过条件：任一侧字重为 `null / undefined`
- 触发：`dv !== av`（严格不等，已经过解析阶段统一归一化为数值）
- 严重级别：始终 `error`
- 描述："字重不匹配"

#### 2.3.3 `fontColor`（文本节点）

- 调用：`diffColor(diffs, ctx, 'fontColor', ds.fontColor, as_.fontColor, '颜色')`
- 跳过条件：
  - 两侧都为空或两侧都等于 `#00000000`
- 单侧缺失：`warning` + "颜色：一侧缺失"
- 触发：`colorDelta(dv, av) > TOLERANCE.colorDelta`，即 `> 0`（严格相等才不报）
- 严重级别：ΔE > 40 为 `error`，否则 `warning`
- 描述："颜色不匹配 ΔE≈N"

#### 2.3.4 `fontFamily`（文本节点）

- 调用：`diffFontFamily(diffs, ctx, ds.fontFamily, as_.fontFamily)`
- 跳过条件：任一侧缺失（**不会**报"一侧缺失"）
- 触发：`!isEquivalentFont(dv, av)`（字体别名表见 `utils/unitUtils.js`）
- 严重级别：`warning`
- 描述："字体不匹配"

#### 2.3.5 `blur`（文本 + 非文本）

- 调用：`diffBlur(diffs, ctx, ds.blur, as_.blur, 'blur', '模糊')`
- 跳过条件：
  - `dv === undefined && av === undefined`
  - `!dv && !av`（两侧都为 0 / falsy）
- 触发：`|d - a| > 2`
- 严重级别：`warning`
- 描述："模糊偏差 X.X"
- 备注：解析阶段区分了 `background blur` 和 `filter blur`，但比对阶段统一合并到 `blur` 一个键，没有按层级输出。

#### 2.3.6 `shadow` / `shadow.radius`（文本 + 非文本）

- 调用：`diffShadow(diffs, ctx, ds.shadow, as_.shadow)`
- 当前只覆盖两类输出：

| 情况 | property | severity | 描述 |
|---|---|---|---|
| 设计有、ArkUI 无 | `shadow` | `warning` | "投影：实现缺失"（设计值 = `radius:N`） |
| 双侧都有，半径差 > 2 | `shadow.radius` | `warning` | "投影模糊半径偏差" |

- 反向（ArkUI 多出来）、颜色差异、偏移量差异**当前都不报**。

#### 2.3.7 `opacity`（文本 + 非文本）

- 调用：`diffOpacity(diffs, ctx, ds.opacity, as_.opacity)`
- 跳过条件：
  - 任一侧不能解析为有限数字
  - 任一侧等于 0（认为是隐藏，不参与比较）
- 触发：`|d - a| > 0.02`
- 严重级别：`warning`
- 描述："透明度偏差"

#### 2.3.8 `borderRadius`（非文本节点）

- 调用：`diffBorderRadius(diffs, ctx, dn, an, ds.borderRadius, as_.borderRadius)`
- 跳过条件：
  - 两侧都没有圆角
  - 双侧 `rawType` 都是 `circle / ellipse`
- 单侧缺失：`warning` + "圆角：一侧缺失"
- 触发：四角中任意一角的**有效圆角**（受 `Math.min(w, h) / 2` 上限裁剪）差 > 0（零容差）
- 严重级别：始终 `warning`
- 描述：`圆角不匹配（topLeft, ...）`，值格式 `12vp`（四角相同）或 `tl/tr/br/bl vp`

#### 2.3.9 `backgroundColor`（非文本节点）

- 调用：`diffBackgroundColor(diffs, ctx, dn, an, ds.backgroundColor, as_.backgroundColor)`
- 判定顺序：

  1. **硬豁免**：任一侧节点 `rawType` ∈ `HARD_IGNORE_BACKGROUND_TYPES = { image, img }` → **无条件 return**，不参与填充对比。
     这类节点的填充语义在 ArkUI 是 `fillColor / objectFit`，不是 `backgroundColor`；强制用 backgroundColor 对比会误报。
  2. **软豁免（按侧）**：计算 `dHasValue / aHasValue`（非空且非透明色 `#00000000`）后，
     - 设计侧节点是软豁免类型 + 设计侧无填充值 → return
     - ArkUI 侧节点是软豁免类型 + ArkUI 侧无填充值 → return
     - 软豁免类型集合：`{ button, path, divider, progress, swiperindicator, icon, video, canvas }`
  3. **symbolglyph 规则**：开发侧 `rawType === 'symbolglyph'` 时，必须两侧均有非透明填充值（`dHasValue && aHasValue`）才继续对比；任一侧缺失或透明 → **return**，不报差异。
     背景：HM Symbol 图标的颜色由字体渲染层控制，设计侧写入 `backgroundColor` 仅作语义记录；若开发侧未显式设置填充色，属正常实现，不应产生误报。
  4. 任一侧是渐变（`linear-gradient(...)`）→ 走 `diffGradient`：
     - 单侧渐变：`warning` + "填充：设计侧为渐变色，开发侧为纯色"（或反之）
     - 双侧渐变：比角度（容差 5°）、比 stops 数量、比每个 stop 的颜色（ΔE > 0）和位置（差 > 2%），任一项不一致 → `warning` + "渐变不匹配"
  5. 纯色比较：
     - 任一侧为空 / 透明（`comparableBackground` 返回 null）→ **不报**，直接忽略
     - 双侧均有实色，`colorDelta > 0` → `warning`（>40 为 `error`），描述 "填充不匹配 ΔE≈N"

- 典型 case 验证（case1）：ArkUI `1792 (Image, praiseBtn)` ↔ Design `14:2238 (BOOLEAN_OPERATION "形状", #99000000)`。
  - 旧规则：设计侧 `dHasValue=true` 绕过 image 豁免 → 报 "填充：一侧缺失"
  - 新规则：ArkUI 侧 rawType=image → 命中硬豁免 → 直接 return，不再产出此 diff

#### 2.3.10 `borderWidth` + `borderColor`（非文本节点）

- 调用：`diffBorder(diffs, ctx, ds.border, as_.border)`
- 内部分两段：

  - **borderWidth**：
    - 两侧 `width` 都为 `null / undefined` → 不报
    - 单侧缺失 → `warning` + "描边宽度：一侧缺失"
    - `|dw - aw| > 0`（零容差）→ `warning`（> 2 为 `error`）+ "描边宽度偏差 X.X"
  - **borderColor**：调用 `diffColor(diffs, ctx, 'borderColor', dv?.color, av?.color, '描边颜色')`，规则与 `fontColor` 完全一致

### 2.4 `compareStyles` 中**当前未启用**的函数

下列函数在 `styleComparator.js` 中存在，但 `compareStyles` 没有调用，所以这些 `property` **不会**进入差异报告：

| 函数 | 预期 property | 现状 |
|---|---|---|
| `diffTextAlign` | `textAlign` | 未挂入 |
| `diffPadding` | `padding` | 未挂入 |
| `diffItemSpacing` | `itemSpacing` | 未挂入 `compareStyles`，需外部手动调用 |
| `diffFontScale` | `fontSize.scale` | 未挂入 |

## 3. `compareSpatialRelations(pairs)`：节点之间的空间关系对比（已禁用）

### 3.1 入口逻辑

```
1. 建立 pairByDesignPath / pairByArkuiPath：以路径为键，方便快速查父
2. 把所有 pair 按"已匹配的父节点"分组：
   - getMatchedParentPair(pair) 要求 design 父和 arkui 父都已匹配
     且映射到的是同一个 parentPair（双向一致）
   - 否则不进入空间比较
3. 对每个组：
   - siblings 按 design.path 字典序排序（`utils/pathOrder.js#comparePaths`，等价 DFS 顺序）
   - len >= 2 → compareSiblingRelations(siblings)
   - len == 1 → compareParentChildRelation(parentPair, siblings[0])
```

### 3.2 `compareSiblingRelations`（兄弟节点之间）

- 仅比较**相邻**两个 sibling（`siblings[i]` 与 `siblings[i+1]`）。
- 主轴判定 `dominantAxis(left.design.rect, right.design.rect)`：横向交叠率 vs 纵向交叠率，取大者作为主轴。
- 计算两侧 metrics：
  - 主轴 `gap = max(0, b.start - a.end)`
  - 交叉轴 `crossOffset = |center(a) - center(b)|`
- 触发条件 1：`gapDelta > 0.015` 或 `alignDelta > 0.02`（单位：归一化 0-1）
  - property: `spacing.horizontal` 或 `spacing.vertical`
  - severity: `gapDelta > 0.03` → `warning`，否则 `info`
  - 描述："兄弟节点横向/纵向间距偏差 ±X.XXvp"
- ~~触发条件 2：alignment 偏差~~（已移除，对齐偏差不参与对比报告）

### 3.3 `compareParentChildRelation`（父子单节点）

- **仅当某父节点下"已匹配的兄弟数量 == 1"时**才走父子比较。
- 计算父子四边内缩 `{ left, top, right, bottom }`。
- `insetDelta = max(|d.* - a.*|)`
- 触发：`insetDelta > 0.02`（≤ 0.02 时返回 null，不报）
- severity：`> 0.05` → `warning`；其他 → `info`
- property：`spacing.parent-child`
- 描述："父子层内边距/对齐偏差 ±X.XXvp"

### 3.4 输出 diff 的特殊字段

`makeSpatialDiff` 比 `makeDiff` 多带：

- `relatedDesignNodeId / relatedArkuiNodeId`（参与关系的另一个节点 ID）
- `relatedDesignName / relatedArkuiName`
- `relationKind`：`sibling` 或 `parent-child`
- `relationAxis`：`horizontal` / `vertical` / `mixed`
- `relationRects`：`{ design: [...], arkui: [...], axis }` ——前端用来双层高亮关系区域

## 4. 严重级别汇总（routes/check.js）

```
errorCount   = diffs.filter(d => d.severity === 'error').length
warningCount = diffs.filter(d => d.severity === 'warning').length
infoCount    = diffs.filter(d => d.severity === 'info').length
```

`info` 级别的差异（基本来自 `compareSpatialRelations` 的对齐/小间距偏差）**不参与扣分**：

```
penalty = (errorCount * 3 + warningCount * 1) / pairs.length
coveragePenalty = (1 - matchCoverage) * 25
lowConfidencePenalty = lowConfidencePairs / pairs.length * 10
score = max(0, min(100, round(100 - penalty*10 - coveragePenalty - lowConfidencePenalty)))
```

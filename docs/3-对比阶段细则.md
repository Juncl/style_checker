# 3 - 对比阶段细则

本文档记录"匹配完成后"进入差异比对阶段的全流程，包括调用链、字段分发、每个属性的具体规则、当前实现与设计意图之间的偏差。

来源代码：
- [`server/src/comparators/styleComparator.js`](../server/src/comparators/styleComparator.js)
- [`server/src/comparators/spatialComparator.js`](../server/src/comparators/spatialComparator.js)
- [`server/src/routes/check.js`](../server/src/routes/check.js)（调用方）
- [`client/src/components/DiffReport.vue`](../client/src/components/DiffReport.vue)（前端展示）

---

## 1. 总流程

`runCheck` 在拿到 `pairs`（已匹配节点对）后，执行两次比对，再汇总差异：

```
matchNodes(...) → { pairs, unmatchedDesign, unmatchedArkui, ... }
        │
        ├─→ compareAll(pairs)                 // 节点内部样式对比 → diffs[]
        │       └─ for each pair: compareStyles(pair) → StyleDiff[]
        │
        └─→ compareSpatialRelations(pairs)    // 相邻节点间距比对 → spacing.top / spacing.left diffs
```

最终 `diffs = [...compareAll, ...compareSpatialRelations]`，每条 diff 会被挂上 `designRect / arkuiRect`（供前端高亮），按 `severity` 累加为 `errorCount / warningCount / infoCount`，再参与还原度评分。

> **重要**：不在 `pairs` 里的节点（`unmatchedDesign / unmatchedArkui`）**不会进入比对阶段**，只在末尾以"未匹配"汇总形式出现，不会生成属性级 diff。

---

## 2. `compareStyles(pair)`：节点内部样式对比

### 2.0 全局豁免：一侧缺失不进入报告

**规则**：`compareAll` 在汇总各 pair 的 diffs 时，统一过滤掉任一侧值为 `'—'` 的 diff 条目。各 diff 函数内部仍会生成这类 diff 对象（原有豁免条件不变），但最终不进入差异报告、不参与评分。

- 触发条件：`diff.designValue === '—' || diff.arkuiValue === '—'`，且 `diff.property !== 'blur'`
- 覆盖场景：`fontColor/borderColor` 一侧缺失、`borderRadius` 一侧缺失、`borderWidth` 一侧缺失、`shadow` 实现缺失（设计有 ArkUI 无）、`itemSpacing` 实现缺失
- **例外**：`blur`（模糊）不受此规则约束，"模糊：实现缺失"仍正常进入报告
- **设计侧悬浮框**：对方侧（设计侧）有值时，该属性行在设计侧 inspector 中仍正常显示，只是不标红

### 2.1 字段分发

`compareStyles` 按 `pair.design.type` / `pair.arkui.type` 决定走哪一组对比：

| 条件 | 调用的 diff 函数 |
|---|---|
| `dn.type === 'text' && an.type === 'text'` | `diffNumber('fontSize')`、`diffFontWeight`、`diffColor('fontColor')`、`diffFontFamily`、`diffBlur`、`diffShadow`、`diffOpacity` |
| `dn.type !== 'text' && an.type !== 'text'` | `diffBackgroundColor`、`diffBorderRadius`、`diffBorder`（内部派生 `borderWidth` + `borderColor`）、`diffOpacity`、`diffBlur`、`diffShadow` |
| 一侧 text、一侧 container | **不对比任何属性**（不会生成 diff） |

> 文本节点 `fontSize` 在 ArkUI 侧 `rawType === 'titlebar'` 时跳过（标题栏字号由系统主题接管）。

### 2.2 容差表

```js
TOLERANCE = {
  fontSize:      0,     // 零容差；分级：≤2 → warning，>2 → error
  lineHeight:    0.5,   // 当前未接入 compareStyles
  letterSpacing: 0.5,   // 当前未接入 compareStyles
  borderRadius:  0,     // 零容差：任一角差 > 0 即报 warning
  padding:       1.0,   // 当前未接入 compareStyles
  opacity:       0.02,
  blur:          0,     // 零容差：任何偏差都报
  shadowRadius:  0,     // 零容差：任何偏差都报
  shadowOffset:  0,     // 零容差：任何偏差都报
  colorDelta:    0,     // ΔE 阈值（0 表示严格相等才不报；> 40 升级为 error）
}
```

### 2.3 各对比项详细规则

下面每节都标明**实际产生的 `property` 名称**、触发条件、阈值、严重级别。所有 diff 都通过 `makeDiff` 工厂统一打上 `nodeType / matchType / confidence / iou / topologyScore / regionScore` 等元数据。

---

#### 2.3.1 `fontSize`（文本节点）

- 调用：`diffNumber(diffs, ctx, 'fontSize', ds.fontSize, as_.fontSize, 'dp/vp', 0, '字号', 2)`
- 跳过条件：
  - 任一侧 `fontSize` 为 `null / undefined`
  - ArkUI 侧 `rawType === 'titlebar'`
- 触发：`|d - a| > 0`（任何偏差都报）
- 严重级别：`≤ 2`：`warning`；`> 2`：`error`
- 描述："字号偏差 ±X.Xdp/vp"

#### 2.3.2 `fontWeight`（文本节点）

- 调用：`diffFontWeight(diffs, ctx, ds.fontWeight, as_.fontWeight)`
- 跳过条件：任一侧字重为 `null / undefined`
- 触发：`dv !== av`（严格不等，已经过解析阶段统一归一化为数值）
- 严重级别：始终 `error`
- 描述："字重不匹配"

#### 2.3.3 `fontColor`（文本节点）

- 调用：`diffColor(diffs, ctx, 'fontColor', ds.fontColor, as_.fontColor, '颜色')`
- 跳过条件：
  - 两侧都为空或两侧都等于 `#00000000`
- 单侧缺失：函数内部生成 diff，但被全局过滤器（§2.0）拦截，**不进入报告**
- 触发：`colorDelta(dv, av) > TOLERANCE.colorDelta`，即 `> 0`（严格相等才不报）
- 严重级别：ΔE > 40 为 `error`，否则 `warning`
- 描述："颜色不匹配 ΔE≈N"

#### 2.3.4 `fontFamily`（文本节点）

- 调用：`diffFontFamily(diffs, ctx, ds.fontFamily, as_.fontFamily)`
- 跳过条件：
  - 任一侧缺失（**不会**报"一侧缺失"）
  - **平台为 `web`**（web 侧采集到的 `fontFamily` 无法准确反映实际渲染字体，跳过对比）
- 触发：`!isEquivalentFont(dv, av)`（字体别名表见 `utils/unitUtils.js`）
- 严重级别：`warning`
- 描述："字体不匹配"

#### 2.3.5 `blur`（文本 + 非文本）

- 调用：`diffBlur(diffs, ctx, ds.blur, as_.blur, 'blur', '模糊')`
- **字段格式**：解析阶段已将背景模糊和高斯模糊统一为字符串 `"背景模糊 Npx"` / `"高斯模糊 Npx"`，背景模糊优先（覆盖高斯）。
- 跳过条件：`!dv && !av`（两侧都为 falsy）
- 判定顺序：
  1. design 有、ArkUI 无 → `warning` "模糊：实现缺失"（**不受全局过滤器约束，仍进入报告**）
  2. ArkUI 有、design 无 → 不报（开发侧多加模糊不属于问题）
  3. 两侧都有 → 解析类型和数值：
     - 格式解析失败 → `warning` "模糊格式异常"
     - 类型不同（背景模糊 vs 高斯模糊）→ `warning` "模糊类型不匹配"
     - 数值不等（零容差）→ `warning` "模糊偏差 X.Xpx"
- 严重级别：`warning`

#### 2.3.6 `shadow` / `shadow.radius` / `shadow.offset`（文本 + 非文本）

- 调用：`diffShadow(diffs, ctx, ds.shadow, as_.shadow)`
- **字段格式**：解析阶段已将阴影统一为字符串 `"外阴影 #color Rpx X:x, Y:y"` / `"内阴影 ..."；needShadow 过滤（radius > 0 且偏移量不全为 0 且颜色非透明）。
- 跳过条件：两侧都为 falsy
- 判定顺序：

| 情况 | property | severity | 描述 |
|---|---|---|---|
| 设计有、ArkUI 无 | `shadow` | — | 被全局过滤器（§2.0）拦截，**不进入报告** |
| ArkUI 有、设计无 | — | — | 不报 |
| 格式解析失败 | `shadow` | `warning` | "投影格式异常" |
| 两侧有差异（类型/radius/offset 任一不等） | `shadow` | `warning` | "投影不匹配：XXX，XXX"（合并描述） |

- 多项不一致时合并为一条 diff，描述中列出所有问题（如 "投影不匹配：radius偏差 4.0px，偏移偏差"）。
- 颜色差异**当前不报**（阴影颜色 alpha 差异噪音较多，暂不纳入）。

#### 2.3.7 `opacity`（文本 + 非文本）

- 调用：`diffOpacity(diffs, ctx, ds.opacity, as_.opacity)`
- 跳过条件：
  - 任一侧不能解析为有限数字
  - 任一侧等于 0（认为是隐藏，不参与比较）
- 触发：`|d - a| > 0.02`
- 严重级别：`warning`
- 描述："透明度偏差"

#### 2.3.8 `borderRadius`（非文本节点）

- 调用：`diffBorderRadius(diffs, ctx, dn, an, ds.borderRadius, as_.borderRadius)`
- 跳过条件：
  - 两侧都没有圆角
  - 开发侧 `rawType === 'circle'` || 设计侧 `rawType === 'ellipse'`
  - 开发侧 `rawType === 'symbolglyph'`（HM Symbol 图标的圆角由字体渲染决定，不参与对比）
- 单侧缺失：函数内部生成 diff，被全局过滤器（§2.0）拦截，**不进入报告**
- 触发：四角中任意一角的**有效圆角**（受 `Math.min(w, h) / 2` 上限裁剪）差 > 0（零容差）
- 严重级别：始终 `warning`
- 描述：`圆角不匹配（topLeft, ...）`，值格式 `12vp`（四角相同）或 `tl/tr/br/bl vp`

#### 2.3.9 `backgroundColor`（非文本节点）

- 调用：`diffBackgroundColor(diffs, ctx, dn, an, ds.backgroundColor, as_.backgroundColor)`
- 判定顺序：

  1. **硬豁免**：任一侧节点 `rawType` ∈ `HARD_IGNORE_BACKGROUND_TYPES = { image, img, canvas }` → **无条件 return**，不参与填充对比。
     - `image / img`：填充语义在 ArkUI 是 `fillColor / objectFit`，不是 `backgroundColor`；强制用 backgroundColor 对比会误报。
     - `canvas`：canvas 背景色无展示意义，对比无意义。
  2. **软豁免（按侧）**：计算 `dHasValue / aHasValue`（非空且非透明色 `#00000000`）后，
     - 设计侧节点是软豁免类型 + 设计侧无填充值 → return
     - ArkUI 侧节点是软豁免类型 + ArkUI 侧无填充值 → return
     - 软豁免类型集合：`{ button, path, divider, progress, swiperindicator, icon, video }`
   3. **symbolglyph 规则**：开发侧 `rawType === 'symbolglyph'` 时，必须两侧均有非透明填充值（`dHasValue && aHasValue`）才继续对比；任一侧缺失或透明 → **return**，不报差异。
      背景：HM Symbol 图标的颜色由字体渲染层控制，设计侧写入 `backgroundColor` 仅作语义记录；若开发侧未显式设置填充色，属正常实现，不应产生误报。
   4. **单侧缺失忽略**：不论纯色还是渐变色，任一侧无填充值或为透明色 → **return**，不产生差异报告（不再报"填充：一侧缺失"或"一侧渐变一侧纯色"）
   5. 两侧均有值，任一侧是渐变（`linear-gradient(...)`）→ 走 `diffGradient`：
      - 单侧渐变（另一侧是纯色）：`warning` + "填充：设计侧为渐变色，开发侧为纯色"（或反之）
      - 双侧渐变：比角度（容差 5°）、比 stops 数量、比每个 stop 的颜色（ΔE > 0）和位置（差 > 2%），任一项不一致 → `warning` + "渐变不匹配"
   6. 纯色比较：
      - 双侧均为实色，`colorDelta > 0` → `warning`（>40 为 `error`），描述 "填充不匹配 ΔE≈N"

- 典型 case 验证（case1）：ArkUI `1792 (Image, praiseBtn)` ↔ Design `14:2238 (BOOLEAN_OPERATION "形状", #99000000)`。
  - 旧规则：设计侧 `dHasValue=true` 绕过 image 豁免 → 报 "填充：一侧缺失"
  - 新规则：ArkUI 侧 rawType=image → 命中硬豁免 → 直接 return，不再产出此 diff

#### 2.3.10 `borderWidth` + `borderColor`（非文本节点）

- 调用：`diffBorder(diffs, ctx, ds.border, as_.border)`
- 内部分两段：

  - **borderWidth**：
    - 两侧 `width` 都为 `null / undefined` → 不报
    - 单侧缺失 → 被全局过滤器（§2.0）拦截，**不进入报告**
    - `|dw - aw| > 0`（零容差）→ `warning`（> 2 为 `error`）+ "描边宽度偏差 X.X"
  - **borderColor**：调用 `diffColor(diffs, ctx, 'borderColor', dv?.color, av?.color, '描边颜色')`，规则与 `fontColor` 完全一致
    - 豁免：设计侧 `border.color` 无值（null/undefined）时直接跳过，不对比描边颜色

## 3. `compareSpatialRelations(pairs)`：相邻节点间距对比

代码位置：[`server/src/comparators/spatialComparator.js`](../server/src/comparators/spatialComparator.js)

间距对比不依赖父节点分组，而是基于**几何邻居关系**：对每个已匹配节点，找到其最近的"上邻居"和"左邻居"，分别在设计侧和开发侧计算间距，对比差值。

### 3.1 算法流程

```
1. 为每个已匹配节点计算邻居关系（getNeighbor）：
   - left / right：在节点高度范围内，紧贴左侧 / 右侧的节点
   - top / bottom：在节点宽度范围内，紧贴上方 / 下方的节点
   - parentLay：完全包含该节点的父节点
2. 对每个匹配对 (arkuiNode ↔ designNode)：
   a. 找 arkuiNode 最近的 "上邻居" hmTopNode
      优先选 top 邻居中 y+h 最大者；若无则从 parentLay 中取 y 最大者
   b. 通过 hmIdMap 找到 hmTopNode 匹配的设计侧节点 deTopNode
   c. 若 deTopNode 存在，分别计算两侧上间距：
      distance = 本节点.y - 邻居.y（或 - 邻居.h，视父子关系而定）
   d. 同法找 "左邻居" 比较左间距
3. 过滤异常：
   - 间距过大（超出画布 1/3）→ 忽略
   - 差值悬殊（minD > 20 时 diff >= 2*minD；minD 很小时 diff >= 10*minD）→ 忽略
4. diffDistance > SPACE_ERR_RANGE（3vp）时输出 StyleDiff
```

### 3.2 邻居判定规则

| 方向 | 条件 | 排序选最近 |
|---|---|---|
| top | 宽度有交集且 n2 底部 ≤ n1 顶部 | y+h 最大者 |
| left | 高度有交集且 n2 右侧 ≤ n1 左侧 | x+w 最大者 |
| parentLay（含） | n2 完全包含 n1 | y 最大（上）或 x 最大（左） |

存在 parentLay 时优先选父节点作为参考（`rel = 'c'`），否则用兄弟节点（`rel = 'd'`）。

### 3.3 展示距离与判断距离

- **展示距离**（`displayDistance`）：使用 `node.size`（设计侧原始 dp）/ `node.rect`（开发侧 vp），供前端展示
- **判断距离**（`distance`）：统一用 `node.rect`（vp 维度）计算差值过滤
- 允许误差：`diffDistance <= SPACE_ERR_RANGE（3vp）` 时视为一致，不报差异

### 3.4 输出 diff 字段

`toStyleDiff` 生成的 `StyleDiff` 包含：

| 字段 | 说明 |
|---|---|
| `property` | `spacing.top` 或 `spacing.left` |
| `severity` | 固定 `warning`（低置信一律 warning） |
| `confidence` | 固定 `low` |
| `designValue / arkuiValue` | 两侧间距展示值（string） |
| `spaceId / designSpaceId` | 间距关系唯一 ID：`{anchorId}-{nodeId}-{x\|y}` |
| `relatedDesignNodeId / relatedArkuiNodeId` | 间距参考的邻居节点 ID |
| `relationKind` | `sibling`（兄弟）或 `parent-child`（父子含） |
| `relationAxis` | `vertical` 或 `horizontal` |
| `relationRects` | `{ design: [anchor.rect, node.rect], arkui: [anchor.rect, node.rect], axis }` |

### 3.5 间距矩形

`spaceRectVertical` / `spaceRectHorizontal` 构造间距矩形（前端高亮间距条用）：

- 上间距矩形：从 `anchorRect.bottom`（或 `anchorRect.top`，父包含时）到 `selfRect.top`
- 左间距矩形：从 `anchorRect.right`（或 `anchorRect.left`，父包含时）到 `selfRect.left`

## 4. 严重级别汇总（routes/check.js）

```
errorCount   = diffs.filter(d => d.severity === 'error').length
warningCount = diffs.filter(d => d.severity === 'warning').length
infoCount    = diffs.filter(d => d.severity === 'info').length
```

`info` 级别的差异（来自部分旧版 `compareSpatialRelations` 输出的对齐/小间距偏差）**不参与扣分**；当前实现中 `compareSpatialRelations` 固定输出 `warning` 级别：

```
penalty = (errorCount * 3 + warningCount * 1) / pairs.length
coveragePenalty = (1 - matchCoverage) * 25
lowConfidencePenalty = lowConfidencePairs / pairs.length * 10
score = max(0, min(100, round(100 - penalty*10 - coveragePenalty - lowConfidencePenalty)))
```

# 节点匹配方案说明

> 适用范围：设计侧节点与开发侧节点的一对一匹配、可视化差异定位、问题回溯与后续优化。

## 1. 目标

节点匹配的目标不是简单“找同名节点”，而是在设计稿与开发渲染结果之间，尽可能稳定地找到语义、位置、样式和视觉都一致的节点对。

这套方案重点解决四类问题：

1. 文本内容不完全一致，但属于同一业务槽位。
2. 图形、图标、插画等非文本图层，需要依靠视觉特征和几何特征辅助匹配。
3. 大量布局容器、空节点、不可见节点必须过滤掉，避免污染结果。
4. 匹配必须全局一对一，不能出现多个开发节点对应同一个设计节点。

---

## 2. 输入与中间结构

系统会先把设计侧和开发侧分别解析成统一节点结构，再进入匹配阶段。

统一节点通常包含这些信息：

- `id`：节点唯一标识
- `type`：`text / container`
- `rawType`：保留原始组件类型，供几何、样式和特殊规则判断
- `name`：节点名
- `textContent`：文本内容
- `rect`、`normRect`：原始坐标与归一化坐标
- `style`：字体、颜色、圆角、背景、透明度等样式
- `path`：节点层级路径
- `paintIndex`：绘制顺序
- `pixelVisibility`：像素可见性分析结果
- `visualOccluded`：是否被视觉上遮挡

归一化坐标用于跨画布比较，匹配时会同时参考几何距离、排序关系和局部拓扑。

---

## 3. 节点预处理与过滤

在真正匹配之前，会先做一轮节点清洗。这个阶段的核心目标是去掉“根本不应该参与比较”的节点。

当前生效的过滤规则包括：

- `opacity = 0` 的节点不参与比较。
- 文本内容为空的节点不参与比较。
- 开发侧 `Blank` 节点直接过滤，不进入候选池，也不进入树选择。
- 设计侧和开发侧的隐藏框架祖先节点会被谨慎处理，不能只凭树结构直接判死。
- 视觉上已被遮挡的节点会被标记为不可匹配。
- `#illustration`、`ic_` 开头的图层按语义整体处理，子节点不再拆分参与匹配。

这一步很关键，因为很多误配不是“算法不够聪明”，而是“候选池里混进了不该出现的节点”。

---

## 4. 匹配主流程

当前的匹配顺序是一个分层递进的过程，优先高置信规则，后面再用低置信兜底。

### 4.1 文本精确匹配

优先处理完全同文的文本节点。

规则重点：

- 只比较可用文本节点。
- 同文本优先看位置（中心点距离）、字号/字重/字色、可见性；文本节点的匹配不依赖 IoU（面积重叠）。
- 对短单位、短数字、时间、日期、星期等高重复文本，会加更严格的位置门槛。
- 视觉不可见或弱可见文本，不会被宽松救援逻辑轻易拉进来。
- **强锚点**（y 距离 < 0.05 且样式得分 ≥ 0.78）会进入锚点池，用于引导后续容器节点的匹配；阈值有意收紧，宁缺毋滥。

### 4.2 动态文本槽位

用于处理设计稿与开发数据内容不同，但语义角色一致的场景，例如：

- 时间
- 星期
- 数值
- 单位
- 标题槽位

这类匹配不会只看文本是否相等，而是结合：

- 文本类型
- 语义相似度
- 行内位置
- 局部拓扑关系

### 4.3 区域内最优文本匹配

如果一个视觉区域内存在多个文本节点，会在同一局部区域里做全局最优分配。

这一步用来解决：

- 列表项串位
- 卡片标题与副标题混配
- 同风格重复文本被错配

这里采用的是局部图内的最优组合，而不是单点贪心。

### 4.4 强锚点拓扑推断

当部分节点已经高置信匹配成功后，会把这些节点当作强锚点，进一步推断周围弱节点的对应关系。

这一步特别适合：

- 卡片内部的辅助文本
- 按布局顺序排列的多个标签
- 周边位置稳定、语义相近但文本不同的节点

### 4.5 文本位置回退

当文本内容不一致，但位置结构很接近时，会允许位置回退匹配。

这个通道是”补洞”用的，不能喧宾夺主，匹配得分由以下因素共同决定：

- y 轴中心点距离（权重最高）
- x 轴中心点距离
- 语义相似度（内容）
- 字号、字重、字色（textStyleSimilarity）

**不使用 IoU**：文本节点的尺寸（高度、宽度）不纳入匹配判断，避免因两侧渲染差异导致误判。

### 4.6 几何匹配

对所有非文本节点，也就是统一成 `container` 的节点，几何匹配仍然是基础。

主要参考：

- IoU
- 宽高比
- 尺寸比例
- 中心点距离
- 局部区域相对位置

其中，非文本图层的宽高比已经作为一个明确规则纳入匹配判断。

对于非文本节点，当前不再采用“边扫描边占位”的贪心方式，而是先把所有候选 pair 都算出来，再统一做全局一对一选择。
也就是说，所有 `container` 节点会先进入候选池，再由最终的全局选择阶段决定谁真正保留。
这样可以避免一个早到但并不最优的候选先占住资源，导致后面更合适的候选失去机会。

### 4.7 空间担保匹配

这是一种特殊的几何匹配，专用于处理因**画布宽度差异导致 IoU=0** 但位置关系明确的小型容器/图标节点。

**触发场景**：

- 设计画布和开发画布宽度不同（如 360dp vs 376vp），导致相同的 UI 元素在归一化后的 x 坐标有系统偏差
- 候选节点本身尺寸接近、y 轴对齐良好，但 x 轴完全不重叠，IoU=0
- 却被**已匹配的左右两对担保节点从两侧夹住**，方向一致

**匹配规则**：

1. 找出该候选节点的左右**担保对**（已经在 Pass 1~6 确认匹配的节点对）
2. 验证两个担保对本身在同一行：
   - Design 侧 y 中心距离 ≤ 0.06
   - ArkUI 侧 y 中心距离 ≤ 0.06
3. 验证候选节点也在这一行，且被两个担保对从左右"夹住"：
   - Design 左担保的右边界 ≤ 候选设计节点的 x 中心
   - Design 右担保的左边界 ≥ 候选设计节点的 x 中心
   - ArkUI 左担保的右边界 ≤ 候选 ArkUI 节点的 x 中心
   - ArkUI 右担保的左边界 ≥ 候选 ArkUI 节点的 x 中心
4. 尺寸比例一致（w、h 的 sizeRatio ≥ 0.5）
5. 候选节点与设计侧候选的 y 距离 ≤ 0.05（直接距离约束，防止跨行误配）

**置信度**：

取左右两个担保对中**置信度更低的那个**。例如：
- 左担保 = medium，右担保 = high → 结果 = medium
- 两个担保都是 medium → 结果 = medium

这样可以确保只有当两侧都有相当置信度的担保时，才会确认这类空间位置明确但几何重叠不佳的配对。

**例子**：

```
设计行：[15:4680 文本 ——————] [15:4678 图标] [—— 15:4672 按钮]
                    左担保对              目标对           右担保对

开发行：[5393 文本 ——————————] [5396 图标]    [——— 5397 按钮]
                   左担保对              目标对          右担保对
```

5396 ↔ 15:4678 的 x 坐标实际上不重叠（IoU=0），但因为两个担保对（左：5393↔15:4680、右：5397↔15:4672）在两侧对齐，可以空间担保这对匹配，置信度取 min(担保对置信度)。

### 4.8 视觉匹配

对于原始类型本身就带视觉语义的 `container` 节点，除了几何信息，还会补充视觉特征，例如：

- 裁剪图像相似度
- 色彩直方图
- 边缘与轮廓特征
- 图标/插画的整体视觉轮廓
- 文本渲染粗细估计

这部分主要用于补足：

- fontWeight 误差
- 图标识别
- 插画整体识别
- 圆角与形状视觉效果

### 4.9 低置信兜底

最后才会进入低置信兜底。

兜底不是“什么都能配”，仍然会要求：

- 文本类型一致
- 语义类型一致
- 距离足够近
- 一对一约束不被破坏

这一步的目标是尽量补上漏配，而不是扩大误配。

---

## 5. 一对一约束与优先级

节点匹配只能一对一。

当前策略是：

1. 先收集所有候选配对。
2. 每条候选配对计算置信度。
3. 在全局层面做冲突消解。
4. 优先保留置信度更高的配对。
5. 如果两个非文本候选的分数相同，会优先保留双方背景状态一致的 pair，也就是”都有背景色”或”都没有背景色”的那一类。

冲突消解的排序规则：置信度 → matchType 优先级 → topologyScore → IoU（**仅容器 pair 参与，文本 pair 不使用 IoU 排序**）。

这样可以避免：

- 一个设计图层被多个开发图层同时命中
- 一个开发节点被多条规则重复占用
- 低置信规则抢占高置信规则的结果

对于文本节点，这种一对一消解主要是在同文本、同槽位、多候选之间做去重；对于非文本节点，它更关键，因为非文本候选需要先全量评估，再统一决定最终归属。

---

## 6. 间距、对齐与局部拓扑

间距与对齐不是单节点属性，而是两个图层之间的关系。

因此，算法会优先看：

- 同父兄弟节点之间的间距
- 父子层级内的对齐
- 邻近节点的顺序关系
- 局部拓扑结构

这类规则特别适合：

- 列表
- 表单
- 卡片信息块
- 左图右文、上图下文的布局

在 UI 上，间距问题的高亮也应该直接指向两个图层之间的空隙，而不是只标单个节点。

---

## 7. 可见性与不可见节点处理

不可见节点不能只看一个属性，要综合判断。

当前比较重要的规则是：

- `opacity = 0` 直接排除。
- 文本像素可见性过低时，不进入宽松救援通道。
- 已被更高层视觉图层完全覆盖的节点，应视为不可匹配。
- 设计侧和开发侧都要统一处理，不要一边宽松一边严格。

这里还要注意一个边界：

- “视觉不可见”不等于“树结构存在”。
- “层级存在”也不等于“应该参与匹配”。

所以可见性最好由像素、绘制顺序、遮挡关系共同决定。

---

## 8. 现阶段的经验规则

这套方案在实际修正中，已经沉淀出一些比较稳定的经验规则：

- 空文本不比。
- 透明节点不比。
- `Blank` 不选也不匹配。
- `#illustration`、`ic_` 图层按整体处理。
- 短数字、时间、星期、单位要加严位置门槛。
- 弱可见文本不能靠语义分数硬救。
- 图标、插画、圆点这类节点要结合视觉特征和几何特征。
- 非文本节点的宽高比可以作为有效约束。

---

## 9. 后续可继续增强的方向

当前方案已经能覆盖大部分 case，但还有几个值得继续推进的方向：

1. 细化覆盖遮挡判断，只在同父、同栈层、明确后绘制覆盖时才排除。
2. 增强局部拓扑网络，让强锚点周边推断更稳。
3. 进一步补充图像侧特征，提升图标和插画类节点的识别率。
4. 按节点角色区分文本救援策略，避免标题、副标题、说明文本互相串配。
5. 对间距与对齐问题做专门的关系型输出，便于 UI 高亮两个图层之间的关系。

---

## 10. 验证方式

每次调整节点匹配规则后，建议至少做三类验证：

- `npm run golden`
- `npm run stats`
- 典型 case 的定点回归检查

这样可以同时看：

- 总体召回是否下降
- 一对一约束是否被破坏
- 具体误配是否被修正

---

## 11. 小结

节点匹配的本质，是把“文本、视觉、几何、拓扑、可见性”合成一个有优先级的决策过程。

最稳的做法不是单一规则，而是：

- 先做严格过滤
- 再做高置信精确匹配
- 再做局部结构推断
- 最后才用低置信兜底

这样既能提升匹配率，也能尽量控制误配范围。

## 一、当前项目关键技术实现

### 1.1 整体架构

```
输入：design.json + arkui.json + (可选) design.png + arkui.png
         ↓
    Parser（解析 + 归一化）
         ↓
    Matcher（8-Pass 节点匹配）
         ↓
    Comparator（逐属性样式比对）
         ↓
    JSON 报告 → Vue 3 前端渲染
```

- **前端**：Vue 3 + Element Plus + Vite，三栏布局（左侧控制面板 / 中间双图对比 / 右侧差异报告+节点树）
- **后端**：Node.js + Express，纯同步计算，无数据库，文件通过 multer 内存存储处理

### 1.2 解析层

| 模块 | 关键处理 |
|------|---------|
| DesignParser | Figma 扁平节点数组 → 树形结构（用 `path` 字段还原），颜色格式 `#RRGGBBaa` → `#AARRGGBB` |
| ArkUIParser | ArkUI Inspector 递归 JSON 树遍历，过滤框架内部节点（visibility=None、rect=0、opacity=0），`$rect` 物理像素字符串解析为 vp 坐标 |
| 单位归一化 | x/y/w/h 统一归一化到 `[0,1]` 范围，字重枚举（FontWeight.Medium）→ 数值（500），字体族别名统一 |

**格式转换规则（部分）**：

| 属性 | Design 格式 | ArkUI 格式 |
|-----|------------|-----------|
| 颜色 | `#RRGGBBaa`（alpha 在末尾） | `#AARRGGBB`（alpha 在首位） |
| 字重 | 数值 400/500/700 | 枚举 FontWeight.Normal/Medium/Bold |
| 圆角 | `[tl, tr, br, bl]` dp 数组 | `{topLeft, topRight, ...}` vp 字符串 |
| 内边距 | `[top, right, bottom, left]` dp | `"N.00vp"` 或 JSON 字符串 |

### 1.3 节点匹配算法（8-Pass）

| Pass | 策略 | 目标节点 | 阈值 |
|------|------|---------|------|
| 1 | 文本内容精确匹配 + 样式相似度加权选最优 | text | y 距离 < 0.14（动态） |
| 2 | 区域内文本全局最优分配（≤18节点用DP，否则贪心） | text | score ≥ 0.58 |
| 2.5 | 文本语义角色匹配（title/subtitle/body） | text | score ≥ 0.85 |
| 3 | 强锚点周边拓扑关系匹配（相对位置向量） | 所有类型 | topologyScore > 0.58 |
| 4 | 文本位置回退匹配（处理 Mock 数据不一致） | text | score > 0.35 |
| 5 | 几何 IoU 匹配 | container | 有装饰 > 0.40，纯布局 > 0.60 |
| 6 | 渐变节点几何匹配 | 含 gradient 节点 | IoU > 0.30 |
| 7 | shape/image/other 几何匹配 | shape/image/other | IoU > 0.55 |
| 8 | 低置信度兜底（Rescue） | 剩余所有 | IoU > 0.25 |

**关键子机制**：
- **区域分割（Region Segmentation）**：按 y 坐标间距将节点切为"楼层"，两侧区域互相评分配对，将候选池缩小到同区域内，降低错匹配率
- **视觉遮挡标注**：识别 ArkUI 侧被不透明节点覆盖的文本，排除不可见节点参与匹配
- **双向匹配**：支持 `arkui-first`（以 ArkUI 为主遍历）和 `design-first` 两种方向，取并集去重

### 1.4 样式比对层

逐属性对比并计算差异严重度：

| 属性组 | 属性 | 容差 |
|-------|------|------|
| 文字类 | fontSize / fontWeight / fontColor / fontFamily / textAlign / lineHeight | fontSize ±0.5vp |
| 容器类 | backgroundColor / borderRadius / padding / margin / opacity | 间距 ±1vp |
| 效果类 | blur(backdropBlur) / shadow(radius/color/offset) | 模糊 ±2vp |

严重度规则：`error`（明显偏差）/ `warning`（轻微差异）/ `info`（可能正常的差异）

### 1.5 评分系统

```
score = 100
      - (errorCount×3 + warningCount×1) / matchedPairs × 10   ← 差异扣分
      - (1 - matchCoverage) × 25                               ← 覆盖率扣分
      - lowConfidencePairs / matchedPairs × 10                 ← 低置信扣分
```

### 1.6 前端交互功能

- **双图同步高亮**：点击差异条目 → 两侧图片分别高亮对应节点区域
- **节点树 + 锁定**：可在节点树中选中节点，锁定后不参与交互高亮
- **置信度标注**：颜色区分 high/medium/low 置信度的匹配结果
- **文件智能识别**：拖拽上传时按文件名规则自动分配到 4 个槽位

| Case | 设计节点 | 匹配对数 | 匹配率 |
|------|---------|---------|--------|
| case1 | 138 | 5 | 4% |
| case4 | 433 | 13 | 3% |
| case7 | 1003 | 24 | 2% |
| 平均 | — | — | **1~4%** |

根本原因：Mock 数据与真实数据不一致（文本无法精确匹配）、画布高度比例差异导致坐标系偏移、大量纯布局容器和 shape/other 节点缺乏有效匹配通道。详见 `改进方向.md`。

## 三、设计师使用的核心痛点（现状 vs 需要）

| 维度 | 当前状态（面向开发者） | 设计师需要 |
|------|-------------------|----------|
| 术语 | IoU、normRect、matchType、置信度 | 人话描述差异，无技术术语 |
| 差异展示 | 属性名 + 原始数值对 | 视觉对比（色块、字号感知） |
| 发现路径 | 阅读文字列表找问题 | 直接在图上看到哪里错了 |
| 输入流程 | 手动准备 4 个文件，需了解文件格式 | Figma 插件一键导出，或只传设计图 |
| 可操作性 | 只读，不能标注和反馈 | 标记问题、添加备注、导出给开发 |
| 覆盖率理解 | 匹配率 3% 看不懂 | 绿/红/灰 三态标注哪些元素被检查了 |
| 规范感知 | 无规范信息 | 标注"不符合设计规范（色彩/间距 Token）" |

## 四、改进方向

### 4.1 前置条件：匹配率提升

差异报告对设计师有价值的前提是匹配到足够多的节点。当前 1-4% 的匹配率导致大量元素处于"未检查"状态，报告可信度不足。需先按 `改进方向.md` 的建议将匹配率提升至 40-60%。

优先级最高的改进项（代码改动小、收益大）：
1. **修复坐标归一化**：y 坐标改为除以画布高度而非宽度（4 行改动）
2. **移除 hasBackground 容器过滤**：对所有容器做几何匹配，按是否有装饰分档设阈值

### 4.2 差异可视化增强（P0）

**目标**：设计师不看列表，在图上直接定位问题。

- 在设计图和截图上叠加**标注层**：
  - 红框 = error，黄框 = warning，绿框 = 检查通过，灰框 = 未匹配
- 点击框弹出**差异气泡卡**，展示：
  - 属性名（中文）
  - 设计值（含色块、字号预览）vs 实现值
  - 例：`文字颜色` 设计 ████ `#5A6170`  →  实现 ████ `#000000`（偏深）
- 支持**差异热力图模式**：整图错误密度渐变叠加，一眼看出问题集中区域

### 4.3 报告语言设计师化（P0）

将所有面向开发者的技术描述转换为设计师熟悉的语言。

**当前输出示例**：
```
fontColor: #FF303133 → #FF000000 (error)
fontSize: 16 → 14 (error)
```

**目标输出示例**：
```
文字颜色偏深
  设计稿 ████ 深灰色 #303133
  实现   ████ 纯黑色 #000000
  
字号偏小
  设计稿 16px → 实现 14px，小了 2px
```

属性中文映射表（部分）：

| 技术属性 | 设计师术语 |
|---------|---------|
| fontSize | 字号 |
| fontWeight | 字重 / 粗细 |
| fontColor | 文字颜色 |
| fontFamily | 字体 |
| textAlign | 文字对齐 |
| backgroundColor | 背景色 |
| borderRadius | 圆角 |
| padding | 内边距 |
| backdropBlur | 背景模糊 |
| shadow | 阴影 |

### 4.4 三态节点覆盖视图（P1）

在设计图上用颜色标出所有节点状态，让设计师直观了解检查覆盖情况：

- **绿色**：已匹配且无差异（样式还原正确）
- **红/黄色**：已匹配但有差异（error/warning）
- **灰色**：未匹配到对应实现（无法检查，可能是元素缺失）

这一视图同时也会暴露匹配率低的问题，形成对匹配算法的质量压力。

### 4.5 输入流程简化（P1）

**短期**：
- 优化文件上传提示，用截图说明各文件如何获取
- 支持只传设计截图 + ArkUI 截图，调用视觉 AI（Claude Vision）做纯图像模式对比（精度较低，但无需 JSON）

**中长期**：
- 接入 Figma Plugin API，从 Figma 直接导出选中帧的结构化数据，设计师无需感知 JSON 格式
- 与华为开发者工具集成，ArkUI Inspector 数据自动同步

### 4.6 问题标注与导出工作流（P1）

设计师在评审结果后需要将问题同步给开发：

- 在差异列表旁增加状态标记：`已确认问题` / `可接受` / `无需修复`
- 一键导出：
  - 带标注的截图（PNG）
  - Markdown / PDF 问题清单（含设计值/实现值对比、截图区域裁剪）
- 生成可分享评审链接（需要后端加持久化存储）

### 4.7 设计规范库集成（P2）

当前只做设计稿 vs 实现的对比，但设计师更关心"实现是否符合规范"：

- 接入设计 Token（色彩 Token、字体 Token、间距 Token）
- 检查实现值是否属于规范内的合法值
- 标注方式：
  - "文字颜色不在规范色板内"比"颜色不一致"对设计师更有价值
  - "圆角 8vp 不符合规范，规范值为 12vp"给出明确修复指引

## 六、关键风险与对策

| 风险 | 描述 | 对策 |
|-----|------|-----|
| 匹配率不足导致报告失公信力 | 大量灰色节点让设计师质疑结果 | Phase 1 必须先解决，否则后续优化价值打折 |
| 纯图像模式精度不足 | 视觉 AI 对精确数值（1px 差异）感知有限 | 明确标注为"估算模式"，与 JSON 模式区分 |
| 设计 Token 格式多样 | 不同项目 Token 定义方式不同 | 先支持 Figma Style/Variable 格式，再扩展 |
| Mock 数据与真实数据天然不同 | 文本节点永远无法 100% 精确匹配 | 增加模糊语义匹配，告知设计师"内容类差异已排除" |

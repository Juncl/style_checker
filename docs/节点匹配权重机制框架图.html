节点匹配权重机制框架图

这张图把节点匹配里的权重机制拆成三层：先用硬门槛过滤不该进来的节点，再用多维软权重给候选打分，最后通过全局排序和一对一收口确定最终 pair。

1. 硬门槛 / 候选过滤

过滤
不可比节点先剔除

空文本、透明节点、Blank、弱可见节点、明显遮挡节点先排除，不进入后续打分。

语义门槛
类型不对直接拦截

时间、星期、数字、单位、标题等文本类型先分类，类型不一致就不会进入宽松救援。

可见性
弱可见文本收紧

低像素可见度文本不能靠语义硬救，需要更近的位置关系和更强的类型一致性。

2. 候选打分 / 软权重

文本精确
同文本候选优先
  • 纵向位置接近度
  • 横向位置接近度
  • 样式相似度
  • 尺寸 / 高度比例
  • 像素可见性
动态槽位
时间 / 星期 / 数值槽位

用位置、顺序、样式、区域亲和力一起打分，解决 Mock 文案和真实数据不一致的问题。

区域文本
区域内最优分配

先在同一区域里做局部最优，再回到全局，避免列表、卡片里的文本互相串位。

3. 局部结构 / 拓扑增强

强锚点
先建立局部坐标系

先用高置信文本命中形成 strongAnchors / topologyAnchors，再围绕这些锚点推断周边节点。

拓扑分数
相对关系优先
  • 相对锚点 dx / dy
  • 宽高比例
  • 文本样式
  • 语义 / 角色
  • IoU + 区域亲和力
周边推断
候选池只看局部

candidatePool 先优先局部区域候选，再回落到全局，减少跨区域误配和远距离串位。

4. 全局收口 / 最终排序

优先级
matchType 排序

text-content、region-text-optimal、text-role、anchor-topology、dynamic-text-slot 等类型各自有权重优先级。

置信度
高置信先保留

高 / 中 / 低置信度会影响最终取舍，低置信 pair 不能抢占高置信结果。

一对一
全局冲突消解

最终只保留一对一配对，避免多个开发节点同时匹配同一个设计节点。

权重机制的核心流程

  1. 先过滤掉不可见、空文本、Blank、弱可见和遮挡节点。
  2. 再用文本精确、动态槽位、区域最优、语义角色等规则打软分。
  3. 随后用强锚点建立局部拓扑坐标系，推断周边节点关系。
  4. 最后按置信度、类型优先级、拓扑分、IoU 和视觉分做全局收口。

权重层级一句话

先有硬门槛决定“能不能进池子”，再有软权重决定“谁更像”，最后靠全局排序决定“谁留下”。

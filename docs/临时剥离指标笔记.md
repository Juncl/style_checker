# 临时剥离指标笔记

本文档记录当前被暂时剥离的三项对比指标，后续如果要恢复，可以直接按这里的实现思路加回。

被剥离的项：
- `gradient`
- `icon.size` / `icon.content`
- `visual.component`

---

## 1. `gradient`

### 现有实现位置
- 设计侧解析：`server/src/parsers/design/buildTree.js`（`extractDesignStyle` 中的渐变抽取）
- 开发侧解析：`server/src/parsers/arkui/buildTree.js`（`extractArkuiStyle`）
- 匹配阶段：`server/src/matchers/nodeMatcher.js` 中的渐变专用匹配通道
- 对比阶段：`server/src/comparators/styleComparator.js` 中的 `diffGradient`

### 当时的行为特点
- 只在设计侧和 ArkUI 侧都存在渐变时才进入对比。
- 设计侧渐变会从 `background` 中抽取 `linear / radial / angular` 三类。
- ArkUI 侧渐变会从 `linearGradient / sweepGradient / radialGradient` 归一化成统一结构。
- 匹配阶段会把带渐变的节点单独拿出来做 `gradient-iou` 匹配。
- 对比阶段只比对两个部分：
  - 渐变色标数量
  - 每个色标的颜色
- 没有比对：
  - 渐变类型
  - 角度
  - 每个色标的位置

### 典型输出
- `gradient`，设计值和实现值都是格式化后的渐变串
- 描述常见为：
  - `渐变：实现缺失`
  - `渐变色标数量不匹配`
  - `渐变第 N 个色标不匹配`

### 恢复时的注意点
- 如果要恢复，不只要把 `diffGradient` 加回去，还要把 `gradient-iou` 的匹配通道补回。
- 如果还想维持原来的配对质量，需要保留渐变节点的区域上下文和对应优先级。

## 2. `icon.size` / `icon.content`

### 现有实现位置
- 设计侧图标碎片标记：`server/src/parsers/design/pruneTree.js`（`semanticCollapse` 函数）和 `server/src/matchers/nodeVisibility.js`
- 节点对比：`server/src/comparators/styleComparator.js`
- 前端展示：`client/src/components/DiffReport.vue` / `client/src/components/ImagePanel.vue`

### 当时的行为特点
- `icon.size`
  - 只要设计侧或 ArkUI 侧任意一边是 `image`，就会做尺寸比较。
  - 阈值是宽高任一方向差值超过 `2`。
  - 输出等级是 `info`，更多是提示图标资源是否缩放失真。
- `icon.content`
  - 只在 ArkUI 侧存在 `iconContent` 时输出。
  - 本质上是资源追踪信息，帮助确认图标 unicode / 资源名是否一致。

### 和匹配链路的关系
- 图标碎片会被打上 `iconFragment` 标记，避免碎片化 path 被当成正常节点参与匹配。
- 这部分主要是为了提高图标类节点的配对稳定性。

### 恢复时的注意点
- 如果只恢复对比，不恢复碎片过滤，图标节点会更容易出现误配。
- 如果恢复 `icon.content`，最好同时确认 ArkUI 侧是否能稳定提供 `iconContent`。

## 3. `visual.component`

### 现有实现位置
- `server/src/routes/check.js`
- `server/src/utils/imageFeatures.js`
- 匹配上下文：`server/src/matchers/regionContext.js`
- 匹配策略：`server/src/matchers/matchStrategies.js`

### 当时的行为特点
- 只对非文本节点启用。
- 只在同时提供设计图和 ArkUI 截图时计算。
- 只对置信度不是 `low` 的匹配对启用。
- 通过节点裁剪后的视觉特征做相似度比较，核心是：
  - 颜色直方图
  - 边缘直方图
  - 感知哈希
  - 平均亮度
  - 前景比例
  - 边缘密度
  - 轮廓中心点
- 报告阈值：
  - 相似度 `< 0.68` 才报
  - `< 0.50` 记为 `warning`
  - 否则记为 `info`

### 匹配链路上的影响
- 区域评分里会混入视觉相似度。
- `bestVisualIoUMatch` 会把几何和视觉特征一起算分。
- 某些节点在 Region 层会因为视觉特征相似而更容易被归到同一组。

### 恢复时的注意点
- 如果要恢复这个指标，建议先恢复区域视觉特征，再恢复 `bestVisualIoUMatch` 和最终差异输出。
- 恢复前最好先确认截图输入在所有 case 上都可用，否则这个指标会退化成空报。

---

## 4. 恢复顺序建议

如果后续要一次性加回，建议顺序是：
1. 先恢复解析层的数据结构
2. 再恢复匹配阶段的专用通道
3. 最后恢复对比阶段和前端展示

这样可以避免“有字段但没有匹配对”或“有匹配对但没有展示项”的中间态。

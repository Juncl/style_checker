# 进入节点匹配前原始数据处理流程

> 说明：本文整理"原始数据进入节点匹配之前"的处理流程。
> 解析阶段已重构为 **5 步独立流水线**，开发侧（ArkUI）和设计侧（Design）各自实现一套同名步骤，方便插入新的过滤策略而不互相耦合。
>
> 入口：
> - 开发侧：`server/src/parsers/arkui/index.js` 导出 `parseArkui(arkuiJson, { imageBuffer })`
> - 设计侧：`server/src/parsers/design/index.js` 导出 `parseDesign(designJson, { imageBuffer })`
>
> 两侧入口都返回扁平的 `UnifiedNode[]`，可直接喂给 `matchNodes`。

---

## 总体架构：5 步流水线

```
原始 JSON
   │
   ▼
┌──────────────────────────────────────────────────────────────────┐
│ step 1  buildTree       原始结构 → UnifiedNode 树（保留 children）│
│                          只做"字段转换"，不做任何过滤            │
├──────────────────────────────────────────────────────────────────┤
│ step 2  pruneTree       硬剪枝（整棵子树删）+ 软剪枝（删自身保子）│
├──────────────────────────────────────────────────────────────────┤
│ step 2.5 normalizeTree  消除"父子尺寸完全一致 + 无视觉装饰"的    │
│                          冗余包装层（两侧共用 utils/normalizeTree）│
├──────────────────────────────────────────────────────────────────┤
│ step 3  annotateTree    像素 / OCR / 后项杀前项 标注 + 剔除      │
├──────────────────────────────────────────────────────────────────┤
│ step 4  flattenTree     树 → 扁平 UnifiedNode[]（DFS 顺序）      │
└──────────────────────────────────────────────────────────────────┘
   │
   ▼
matchNodes(designNodes, arkuiNodes, ...)
```

每一步只做一件事，输入是上一步的输出，方便单独替换或在中间插入新策略。

### 5 步通用语义

| 步骤 | 职责 | 修改方式 |
|---|---|---|
| step 1 buildTree | 把原始 JSON 结构转换成 UnifiedNode 树。每个节点保留 `children` 数组和 `_raw`/`_attrs` 等调试字段。**不剔除任何节点。** | 创建新树 |
| step 2 pruneTree | 两个子操作：**硬剪枝**（整棵子树删除，如 `visibility=Hidden` / `opacity=0` / 越界）+ **软剪枝**（删自身保留子节点，如框架占位节点、透明 layout 容器、极小节点、全屏包裹层）。 | 就地修改 `node.children` |
| step 2.5 normalizeTree | 父子 `rect.x/y/w/h` 完全相等、`child` 非叶子非文本、且 child `style` 无任何视觉装饰时，删掉 child 把孙子接上来。处理多层嵌套包裹。 | 就地修改 `node.children` |
| step 3 annotateTree | 调用 `annotatePixelVisibility` / `annotateTextOcrVisibility` / `annotateVisualOcclusion` 给节点挂标注字段（`pixelVisibility` / `ocrVisibility` / `visualOccluded` …），然后基于 `isPipelineVisibleNode` 把不可见节点做 unwrap（删自身保子）。 | 标注 + 就地修改 |
| step 4 flattenTree | DFS 输出最终 `UnifiedNode[]`，去掉所有内部字段（`_attrs` / `_raw` / `children` / `_frameworkType` 等）。不包含 root 节点本身。 | 返回新数组 |

### 重要的"全局决策"

1. **`paintIndex` 概念已删除**：旧版本基于"通过过滤后的 push 顺序"赋值，不能稳定表示绘制顺序。新流水线不再赋此字段；下游所有节点排序统一改用 `utils/pathOrder.js#comparePaths`（按 path 字典序，等价 DFS 顺序）。
2. **`pruneOccludedSiblings`（兄弟遮挡剪枝）已实装**：接收树根（`root`）和方向参数，bottom-up 递归遍历每层 `children`，直接从树中删除被完全覆盖的节点，**不挂任何标注字段**。ArkUI 侧用 `type='reverse'`（后项杀前项），设计侧用 `type='forward'`（前项杀后项）。两侧均在 step 3 最先调用，先于像素和 OCR。
3. **OCR 异步性**：`parseArkui` / `parseDesign` 都是 `async`（开发侧 step 3 调用 Tesseract）。
4. **两侧 step3 不对称**：设计侧有前项杀后项剪枝 + 像素标注，两步相互独立；开发侧有后项杀前项剪枝 + 像素标注 + OCR 标注，三步相互独立。

## 开发侧（ArkUI）

代码位置：`server/src/parsers/arkui/`

### step 1 · `buildTree.js` — 原始 `$children` 树 → UnifiedNode 树

1. 从 `arkuiJson.content` 读取 `$resolution`（默认 3.5）、`width` / `height` 计算 vp 画布尺寸。
2. 从根节点开始 DFS 递归构建。每个节点产出形如下结构：
   ```
   {
     id, source: 'arkui', type: 'text'|'container', rawType, name, path,
     rect: { x, y, w, h },        // vp 坐标
     normRect: { ... },           // 0-1 归一化
     visible: true,
     style: { ... },              // 已规范化（颜色、圆角、阴影、字号等）
     textContent?: string,
     children: [],
     _frameworkType, _spanType, _blankType, _attrs, _rectRaw,  // 调试用
   }
   ```
3. **不剔除任何节点**：哪怕是 `visibility=Hidden` / `opacity=0` / 越界 / 框架节点（root / Navigation / Span / Blank）等，都建到树里。后续 step 2 统一处理。
4. `style` 在此步就已经按目标格式抽取，包括：
   - `opacity`、`width` / `height`、`backgroundColor`、`borderRadius`、`border`、`blur`（统一字符串：`"背景模糊 Npx"` / `"高斯模糊 Npx"`，背景优先）、`shadow`（统一字符串：`"外阴影 #color Rpx X:x, Y:y"`）、`padding` / `margin`、`itemSpacing`、`blendMode`
   - **例外**：`Image` 类型节点的 `backgroundColor` 在此处直接清空（不写入 style）。ArkUI Image 的 backgroundColor 是图片加载失败时的占位色，正常渲染时不显示，写入 style 会与设计侧产生误比对。
   - 文本节点额外：`fontSize` / `fontWeight` / `fontColor` / `fontFamily` / `lineHeight` / `letterSpacing` / `textAlign` / `actualFontSize` / `declaredFontSize`

5. **Button / TextInput 文本拆分**（仅开发侧）：

   ArkUI 的 `Button` 和 `TextInput` 节点自带文本属性（`label` / `text` / `placeholder`），但设计侧没有"文本+非文本混合"的对应类型——Figma 中按钮文本和按钮容器是分开的两个节点。为消除这种结构不对称，**在 buildTree 阶段为这两类节点凭空造一个虚拟 Text 子节点**：

   | 节点类型 | 触发条件 | 文本内容 | 字体属性来源 | 默认对齐 |
   |---|---|---|---|---|
   | `Button` | `$attrs.label` 非空 | `label` | `fontSize` / `fontWeight` / `fontColor` / `fontFamily` | `center` |
   | `TextInput` | `$attrs.text` 非空 → 用 text；否则 placeholder 非空 → 用 placeholder | `text` 优先 | text 走顶层字体；placeholder 走 `placeholderColor` + 顶层 fontSize | 遵循 `textAlign`，缺省 `start` |

   虚拟 Text 子节点的规则：
   - **`id`** = 原节点 id + `:t`（例：`1951` → `1951:t`）
   - **`rect`** 计算：父节点 `vpRect` 内部，垂直居中，水平按 textAlign 对齐（考虑 padding）。文本宽度按字符估算（中文字符=fontSize，其他=fontSize × 0.55）
   - **`_rectRaw`** 也需要填上（vp × resolution 反算物理像素），否则会被 step 2 硬剪枝以 `no-rect` 剔除
   - **`_splitFromParent: true`** 标记，便于调试

   不满足条件（无 label / 无 text 也无 placeholder / 无 vpRect / 无 fontSize）的 Button / TextInput 不拆分。

6. **clip borderRadius 向下传播**（仅开发侧，`walk` 函数 DFS 时执行）：

   ArkUI 中图片圆角的惯用写法是：在外层容器设置 `clip=true + borderRadius`，Image 本身不设 borderRadius，由父容器裁剪实现圆角效果。为使 Image 节点携带正确的圆角值参与样式比对，buildTree 阶段在 DFS 时向下传播 `clipRadius` 上下文：

   - 当一个节点满足 `clip=true` 且自身 `borderRadius` 非零时，产生新的 `clipRadius` 上下文（内层覆盖外层）。
   - 递归子节点时，**只将 `clipRadius` 传给与当前节点 rect 完全一致（x/y/w/h 差 < 0.5）的子节点**；不同 rect 的子节点忽略，不传。
   - `Image` 类型节点：若自身 `borderRadius` 全为 0 且接收到非空 `clipRadius`，则将 `clipRadius` 赋给 `style.borderRadius`。

### step 2 · `pruneTree.js` — 硬剪枝 + 软剪枝

**先做硬剪枝（整棵子树删除）**：

| 命中条件 | 标记原因 |
|---|---|
| `attrs.visibility === 'Visibility.Hidden'` | `visibility-hidden` |
| `opacity === 0` | `opacity-zero` |
| `rawType` 为 `leftarrow` / `rightarrow` | `special-component` |
| 无 `$rect`（非框架节点） | `no-rect` |
| 越界：`x > canvasW` 或 `y > canvasH` 或 `x+w <= 0` 或 `y+h <= 0` | `out-of-bounds` |
| 异常超宽：`w > canvasW × 3` | `too-wide` |
| `Text` 节点且 `textContent` 为空 | `empty-text` |

**再做软剪枝（删自身保子）**：

| 命中条件 | 备注 |
|---|---|
| `Span` 节点 | 无独立布局 rect |
| `Blank` 节点 | 仅作布局占位 |
| 框架节点（root / Navigation / NavBar / NavBarContent / __Common__ / TitleBar / ToolBar / Divider / ScrollBar / TabBar / BackButton / NavigationMenu / JsView）**且无背景色** | 有背景色时保留（视为视觉节点）。root 永不 unwrap，仅作树根容器 |
| 透明 layout 容器（`jsview` / `stack` / `column` / `row` / `flex` / `list` / `listitem` / `group` / `__common__` / `gridcol` / `gridrow` / `blank` / `spacer`）**且无背景色** | 有背景色时保留 |
| 极小节点（`w ≤ 4` 或 `h ≤ 4` vp） | 无条件 unwrap |
| 全屏包裹层（`normRect.w ≥ 0.999` 且 `normRect.h ≥ 0.999`）**且无背景色** | 有背景色（如底层蒙版）时保留 |

### step 2.5 · `utils/normalizeTree.js` — 消除冗余包装

折叠条件（child 同时满足）：
1. child 非叶子（`children.length > 0`）
2. `child.type !== 'text'`
3. 父子 `rect.x/y/w/h` **严格相等**（同侧解析后不存在浮点误差，不给 EPS）
4. child `style` 无视觉装饰：`style.backgroundColor` 透明 + 无 `style.borderRadius` + 无 `style.shadow` + 无 `style.blur` + 无 `style.border`

折叠后 child 的孙子节点原位接到父节点 `children`，i 不递增以处理多层嵌套包裹。

### step 3 · `annotateTree.js` — 后项杀前项剪枝 + 像素标注 + OCR 标注 + 不可见 unwrap

四个子操作相互独立，不通过标注字段联动：

1. 调 `pruneOccludedSiblings(root)` —— 后项杀前项，传树根，bottom-up 递归直接删除被覆盖节点，无标注。先于像素/OCR 运行，保证后续标注只处理真正可见的节点。
2. 收集树中所有非 root 节点的引用列表（基于剪枝后的树）。
3. 若提供 `imageBuffer`：
   - 调 `annotatePixelVisibility(nodeList, imageBuffer, canvas, { source: 'arkui' })` 给节点挂 `pixelVisibility` / `pixelInvisible`。
   - 调 `await annotateTextOcrVisibility(nodeList, imageBuffer, canvas, { source: 'arkui', mode: 'all' })` 给文本节点挂 `ocrVisibility`。
4. 基于 `isPipelineVisibleNode` 对树做 unwrap：节点本身判定为不可见时，删自身把它的子节点接到祖父。

`isPipelineVisibleNode` 的判定门禁（来自 `matchers/nodeVisibility.js`，未改变）：
- `visible !== false`
- `!pixelInvisible`
- 文本节点不能挂 `hiddenFrameworkAncestor`（新架构下该字段不再被设置，等价于不命中）
- `!visualOccluded`
- 文本节点的 OCR 严格条件：`ocrVisibility.visible === false` **且** `textStrokeScore < 0.05` **且** `visiblePixelRatio > 0.20` **且** 非浅色字体 → 剔除

### step 4 · `flattenTree.js` — DFS 输出

- 从 root 的 children 开始 DFS（不包含 root 自身）。
- 每个节点 push 前去掉内部字段：`children` / `_attrs` / `_rectRaw` / `_frameworkType` / `_spanType` / `_blankType` / `_prunedReason`。
- 输出干净的 UnifiedNode[]。

### 编排（`index.js`）

```js
const { canvasWidthVp, canvasHeightVp, resolution, root } = buildArkuiTree(arkuiJson)
pruneArkuiTree(root, canvasWidthVp, canvasHeightVp)
normalizeTree(root)
const { stats: annotateStats } = await annotateArkuiTree(root, { imageBuffer, canvasWidthVp, canvasHeightVp })
const nodes = flattenArkuiTree(root)
return { canvasWidthVp, canvasHeightVp, resolution, nodes, annotateStats, _root: root }
```

## 设计侧（Design / Figma）

代码位置：`server/src/parsers/design/`

设计侧的关键差异：**原始 `design.json.data` 是扁平数组**，每个节点带 `path` 数组表示在原树中的位置。step 1 需要按 path 重建为树。

### step 1 · `buildTree.js` — 扁平 `data[] + path` → UnifiedNode 树

按用户决策，内部分两步：

1. **1a 重建结构**：用 `Map<pathKey, treeWrapper>` 索引每个 raw 节点，按 path 长度从短到长依次接到 parent（`parentKey = path.slice(0, -1).join('.')`）。**遇到孤儿节点（parent 不存在）直接丢弃。** 根节点是 `path = [0]`。
2. **1b 字段统一**：在重建后的树上递归把每个 raw 节点转成 UnifiedNode 形状，跟开发侧字段命名对齐：
   - `id` 来自 `guid`，`source: 'design'`
   - `type`：`TEXT → 'text'`，其余 → `'container'`
   - `rawType`：Figma 原始类型小写
   - `size`：原始 Figma dp 坐标（`{ x, y, w, h }`），供样式比对阶段使用（间距、内边距等）
   - `rect`：在 step 1 阶段按 `scale = arkuiCanvasWidthVp / origCanvasWidth` 等比缩放后的坐标，与 ArkUI vp 坐标系对齐，用于匹配阶段
   - `normRect`：基于缩放后的画布尺寸（`canvasWidth = origCanvasWidth × scale`）归一化，使两侧 normRect 可直接比较
   - `style`：经 `extractDesignStyle` 提取，对齐开发侧（`backgroundColor` / `borderRadius` / `border` / `blur` / `shadow` / `padding` / `itemSpacing` / 文本属性等）
   - 文本节点补 `textContent = raw.content`
   - 保留 `_raw` 引用供 step 2 折叠/剪枝使用
3. **HM Symbol 文本节点的特殊处理**：在 buildTree 阶段判断 `style.text[0].fontFamily` 是否包含 `hm symbol`（不区分大小写）。命中后：
   - `type` 强制改为 `'container'`，`rawType` 改为 `'symbolglyph'`
   - 不设置 `textContent`
   - 字体颜色写入 `style.backgroundColor`（与开发侧 `SymbolGlyph` 对齐）

4. **VECTOR 节点的 borderRadius 提取**（`extractDesignStyle` 内）：

   常规的 borderRadius 来自 `style.borderRadius[]`。但 Figma VECTOR 节点的路径形状圆角不存储在 `style.borderRadius` 里，而存储在 `style.vectorData.styleOverrideTable` 中（每个路径顶点对应一个条目，携带 `cornerRadius` 字段）。

   当 `rawType === 'VECTOR'` 且尚未提取到 borderRadius 时，额外从 `styleOverrideTable` 取值：
   - 遍历所有条目，收集 `cornerRadius > 0` 的值，取最大值
   - 对 rect 做上限裁剪（`min(r, min(w, h) / 2)`）
   - 写入四角统一的 borderRadius 对象

5. **蒙版裁剪（`applyMaskClip`）**：

   Figma 中 `style.mask === true` 的节点定义一个可见区域，其前序兄弟（在图层列表中排在该节点上方、即 path 末位索引更小）的所有内容被裁剪到该蒙版 rect 范围内。

   实现位置：**1b 字段统一阶段**，在 `convertToUnified` 的 children for 循环结束后、`return unified` 之前调用 `applyMaskClip(unified.children, canvasW, canvasH)`。此时所有直接子节点均已完成转换，可以安全读取和修改 rect。

   裁剪逻辑（`applyMaskClip` + `clipToMask`）：
   - 扫描同层 `children`，找到 `_raw.style.mask === true` 的节点，以其 `rect` 作为蒙版边界
   - 对该节点的所有前序兄弟（`index < maskIndex`）取与蒙版 rect 的**交集**（`newX = max(r.x, mask.x)` 等），同步更新 `rect` 和 `normRect`
   - **递归传播到整棵子树**：Figma 蒙版是视觉裁剪，被裁节点的子树内容同样受约束，因此 `clipToMask` 在修改当前节点后继续对所有 `children` 递归调用
   - 蒙版节点本身的 rect 不变

### step 2 · `pruneTree.js` — 硬剪枝 + 语义折叠 + 后代清空 + 软剪枝

按顺序 4 个子操作：

**2a 硬剪枝（整棵子树删）**：
- `opacity === 0`
- 越界（`x > canvasW` 或 `y > canvasH` 或 `x+w ≤ 0` 或 `y+h ≤ 0`）
- 系统状态栏整组件（`status-bar`）—— 四要素全部满足时硬剪：
  - name 含 `StatusBar`，或 `_raw.componentData.componentDescription === 'StatusBar'` / `componentName` 含 `StatusBar`
  - 贴左上角：`x === 0` 且 `y ≤ 2`（容忍 1-2px 设计稿偏移）
  - 矮条：`h < 50`
  - 子树中存在时间格式文本（`/^\d{1,2}:\d{2}$/`，如 `08:08` / `9:41`）
  - 状态栏（时间/信号/电量）属系统级 UI，不参与还原度比对，故整棵子树删除

**2b 语义折叠** —— icon / illustration 折叠为单节点：

候选 root 的判定（`isSemanticAssetRoot`）：
- 原始 type 是 `FRAME` / `GROUP` / `VECTOR` / `BOOLEAN_OPERATION`
- 名称非空，且命中：
  - 插画类：`/^#?illustration\b/` 或 `/插画|illustration/`
  - 图标类：`/(^|[/_\s-])icon([/_\s-]|$)|(^|[/_\s-])ic_|图标/` 且尺寸 4-48dp、长宽比 ≤4
- 排除条件：
  - 有文本后代 → 不折叠
  - 大插画容器（命名是插画 + 面积占比 ≥ 0.55 + 至少 3 个文本后代）→ 不折叠

折叠操作（`collapseToSemanticAsset`）：
- 选 representative：树内非文本后代中，尺寸与 frame 相差 ≤1px、且有背景色的节点；多个候选取路径最浅者
- 用 representative 的 rect / normRect / style / rawType 覆盖当前节点（保留 id / path / name）
- 设置 `semanticAsset = true`、`style.semanticAsset = 'icon' | 'illustration'`
- 清空 `children`

**2b.5 背景节点同化**（`mergeFrameBackgroundRects`）：

针对 Figma 中"背景 RECTANGLE/FRAME 与容器 FRAME 完全重叠"的模式。对每个 `rawType=FRAME` 的节点，**DFS 搜索其整个子树**（`absorbSameRectInSubtree`），寻找与该 FRAME 同 rect 的后代节点进行同化。同层节点倒序遍历（index 靠后先处理，靠前后处理，靠前覆盖靠后）：

- **同 rect 且类型为 `FRAME`**：
  - **样式合并（有条件）**：child 有视觉装饰时将其 style 合并入祖先 FRAME；
  - **unwrap**：将 child.children 原位提升（splice 替换），i 不变继续处理提升上来的节点；
  - 继续在提升上来的节点中 DFS。
- **同 rect 且类型为 `RECTANGLE`**：
  - **样式合并（有条件）**：同上；
  - **删除**：RECTANGLE 无后代，直接从列表中移除，停止这条路径。
- **其他节点**（非 FRAME/RECTANGLE，或非同 rect）：穿透，继续往下 DFS 其子节点。

同化完成后，对剩余子节点递归调用 `absorbDescendants`（它们自身可能也是 FRAME）。

**效果**：同尺寸背景层的视觉属性（backgroundColor / borderRadius 等）上浮到容器 FRAME，使其从无装饰的结构容器转为有视觉装饰的可匹配节点。中间层（如 GROUP）被自动穿透，不影响深层 RECTANGLE 的同化。

**2c BOOLEAN_OPERATION 后代清空**：节点本身保留，但 `children = []`（避免后代被当成普通节点重复输出）。

**2d 软剪枝（删自身保子）**：
- 非 VISUAL_TYPES 节点（理论上不会出现，兜底保留）
- 空文本 TEXT 节点
- `GROUP` 节点**且无视觉装饰**（backgroundColor / borderRadius / border 均无）；有装饰时保留
- 极小节点（`rect.w ≤ 4` 或 `rect.h ≤ 4` dp）—— 无条件 unwrap
- 全屏包裹层（`normRect.w ≥ 0.999` 且 `normRect.h ≥ 0.999`）**且无视觉装饰**

### step 2.5 · `normalizeTree` 同两侧共享

### step 3 · `3-annotateTree.js` — 前项杀后项剪枝 + 像素标注 + unwrap

两步相互独立，不通过标注字段联动：

1. 调 `pruneOccludedSiblings(root, 'forward')` —— 前项杀后项，直接删除被前绘制兄弟完全覆盖的节点，无标注。
2. 收集节点列表（基于剪枝后的树），调 `annotatePixelVisibility(nodeList, imageBuffer, canvas, { source: 'design' })`。
3. 基于 `isPipelineVisibleNode` 做最终 unwrap。
- **不做 OCR**（设计侧文字内容来自 JSON，不需要 OCR 验证）。

### step 4 · `flattenTree.js` — DFS 输出

去掉 `children` / `_raw` / `_prunedReason` 等内部字段。

### 编排（`index.js`）

```js
const { canvasWidth, canvasHeight, root } = buildDesignTree(designJson)
pruneDesignTree(root, canvasWidth, canvasHeight)
normalizeTree(root)
const { stats: annotateStats } = annotateDesignTree(root, { imageBuffer, canvasWidth, canvasHeight })
const nodes = flattenDesignTree(root)
return { canvasWidth, canvasHeight, nodes, annotateStats, _root: root }
```

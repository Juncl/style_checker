# 过滤阶段特殊规则

这份文档只整理“进入节点匹配之前”的特殊规则。

我把它分成三块：

1. 开发侧规则
2. 设计侧规则
3. 合并入口

---

## 开发侧规则

这部分只适用于 **ArkUI / 开发侧**。

### 1. 文本内容收集

开发侧 `Text` 节点的内容收集规则是：

```js
content || accessibilityText
```

如果 `content` 没有值，但 `accessibilityText` 有值，节点仍然会被保留，并且会把 `accessibilityText` 作为 `textContent`。

### 2. 文本节点是否进入统一节点

开发侧文本节点只有在以下条件成立时才会进入统一节点：

- `content` 有值
- 或 `accessibilityText` 有值

如果两者都为空，该文本节点不会进入统一节点列表。

### 3. 形状背景色归一化

开发侧形状节点有专门的背景色取值规则：

- `Circle` / `Ellipse` / `Rect`
  - 优先取 `fill`
  - 再回退到 `foregroundColor`
- 其他节点
  - 使用 `backgroundColor`

### 4. 框架节点与占位节点

开发侧会跳过这些节点：

- 纯框架节点
  - 例如 `root`、`JsView`、`Navigation`、`Divider`、`TitleBar`、`ToolBar`
- `Blank`
- `Span`
  - `Span` 本身不单独作为视觉节点输出
  - 但会继续遍历它的子节点

### 5. 隐藏祖先标记

如果开发侧文本节点位于隐藏框架祖先下，会被标记为：

- `hiddenFrameworkAncestor = true`

这个标记会影响后面的过滤阶段。

### 6. OCR 过滤

OCR 只作用于开发侧，不作用于设计侧。

开发侧会在主流程里自动跑 OCR，用来辅助判断文本节点是否真的在截图中可见。

### 7. 开发侧前置过滤结果

开发侧节点在进入匹配前，会经过统一过滤判断：

- 像素不可见的节点会被过滤
- OCR 判定为不可见的文本节点会被过滤
- 隐藏框架祖先下的文本节点会被过滤
- 被遮挡的节点会被过滤

## 设计侧规则

这部分只适用于 **设计侧 / design.json**。

### 1. 文本内容收集

设计侧 `TEXT` 节点直接使用：

```js
node.content
```

设计侧不使用 `accessibilityText` 作为兜底。

### 2. 语义资产折叠

部分设计侧节点会在解析时被识别为语义资产：

- 会被归一化成 `image`
- 同时标记 `semanticAsset = true`

这类节点会更偏向按“整体资源”处理，而不是按原始子图层继续拆分。

### 3. BOOLEAN_OPERATION 与零透明过滤

设计侧会特殊处理这些情况：

- `BOOLEAN_OPERATION`
  - 其路径下子节点会被折叠，不直接作为普通视觉节点参与后续流程
- `opacity = 0`
  - 该节点及其后代会被过滤掉

### 4. 设计侧前置过滤结果

设计侧节点在进入匹配前，会经过统一过滤判断：

- 文本内容为空的 `TEXT` 节点会被过滤
- 被语义资产折叠的节点不会进入后续候选池
- `BOOLEAN_OPERATION` 相关层级不会继续作为普通视觉节点参与
- 透明度为 0 的节点及其后代会被过滤

## 合并入口

这部分是设计侧和开发侧最后汇合的地方。

### 1. 前置过滤统一入口

两侧节点都会进入：

```js
isPipelineVisibleNode(node)
```

这个阶段决定一个节点是否还能继续往下走。

### 2. 进入匹配前的数据

过滤后会得到两份节点集合：

- `designVisibleNodes`
- `arkuiVisibleNodes`

只有这两份数据会进入后续节点匹配。

### 3. 后续流程

过滤通过后的数据会依次进入：

1. 节点匹配
2. 一对一节点映射
3. 节点属性对比
4. 后端结果组装
5. 前端展示

### 4. 过滤阶段的意义

过滤阶段不是“前端隐藏”，而是**真正剪掉候选节点**。

所以被过滤的节点不会再进入：

- `pairs`
- `diffs`
- `unmatched`
- 节点树
- 前端高亮

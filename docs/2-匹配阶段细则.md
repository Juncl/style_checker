# 匹配阶段细则

> 匹配阶段将解析流水线输出的两侧 `UnifiedNode[]` 两两配对，生成 `MatchPair[]`。
>
> 入口：`server/src/matchers/nodeMatcher.js` → `matchNodes(designNodes, arkuiNodes, options)`

---

## 一、总流程

### 匹配方向

`matchNodes` 方向固定，不再支持 design-first / arkui-first 切换（`?matchDirection` 参数与 `STYLE_CHECKER_MATCH_DIRECTION` 环境变量已失效）：

- **Pass 1**：以 ArkUI 为外层主序（`matchAllTextNodes`），对每个 ArkUI 文本节点遍历全部 Design 文本节点取最高分
- **Pass 2 及以后**：以 Design 为遍历主序（`matchNodesDesignFirst`）

旧的 `matchNodesArkuiFirst` 双跑合并机制已删除。

### Pass 流水线

### 全局机制

| 机制 | 时机 | 作用 |
|---|---|---|
| **区域切割** | Pass 1 完成后 | 把节点按 Y 分组，限制候选池范围 |
| **isAcceptablePair** | 所有 Pass 结束后 | 过滤类型不兼容、尺寸悬殊等非法配对 |
| **selectOneToOnePairs** | 最终 | 按优先级排序，保证每节点只出现在一个 pair |

## 二、各步骤详细说明

### Pre · 节点入池

输入的 `UnifiedNode[]` 已由解析流水线完成所有可见性与尺寸过滤，匹配阶段不做额外检查。

**isComparableOutputNode**（最终统计和输出用）：`!isStructuralContainer(node)`
- `isStructuralContainer`：`type='container'` 且 `!isRenderableNonTextNode(node)`（无视觉装饰且 rawType 不在内在视觉集合中的纯布局容器不计入覆盖率）

**isAcceptablePair**（配对合法性，所有 Pass 输出后统一过滤）：

**文本对规则（design.type === 'text'）：**

| 条件 | 处置 |
|---|---|
| ArkUI 侧非文本节点 | 拒绝 |
| 任意一侧文本长度 < 6，且两侧长度差值 > 20 | 拒绝（短文本不得配长文本） |
| 其余 | 放行 |

**容器对规则（design.type === 'container'）：**

| 条件 | 处置 |
|---|---|
| 设计侧是 `isStructuralContainer` 且 matchType 不是 `list-index` | 拒绝 |
| 弱匹配类型（anchor-topology / rescue-iou / container-iou / container-geometry）最小尺寸比 < 0.22 | 拒绝 |
| 弱匹配 + 极小图标（w/h ≤ 8）且尺寸比 < 0.35 | 拒绝 |
| 弱匹配 + 两侧 container 宽高比差异过大（< 0.45） | 拒绝 |
| 弱匹配 + arkui=symbolglyph + 设计侧视觉图标 + 尺寸比 < 0.72 | 拒绝（仅保留尺寸保护；原「中心距 > 0.04」检验已移除——两侧系统性偏移会把正确图标对误杀） |
| 低置信度 + 中心距 > 0.08 + 最小尺寸比 < 0.70 的 container | 拒绝 |

### Pass 1 · 全文本节点加权匹配（text-content，matchSource=creText）

实现：[`server/src/matchers/allTextMatcher.js`](../server/src/matchers/allTextMatcher.js) → `matchAllTextNodes(designNodes, arkuiNodes)`

**主流程**：以 ArkUI(hm) 为外层主序，对每个 ArkUI 文本节点遍历全部 Design(pix) 文本节点，5 个子评分加权出 `finalScore`，取最高分候选；`finalScore ≥ 0.9` 视为可信文本，转成 `confidence='high'`、`isAnchor=true` 的强锚点 `MatchPair`。

#### 五个子评分

| 子分 | 函数 | 算法 | 缺值处理 |
|---|---|---|---|
| 内容 | `textSimilar` | 归一化后完全相同 → 1；有前后缀命中时 `edit×0.63 + semantic×0.12 + prefixSuffix×0.25`（取与 editScore 的较大值）；否则 `max(editScore, semanticScore)` | 任一侧空文本 → 0 |
| 颜色 | `getSimilarityColor` | 双侧 ARGB 按 alpha 与白底混合 → 曼哈顿距离 ÷ 765 → `1 - d/765` | 任一侧缺色 → 1 |
| 字号 | `getSimilaritySize` | `gaussianCurveParabola(s1, s2, {x:4, y:0.5}, 12)` | 任一侧缺值 → 1 |
| 字重 | `getWeightScore` | `gaussianCurveParabola(w1, w2, {x:300, y:0.5}, 600)` | 任一侧缺值 → 1（避免误触发一票否决） |
| 位置 | `getPlaceScore` | 两侧 textAlign **均为 center** 时取节点中心点，否则取左上角点；双向计算取最大：① 顶对齐；② 底对齐（各自距底边高度替换 Y）。每套均计算**三维加权**：`euclid×0.6 + x轴×0.3 + y轴×0.1`，三维参数分别为：欧氏 `{x:0.2·diagonal, y:0.5}, 0.5·diagonal`；x轴 `{x:0.3·rootRectW, y:0.5}, 0.6·rootRectW`；y轴 `{x:0.4·rootRectMaxH, y:0.5}, 0.8·rootRectMaxH`。diagonal = 两侧对角线均值，rootRectW = 两侧画布宽度均值，rootRectMaxH = 两侧画布高度最大值 | 总有值 |

其中 `editScore = 1 - levenshtein(t1n, t2n) / max(|t1n|, |t2n|)`，`t1n`/`t2n` 为将归一化文本中所有连续数字替换为单个 `'0'`（`/\d+/g → '0'`）后的结果，使数字值差异不影响结构相似度；`semanticScore = textSemanticSimilarity(...)` 仍使用原始文本。

**前后缀命中条件**：在数字归一化后的字符串上，满足 `|len1 - len2| ≤ 3` 且前2字符相同或后2字符相同，则 `prefixSuffixScore = 0.7 + (1 - lenDiff/3) × 0.3`，触发三权重融合（`edit×0.63 + semantic×0.12 + prefixSuffix×0.25`，取与 editScore 的较大值）。

#### 抛物线-高斯分段曲线（gaussianCurveParabola）

输入 `(num1, num2, point, diffmax)`，记 `diff = |num1 - num2|`：

| 条件 | 输出 |
|---|---|
| `num1 === num2` | 1 |
| `diff > diffmax` | 0 |
| `diff ≤ point.x` | `1 - a · diff²`，其中 `a = (1 - point.y) / point.x²` |
| `diff > point.x` | `exp(-diff² / 2σ²)`，其中 `σ = point.x / √(-2 · ln(point.y))` |

抛物线段与高斯段在 `diff = point.x` 处衔接，两侧值均为 `point.y`。结果 `toFixed(4)` 保留 4 位小数。

#### 最终加权（getTextFinalScore）

```
if (place === 0 || weight === 0) return 0                    // 一票否决
if (color === 1 && size === 1 && weight === 1 && place ≥ 0.9)
  return content * 0.15 + place * 0.6 + 0.25                 // 高可信特殊分支
return content * 0.25 + color * 0.15 + size * 0.15 + place * 0.35 + weight * 0.1
```

- **一票否决**：位置或字重子分为 0 → finalScore 直接为 0（位置远超对角线一半 / 字重相差 ≥600）
- **高可信特殊分支**：字色/字号/字重三者完全一致 且 位置 ≥ 0.9，公式让位置主导得分（最高 1.0）
- **普通加权**：位置权重最高（0.35），其次内容（0.25），颜色/字号各 0.15，字重 0.10

#### 多对一去重

可信集合（`finalScore ≥ 0.9`）内若多个 ArkUI 节点命中同一个 Design 节点，仅保留 `maxScore` 最高的那个，其余舍弃（**不回退到次高分候选**）。落败的 ArkUI 节点本 Pass 不产出 pair，留给下游 Pass 处理。

#### 输出结构

```
{
  pairs,                  // 可信文本（≥0.9）去重后转成的强锚点 MatchPair[]
  textHmMapPix,           // 全部 hm→de 最高分映射（含 <0.9）
  textHmMapPixCredible,   // 仅 ≥0.9 的 hm→de 映射
  textHmMapPixDetail,     // 每个 hm 的完整比对详情（含 compareDeIdS 全候选评分）
}
```

`pairs` 进入主流程的 `pairs` / `usedArkui` / `matchedDesignIds`，并同时推入 `strongAnchors`（驱动 Post-1 区域切割）和 `topologyAnchors`（驱动 Pass 4）。其余三个 map 由 `matchNodes` 透传到返回值的 `textMatch` 字段，供前端可视化 / 调试使用。

### Post-1 · 区域切割（segmentRegions + buildRegionContext）

**时机**：Pass 1 完成后立即执行

**切割规则（segmentRegions）**：按 Y 轴中心升序排列，相邻节点中心间距 > 0.075 且当前区域已有 ≥ 4 个节点时切断产生新区域。每个区域记录：包围盒 `rect`、节点 id 集合、text/container 计数、可见文本集合、装饰节点比例。

**区域对匹配（regionMatchScore）**：两侧区域两两打分（> 0.28 入池），按分数降序一对一贪婪分配：

```
anchorVotes * 0.30    // 已匹配强锚点落在该区域对的数量（最多算 1.0）
geometryScore * 0.22  // 区域中心 Y 距离 + 高度比
histScore * 0.18      // text/container 计数余弦相似度
textScore * 0.10      // 区域内文本集合 Jaccard 相似度
decorationScore * 0.20 // 装饰节点比例差
```

**candidatePool**：为设计侧节点查找 ArkUI 候选时，优先取该节点所属区域对应的 ArkUI 区域内的节点；区域内无候选时回退到全量（全局兜底）。

### Pass 2a · 动态数字槽位（dynamic-number-slot）

**触发**：设计侧文本 `textFieldType = 'number'`，尚未匹配

**评分（numericSlotScore）**：
```
条件：dx ≤ 0.20，dy ≤ 0.16，style ≥ 0.60，hRatio ≥ 0.50
xScore * 0.34 + yScore * 0.24 + style * 0.22 + hRatio * 0.10 + regionScore * 0.06 + lineScore * 0.04
```

**阈值**：score ≥ 0.70 → confidence = score > 0.82 ? 'medium' : 'low'

---

### Pass 2b · 动态时间/星期槽位（dynamic-text-slot）

**触发**：`textFieldType ∈ { 'time', 'weekday' }`，两侧均有此类文本

**算法（orderedSlotAssignment）**：两侧同类文本按阅读顺序（Y 优先，再 X）排列后，动态规划对齐（允许跳过）。

单对评分（slotScore）：
```
条件：dx ≤ 0.24，dy ≤ 0.22，style ≥ 0.55，hRatio ≥ 0.45
xScore * 0.24 + yScore * 0.18 + orderScore * 0.24 + style * 0.16 +
hRatio * 0.10 + regionScore * 0.04 + lineScore * 0.04 + 0.72 * 0.04
```

**阈值**：单对 score ≥ 0.58 → confidence = score > 0.72 ? 'medium' : 'low'

---

### Pass 2c · 同行文本行槽位（text-row-slot）

**触发**：两侧各自存在"同行（centerY 差 < 0.018）且 ≥ 2 个节点"的文本行，行内文本满足 `isRowSlotText`（time/weekday 类型或 2~4 字纯汉字），且 centerY > 0.26（排除顶部状态栏）

**行匹配（bestRow）**：
```
条件：dy ≤ 0.08，得分 ≥ 0.52
(1 - dy/0.08) * 0.45 + countScore * 0.20 + xOverlapScore * 0.35
```

行对确认后，用 `orderedSlotAssignment` 对齐行内节点（score ≥ 0.50 才记录）

**ArkUI 侧额外过滤**：`isSlotVisibleEnough`：visiblePixelRatio ≥ 0.12 或 textStrokeScore ≥ 0.09

### Pass 2d · 长文本位置-样式兜底匹配（long-text-fallback）

实现：[`server/src/matchers/longTextFallback.js`](../server/src/matchers/longTextFallback.js) → `matchLongTextFallback(...)`

**触发**：ArkUI 侧未匹配文本节点，且 `textContent.trim().length > 12`（超过 12 个字符）。从 Design 侧全部未匹配文本节点中搜索候选。

**设计背景**：长描述文本（天气介绍、详情说明等）在两侧内容往往完全不同（mock vs 动态数据），Pass 1–2c 均无法匹配。此阶段主要依赖视觉位置和文字样式作为匹配信号，内容仅作极低权重参考。

#### 一票否决：锚点方向检验（isVetoed）

以当前 ArkUI 节点中心点到各锚点 ArkUI 侧中心点的欧氏距离，选取最近 **3 个**锚点（`topologyAnchors` 中两侧均有 rect 的配对）。

对每个锚点判断方向时：
- **同行判断**：设计侧节点与锚点设计侧的 Y 范围有重叠（`node.y ≤ anchor.y + anchor.h && anchor.y ≤ node.y + node.h`），则视为同行 → **跳过此锚点，不参与方向比对**
- **方向（上/下）**：中心 Y 较小者为 `above`，较大者为 `below`
- **否决条件**：某锚点（非同行）在 ArkUI 侧的方向与在 Design 侧的方向不一致 → 立即否决该候选对

若锚点池为空，则不否决（正常评分）。

#### 评分

| 分量 | 函数 | 权重 |
|---|---|---|
| 位置分 | `scorePosition` | 0.60 |
| 样式分 | `scoreStyle` | 0.35 |
| 内容分 | `scoreContent` | 0.05 |

**位置分（scorePosition）**：用与 Pass 1 相同的抛物线-高斯曲线（`gaussianParabola`），以两侧画布对角线均值为基准尺度：
- 参考点：`{x: 0.2·diagonal, y: 0.5}`，截断距离：`0.5·diagonal`
- 左上 / 左下双向计算，取较高分：
  - 顶对齐：两侧 textAlign **均为 center** 时取中心点，否则取左上角点，直接欧氏距离
  - 底对齐：以各自"距底边高度"替换 Y 坐标后的欧氏距离（适配底部固定组件）

**样式分（scoreStyle）**：仅看字号（权重 0.405）、字重（权重 0.262）、字色（权重 0.333），缺失维度从权重中排除；全部缺失时返回 0.75。

```
字号分：max(0, 1 − |de.fontSize − hm.fontSize| / 6) × 0.405
字重分：max(0, 1 − |de.fontWeight − hm.fontWeight| / 400) × 0.262
字色分：max(0, 1 − colorDelta(de.fontColor, hm.fontColor) / 510) × 0.333
```

**内容分（scoreContent）**：原文本 Levenshtein 相似度（`1 − editDistance / maxLen`），权重 0.05，不对内容做任何截断或归一化。

#### 阈值与置信度

| 条件 | 结果 |
|---|---|
| `final < 0.55` | 丢弃 |
| `0.55 ≤ final < 0.75` | 接受，`confidence = 'low'` |
| `final ≥ 0.75` | 接受，`confidence = 'medium'` |

对所有通过否决检验的候选按 `final` 降序排序，贪婪一对一分配（先匹配分高的，已占用的 id 跳过）。

### Pass 4 · 区域内文本最优匹配（region-text-optimal）

**触发**：regionContext 存在时，按区域对逐对处理剩余未匹配文本节点

**核心算法（maxWeightTextMatching）**：构建二部图，每条边评分：
```
semanticOrRole * 0.38 + style * 0.24 + geom * 0.22 + order * 0.10 + regionScore * 0.06
```
- `semanticOrRole = max(semantic, roleScore * 0.72, fingerprintGeomOk ? 0.66 : 0)`
- `geom = xScore * 0.40 + yScore * 0.60`（xLimit=0.28，yLimit=0.08）
- `order`：按 Y/X 排序的相对位置差
- ≤ 18 节点时用 bitmask DP 求精确最大权值匹配，否则贪婪

**前置过滤**：
- 弱可见文本（visiblePixelRatio < 0.10 && textStrokeScore < 0.08）须通过 `weakVisibleTextCompatible`（centerDist ≤ 0.16，xDist ≤ 0.12，yDist ≤ 0.12）
- 须通过 `passesTextSemanticGate` 或 `roleScore ≥ 0.85` 或 `fingerprintGeomOk` 之一
- 单位文本：Y 距离 ≤ 0.04；短模糊数字：`isNearSameLineSlot(0.10, 0.045)`

**阈值**：score ≥ 0.58；confidence = score > 0.74 ? 'medium' : 'low'

**兜底（region-text-global-rescue）**：区域内匹配结束后，若仍有剩余未匹配文本，全局再跑一次，score ≥ 0.60

---

### Pass 2.5 · 文本语义角色匹配（text-role）

**触发**：设计侧文本未匹配，`hasUsableText` 满足

**评分（textRoleMatchScore）**：
```
条件：dx ≤ 0.08，dy ≤ 0.08，style ≥ 0.72，且角色兼容（inferTextRole）
xScore * 0.30 + yScore * 0.40 + style * 0.30
```

- `inferTextRole`：按字号/字重推断 `page-title / title / subtitle / body`；含数字信号的文本返回 `body`
- `body` 角色不参与此 Pass

**接受条件**：score ≥ 0.85，或通过 `isStrongTitleSlotMatch`（page-title/title 专用严格门槛）

---

### Pass 3 · 强锚点周边拓扑匹配（anchor-topology）

**核心思路**：两侧 UI 布局结构一致时，同一段文本（锚点）周边的其他节点也应保持相似的相对位置关系。以 Pass 1 产生的可信文本对为参考点，用"目标节点相对于锚点的偏移向量"匹配对端候选。

---

#### 第一步：为每个设计侧节点确定参考锚点（nearbyAnchors）

**锚点池**：Pass 1 输出的所有可信文本对（`topologyAnchors`），含 Pass 2c 中 `topologyScore ≥ 0.86` 的行槽位对。

对每个待匹配的 design 节点，按节点类型选锚策略不同：

| 节点类型 | 选锚规则 |
|---|---|
| 普通文本节点（非短 CJK 标签） | 找距离 < **0.35** 的**最近 1 个**锚点 |
| 容器节点 / 短 CJK 标签（`isShortCjkLabel`） | 找距离 < **0.70** 的所有锚点（短标签位置歧义大，多锚点增强鲁棒性） |

若无满足距离阈值的锚点，该节点不参与 Pass 4。

**距离计算（同侧）**：用 `rect` 绝对坐标的左上角点 `(rect.x, rect.y)` 的欧氏距离，除以设计侧画布对角线 `diagDe = hypot(canvasWidth, canvasHeight)` 归一化：

```
dist = hypot(dn.rect.x − anchor.design.rect.x, dn.rect.y − anchor.design.rect.y) / diagDe
```

> 用左上角点而非中心点，强调同一角对齐的拓扑一致性；用绝对坐标而非 normRect，避免画布尺寸差异带来的距离扭曲。选锚阶段是同侧比较（都是 design 节点），不涉及跨侧换算。

---

#### 第二步：预扫描 + 排序（candidateDesignNodes）

为了让**高分节点优先消费 ArkUI 资源**，Pass 4 先对所有候选 design 节点打一遍预分，排序后再正式匹配。

对每个未匹配的 design 节点（已有有效锚点的）：

1. 用当前 `usedArkui`（Pass 1–2.5 已占用集合）过滤候选，调用 `bestTopologyCandidate` 得到预估最高分 `bestScore` 和最近锚点距离 `bestAnchorDist`
2. 过滤掉 `bestScore ≤ 0.58` 的节点

剩余节点按以下优先级排序（降序）：

```
① bestScore 从高到低
② bestScore 相同时：bestAnchorDist 从小到大（锚点越近越可信）
③ 以上相同时：design 侧有背景色的优先（避免空壳节点抢占）
④ 以上全相同时：design.path 字典序
```

---

#### 第三步：正式匹配（逐节点分配）

按排序后的顺序依次处理每个 design 节点：

1. **不可用集合**：直接使用 `usedArkui`（Pass 1–3.5 等上游已占用集合）。每个 design 节点独立对全部未匹配的 ArkUI 候选评分，不再设 `localUsedArkui` 内锁——由 `selectOneToOnePairs` 最终按 `confidence → matchType优先级 → topologyScore → iou` 排序裁决一对一。

2. **构建候选池**：`candidatePool(dn, arkuiNodes, regionContext, n => !unavailableArkui.has(n.id) && isCompatibleType(dn, n))`

3. **遍历候选 × 锚点，计算拓扑得分**：对候选池中每个 ArkUI 节点 `an`，对节点的每个有效锚点：

   a. 计算**设计侧关系向量**（`rect` 绝对坐标中心点，单位 dp；各自除以本侧对角线归一化）：
   ```
   diagDe = hypot(canvasWidth, canvasHeight)
   designRelation = {
     dx: (centerX(dn.rect) - centerX(anchor.design.rect)) / diagDe,
     dy: (centerY(dn.rect) - centerY(anchor.design.rect)) / diagDe,
     w: dn.rect.w,
     h: dn.rect.h
   }
   ```

   b. 计算**开发侧关系向量**（`rect` 绝对坐标中心点，单位 vp；各自除以本侧对角线归一化）：
   ```
   diagHm = hypot(canvasWidthVp, canvasHeightVp)
   arkuiRelation = {
     dx: (centerX(an.rect) - centerX(anchor.arkui.rect)) / diagHm,
     dy: (centerY(an.rect) - centerY(anchor.arkui.rect)) / diagHm,
     w: an.rect.w,
     h: an.rect.h
   }
   ```

   c. 计算归一化偏移向量之差：
   ```
   relDist = hypot(designRelation.dx − arkuiRelation.dx, designRelation.dy − arkuiRelation.dy)
   ```
   - `relDist > 0.24` → 跳过此候选
   - **各自用本侧对角线归一化后再做差**：设计稿总高度往往大于开发侧（如 947dp vs 809vp），直接比较绝对 dy 会有系统性偏差；先归一化消除画布比例影响，同一视觉位置的节点 relDist 趋近于 0

   d. 通过 relDist 门控后，计算综合得分（`topologyMatchScore`）：
   ```
   relScore     = max(0, 1 − relDist / 0.22)          权重 0.38
   sizeScore    = (wRatio + hRatio) / 2                权重 0.18（文本对固定 1.0，wRatio/hRatio 用 normRect）
   typeScore    = dn.type === an.type ? 1 : 0.68       权重 0.10
   textScore    = textStyleSimilarity(dn, an)          权重 0.10（容器对固定 0.70）
   semanticScore = textSemanticSimilarity / roleScore   权重 0.14（容器对固定 0.70）
   iouScore     = computeIoU(dn.normRect, an.normRect) 权重 0.06（文本对固定 0）
   regionScore  = regionAffinity(dn, an, ctx)          权重 0.04

   finalScore = relScore*0.38 + sizeScore*0.18 + typeScore*0.10 +
                textScore*0.10 + semanticScore*0.14 + iouScore*0.06 + regionScore*0.04
   ```

   **跨侧 y 位置比较**（单位文本/短数字门控）：不使用 normRect，改用绝对坐标 + 顶底双向对齐：
   ```
   absYDist = min(
     |centerY(dn.rect) - centerY(an.rect)|,                            // 顶部对齐
     |(canvasHeight - centerY(dn.rect)) - (canvasHeightVp - centerY(an.rect))|  // 底部对齐
   ) / diagonal
   ```
   取两者最小值再除以 `diagonal`，适配设计稿比开发侧更高的场景（底部固定组件从底端量起更准确）。

   文本对额外：须先通过 `passesTextSemanticGate` 或 `roleScore ≥ 0.85` 或 `slotFallback` 门控，否则直接 0 分。

   e. 取所有候选中最高得分者为 `best`

4. **决策**：`best.score > 0.58` → 生成 pair（不添加 localUsedArkui，允许后续节点生成重复候选，由 `selectOneToOnePairs` 最终裁决）

   - `score > 0.72` → `confidence = 'medium'`；否则 `confidence = 'low'`
   - 此 Pass 不标记 `usedArkui.add()`（即不独占 ArkUI id），下游 Pass 5–7 仍能看到这些候选

---

#### 局限性说明

Pass 4 的准确性依赖锚点在两侧的**相对位置一致**。若锚点文本在设计稿和开发稿之间存在系统性坐标偏移（如 ArkUI 实现中内容区整体下移），该偏移量会直接传导到 relDist 计算：真正位置准确的候选可能因 relDist 偏大而落败，而坐标恰好与偏移方向一致的"错误"候选反而 relDist 更小，胜出匹配。Pass 5c（container-iou）此时无法补救，因为 Pass 4 已将 design 节点标记为 `matchedDesignIds`。

> **已缓解的问题**：两侧画布高度不同（如设计稿 947dp、开发侧 809vp）导致的系统性 dy 偏差，通过在 `relationToAnchor` 中各自除以本侧 diagonal 归一化得到修正。顶/底双向对齐机制进一步处理了底部固定组件的位置偏移。
>
> **「偏移 ≈ 行间距」的等距图标列整列错位**（系统偏移量与行间距接近时 relDist「差一格更吻合」）已由 **Pass 4 前置 · 同行强锚点桥接** 解决——这类列在桥接阶段就被同行锚点锁定，不再进入 Pass 4 的 relDist 评分。

---

### Pass 3.5 · 同行同类列表顺序匹配（list-index）

**触发条件（全部满足）**：

1. 两侧能各自识别"同行 + 同 rawType + 相近 w（容差 0.005）+ x 不重叠"的列表，且 ≥ 2 元素
2. 两侧列表整体 Y 中心差 ≤ 0.02
3. 两侧列表**上方**均存在已匹配 pair（design 底边 ≤ 列表 top，arkui 同理）
4. 两侧列表**下方**均存在已匹配 pair（design 顶边 ≥ 列表 bottom，arkui 同理）
5. 首节点 IoU ≥ 0.40

**匹配策略**：两侧按 x 升序对齐前 `min(N_d, N_a)` 个，`confidence='high'`，直接锁定

**注意**：本 Pass **不跳过已匹配节点**，高优先级可覆盖前序 Pass 的错配

---

### Pass 5（文本）· 文本位置回退匹配（text-position）

**触发**：设计侧文本未匹配，`hasUsableText` 满足

**前置拦截（allowsTextPositionFallback）**：
- 通过 `passesTextSemanticGate` → 允许
- 任一侧含 `hasNumericSignal`（含数字/单位词） → **拒绝**
- 任一侧是 `isAmbiguousUnitText` → **拒绝**
- 两侧 `textFieldType` 不同 → 拒绝
- 两侧均为 label 且最短文本 ≥ 5 字符 → 允许（长描述性 mock 文本）

**Y 轴限制（yLimit）**：
- `passesTextSemanticGate` 通过：yLimit = **0.05**
- 否则：yLimit = **0.04**；`dx > 0.25` 直接跳过

**评分**：`yScore * 0.45 + xScore * 0.20 + semantic * 0.15 + style * 0.15 + regionBonus * 0.05`

**阈值**：score > 0.35 → confidence = 'medium'

---

### Pass 5b · 数字槽位匹配（numeric-slot）

**触发**：设计侧文本未匹配，且 `textFieldType = 'number'`；ArkUI 候选也须为 `'number'`

**条件**：dy ≤ 0.08，dx ≤ 0.20，`textStyleSimilarity ≥ 0.70`

**评分**：`(1 - dy/0.08) * 0.55 + (1 - dx/0.20) * 0.25 + styleScore * 0.20`

**阈值**：score > 0.40 → confidence = 'medium'

### Pass 5（容器）· 容器 IoU 匹配（container-iou）

**触发**：设计侧 `type='container'`，未匹配

**阈值**：
- 有视觉装饰（`hasVisualDecoration`）：IoU > **0.40** → confidence = 'high'
- 无视觉装饰：IoU > **0.60** → confidence = 'medium'

**注意**：本 Pass 不标记 `usedArkui`，允许后续产生重复候选，由 `selectOneToOnePairs` 最终裁决

---

### Pass 6 · 视觉容器几何匹配（container-geometry）

**触发**：设计侧满足 `isRenderableNonTextNode`（type=container 且有视觉装饰或 rawType 为内在视觉类型），未匹配

**内在视觉 rawType 集合**：`image, button, search, searchfield, symbolglyph, circle, ellipse, rect, vector, boolean_operation`

**评分**：`score = iou * wRatio * hRatio * (1 + regionBonus * 0.25)`

**阈值**：IoU > **0.55** → confidence = 'medium'；本 Pass 不标记 `usedArkui`

---

### Pass 6.5 · 空间担保匹配（spatial-bracket）

**触发**：设计侧 `type='container'` 未匹配，且在 `selectOneToOnePairs` 预览结果中能找到左右夹住的担保对

**担保条件（全部满足）**：
1. 左右各至少一个担保 pair，其设计侧节点与目标节点同行（Y 中心差 < 0.08）
2. 担保 pair 的 ArkUI 侧也与候选节点同行（< 0.08）
3. 左担保设计侧右边 ≤ 目标中心 X，右担保设计侧左边 ≥ 目标中心 X；ArkUI 侧同理夹住候选
4. 两个担保 pair 的设计侧互相同行（Y 差 < 0.06），ArkUI 侧同理
5. 候选节点 Y 中心差 ≤ 0.05；最小尺寸比 ≥ 0.5

**评分**：`wRatio * 0.5 + hRatio * 0.5`；confidence 取两担保对中较低的值

---

### Pass 7 · Rescue 兜底（rescue-iou）

**触发**：设计侧节点未匹配，类型兼容（`isCompatibleType`）

**文本对额外限制**：须通过 `passesTextSemanticGate`；弱可见文本须满足 `isWeakRescueTextAllowed`（label 类须 semantic ≥ 0.55，number 类须 `numericTextCompatible`，其他须文本完全相同）

**阈值**：IoU > **0.25** → `confidence = 'low'`

### Post · 最终一对一选取（selectOneToOnePairs）

所有 Pass 输出合并，经 `isAcceptablePair` 过滤后按以下优先级排序，保证每节点只出现在一个 pair：

```
pairPriority = confidenceScore + anchorScore + matchTypePriority + topologyScore * 10 + iouScore
```

|---|---|
| confidenceScore | high=300 / medium=200 / low=100 |
| anchorScore | isAnchor=25 / 否=0 |
| iouScore（非文本对） | iou * 8 |

**同分裁决**：两侧 `backgroundColor` 是否一致 → Design 节点 path 字典序 → ArkUI 节点 path 字典序

**matchType 优先级**：

|---|---|

| region-text-optimal | 34 |
| text-role | 30 |
| anchor-topology | 24 |
| dynamic-text-slot | 22 |
| text-row-slot | 21 |

| numeric-slot | 19 |
| list-index | 18.5 |

## 三、文本语义辅助函数

### normalizeText

1. 全部转小写、去首尾空格
2. 含 CJK 或 `\d°%/` 的文本：去除所有空白
3. 其他：多空格压缩为单空格

### textFieldType

按正则优先级返回类型标签：

| 类型 | 示例 |
|---|---|
| `time` | `12:30`、`下午 3:00`、`现在` |
| `weekday` | `周一`、`星期三` |
| `duration` | `3小时30分钟`、`45秒` |
| `percent` | `78%`、`-3.5%` |
| `temperature` | `23°`、`-5.5°` |
| `number` | `42`、`1024`、`3.14km`、`100步` |
| `date` | `3月15日`、`2024年1月` |
| `label` | 其他（含数字但不符合上述格式的混合文本也归此类） |

### textSemanticSimilarity

```
ta === tb → 1
双侧 label + CJK + 共同后缀 → suffixBoost
typeScore = (类型相同 && 非 label && 数字时兼容) ? 0.72 : 0
return max(typeScore, prefixScore, suffixScore, charJaccard * 0.45 + bigramJaccard * 0.55)
```

### textStyleSimilarity

| 字段 | 容差 | 权重 |
|---|---|---|
| fontSize | ±6（线性衰减） | 0.34 |
| fontWeight | ±400（线性衰减） | 0.22 |
| fontColor | 欧氏距离 ÷ 180 | 0.28 |
| textAlign | 不同时得 0.55 | 0.16 |

仅参与有值的字段；全部缺失时返回 0.75。

### passesTextSemanticGate

允许非完全相同文本进入位置/拓扑匹配的语义门槛：
- 任一侧为空或完全相同 → true
- 任一侧为 number：双侧同类且 `numericTextCompatible`
- 双侧均含 CJK 且最短 ≤ 4 字符：semantic ≥ 0.24
- 其他 label 对：semantic ≥ 0.22

### textAttributeFingerprint

将文本节点样式量化为可比较的指纹：`type | fontSize(步长1) | fontWeight(步长100) | fontColor | textAlign | letterSpacing(步长0.25)`

两节点指纹相同，且 Y 中心差 ≤ 0.03 且 X 中心差 ≤ 0.06 时，`fingerprintGeomOk=true`，可替代语义门槛通过 Pass 3。

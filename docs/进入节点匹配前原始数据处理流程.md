# 进入节点匹配前原始数据处理流程

> 说明：本文只整理“原始数据进入节点匹配之前”的处理流程，不展开具体的匹配 Pass。  
> 当前工程里，数据会先经过 `runCheck` 的解析、可见性预处理和统一过滤，最后才把 `designVisibleNodes` / `arkuiVisibleNodes` 送入 `matchNodes`。  
> 进入 `matchNodes` 之后，还会先做图标碎片标记、可见性遮挡标记、区域切分和视觉特征准备，然后才开始具体的匹配 Pass。

在进入两个大标题之前，先给一个公共结论：

1. 设计侧和开发侧都会先被解析成统一的 `UnifiedNode` 结构。
2. 然后会做像素级可见性检查；如果提供了图片，还会进一步做视觉层的遮挡判断。
3. 开发侧还会额外跑 OCR 语义可见性校验，设计侧不跑 OCR。
4. 最终统一通过 `isPipelineVisibleNode` 过滤，留下真正允许进入节点匹配的数据。
5. 真正开始各个匹配 Pass 之前，匹配器还会做一层前置准备：
   - 设计侧图标碎片标记
   - 设计侧 / 开发侧可见性遮挡标记
   - 页面区域切分
   - 可选的视觉特征提取

## 开发侧（ArkUI）

开发侧的输入是 `arkui.json`，如果有截图，还会同时使用 `arkui.png`。  
这一侧的特点是：树结构更深、框架节点更多、文本既可能来自 `content`，也可能来自 `accessibilityText`，因此前置处理比设计侧更复杂。

### 步骤1：读取根配置并计算画布基准

1. 从 `arkuiJson.content` 里读取根信息。
2. 取 `$resolution` 作为分辨率，默认值是 `3.5`。
3. 取 `width` / `height` 作为整张画布的像素尺寸，默认分别是 `1316` 和 `2832`。
4. 通过 `canvasWidthPx / resolution`、`canvasHeightPx / resolution` 得到 vp 画布尺寸。
5. 后续所有 ArkUI 节点都会统一到这个 vp 坐标系里。

### 步骤2：递归遍历组件树

1. 从根节点开始递归，初始路径是 `[0]`。
2. 每个节点都会携带三个关键状态：
   - `inheritedVisible`：父链传下来的可见性。
   - `path`：当前节点在树中的路径。
   - `hiddenFrameworkAncestor`：是否存在“隐藏的框架祖先”。
3. 框架节点会被识别出来，包含 `root`、`JsView`、`Navigation`、`NavBar`、`NavigationContent`、`Divider`、`ScrollBar`、`NavBarContent`、`NavigationMenu`、`TitleBar`、`ToolBar`、`TabBar`、`BackButton`、`__Common__` 等。
4. `Blank` 节点只当作布局占位，不会作为可视节点进入后续流程，但它的子节点仍然会继续遍历。
5. `Span` 节点本身没有独立布局 rect，会跳过自身，但仍继续遍历子节点。

### 步骤3：计算节点的可见性传递规则

1. 先读取节点自身属性：
   - `visibility` 不能是 `Visibility.None` 或 `Visibility.Hidden`。
   - `opacity` 不能是 0。
2. 当前节点的可见性由父链可见性和自身可见性共同决定。
3. 子节点是否继承可见性，会区分“框架节点”和“业务节点”：
   - 框架节点即使自己不可见，也可能继续把可见性传给子节点。
   - 非框架节点如果自己不可见，通常会把不可见状态往下传。
4. 如果某个隐藏框架节点下面还挂着文本，则这条祖先关系会被标成 `hiddenFrameworkAncestor`，后面会专门拦截。
5. 开发侧的可见性概念要按“整棵子树”理解：
   - 某个节点如果 `visibility = Visibility.Hidden` 或 `opacity = 0`
   - 这个节点本身及其后代节点都不应进入后续可见性集合
   - 也就是这条分支在概念上会被整棵过滤掉

### 步骤4：剪掉整棵子树

1. 开发侧这里有一个独立的剪枝判定函数层，目前由 `getArkuiSubtreePruneReason(type, attrs, rect, canvasWidthVp, canvasHeightVp)` 承接。
2. 这一步不是单一规则，而是一个可继续扩展的规则集合。
3. 当前已经落地的整棵子树剪枝规则是：
   - `visibility === 'Visibility.Hidden'`
   - `opacity === 0`
   - 超出根节点边界的整棵子树
4. 其中“超出根节点边界”的判断规则是：
   - `x > rootNode.w`
   - `y > rootNode.h`
   - `x + w <= 0`
   - `y + h <= 0`
5. 这里的 `x`、`y`、`w`、`h` 都是基于 `$rect` 计算出来的几何值。
6. 根节点的左上角固定是 `(0, 0)`。
7. 一旦命中任一规则，当前节点不会进入统一节点构建。
8. 这个节点的后代也不会再继续递归遍历。
9. 这一步是“强过滤”，不是普通的节点级不可见标记。
10. 开发侧剪枝规则在实现上是一个可扩展集合，当前统一由 `getArkuiSubtreePruneReason(type, attrs, rect, canvasWidthVp, canvasHeightVp)` 管理。
11. 规则表如下：

| 规则类别 | 判断条件 | 剪枝范围 | 说明 |
| --- | --- | --- | --- |
| 可见性隐藏 | `visibility === 'Visibility.Hidden'` | 整棵子树 | 当前节点和所有后代都不再进入统一节点构建 |
| 透明度为 0 | `opacity === 0` | 整棵子树 | 当前节点和所有后代都直接剪掉 |
| 超出边界 | `x > rootNode.w` 或 `y > rootNode.h` 或 `x + w <= 0` 或 `y + h <= 0` | 整棵子树 | `x/y/w/h` 都基于 `$rect` 计算，根节点左上角固定是 `(0, 0)` |
| 特定组件 | `type === 'leftarrow'` 或 `type === 'rightarrow'` | 整棵子树 | `type` 取 `$type` 的全小写值 |

12. 这些规则命中任意一条，就会直接阻断递归，不再继续遍历后代。

### 步骤5：只过滤当前节点，不影响后代

1. 这一层是“只过滤自身节点”的规则层，不做整棵子树剪枝。
2. 这类规则命中时，只会跳过当前节点的统一节点构建。
3. 当前节点的后代仍然会继续递归遍历。
4. 这一层适合放那些“当前节点不参与匹配，但它的子节点仍然要保留”的规则。
5. 目前这里的核心边界是：不改递归，不改子树，只改当前节点是否输出。

6. 当前开发侧已经落地的自身节点过滤规则是：
   - 节点类型（`$type` 转小写）属于 `jsview, stack, column, row, list, listitem, group, __common__, gridcol, gridrow, blank, spacer`
   - 且节点没有背景色，或者背景色以 `#00` 开头
   - 或者节点自身宽度 `node.w > rootNode.w * 3`
7. 这里的背景色判断使用的是 ArkUI 的 ARGB 格式。
8. 这类规则只过滤当前节点，不会影响子节点继续递归。

### 步骤6：解析 rect 并统一成 vp 坐标

1. 对非框架、非 root、非 `Span` 节点，先读取 `$rect`。
2. 如果 `$rect` 缺失，则不生成当前节点，但继续递归它的子节点。
3. 如果 `$rect` 存在，就用 `toVpRect` 按分辨率换算成 vp 坐标。
4. 这一步会把原始 px 坐标统一成可比较的 vp 几何信息。

### 步骤7：筛掉没有内容的文本节点

1. 对文本类型节点，只要 `content` 和 `accessibilityText` 都为空，就不会进入统一节点。
2. 文本内容收集规则是 `content || accessibilityText`。
3. 也就是说，ArkUI 的文本节点不是“只要类型是 Text 就保留”，而是必须真的有可用文本内容。
4. 这一步能过滤掉大量布局壳文本和空文本容器。

### 步骤8：生成统一节点结构

1. 满足条件的节点会被转换成统一结构 `UnifiedNode`。
2. 字段会统一写入：
   - `id`：优先用 `$ID`，没有时再退化成随机值。
   - `source`：固定为 `arkui`。
   - `type`：映射后的统一类型。
   - `rawType`：原始类型小写。
   - `name`：保留组件原始类型名。
   - `rect`：vp 坐标。
   - `normRect`：归一化坐标，按画布宽高除法计算。
   - `visible`：这里进入统一节点的默认值为 `true`。
   - `style`：ArkUI 样式抽取结果。
3. 如果是文本节点，还会补上 `textContent`，取值优先级是：
   - `content`
   - `accessibilityText`

### 步骤9：抽取并标准化样式信息

1. `opacity` 会转成数字。
2. 背景色会从 `fill`、`foregroundColor`、`backgroundColor` 里读取，并过滤掉透明色。
   - 对 `Circle`、`Ellipse`、`Rect`，优先取 `fill`，再回退到 `foregroundColor`
   - 其它节点则使用 `backgroundColor`
3. 圆角会从 `borderRadius` 解析成统一结构。
4. 描边会从 `borderWidth`、`borderColor`、`borderStyle` 统一出来。
5. 模糊会拆成：
   - `backdropBlur`
   - `blur`
6. 阴影会统一成：
   - `color`
   - `radius`
   - `offsetX`
   - `offsetY`
7. 内边距、外边距都会被解析成标准 padding 结构。
8. `Row`、`Column`、`Flex` 的 `space` 会被转成 `itemSpacing`。
9. 渐变背景会把线性、扫光、径向三类统一成标准 gradient。
10. 对文本节点，还会补齐：
    - `fontWeight`
    - `fontColor`
    - `fontFamily`
    - `lineHeight`
    - `letterSpacing`
    - `textAlign`
    - `fontSize`
11. 如果存在 `actualFontSize`，会优先把它作为真实字号；原始声明字号会保存在 `declaredFontSize`。

### 步骤10：确定节点类别

1. `Text`、`Span`、`Button`、`Search`、`SearchField` 会被归成 `text`。
2. `Row`、`Column`、`Flex`、`Stack`、`RelativeContainer`、`GridRow`、`GridCol`、`Scroll`、`List`、`ListItem`、`Tabs`、`TabContent`、`Swiper` 会被归成 `container`。
3. `Image`、`SymbolGlyph` 会被归成 `image`。
4. `Circle`、`Ellipse`、`Rect` 会被归成 `shape`。
5. 其它类型会归成 `other`。

### 步骤11：把图标和语义资产提前折叠

1. ArkUI 本身没有单独做“图标 union 节点”，但设计侧会做；开发侧这里需要先理解设计侧的对应策略，才能在后续匹配时对齐。
2. 对 ArkUI 来说，真正重要的是把 `SymbolGlyph`、文本、容器这些语义明确的节点保留下来，不做额外的语义折叠。
3. 这意味着开发侧的前处理更偏“保真”，尽量把原始组件层保留下来，交给后面的可见性和匹配流程判断。
4. 当前匹配入口还会对设计侧 `image` 节点做一次图标碎片标记，后续 `isMatchableNode` 会直接把 `iconFragment` 节点排除掉。

### 步骤12：像素级可见性检查

1. 如果提供了 `arkui.png`，会调用 `annotatePixelVisibility`。
2. 这一步只检查开发侧节点。
3. 它会先把 PNG 解码，再对候选节点做采样。
4. 候选节点范围是：
   - 可见节点
   - 没有被前一步标成 `visualOccluded`
   - 尺寸足够大
   - 文本节点必须有真实文本
   - 非文本节点必须是图片、形状，或者带装饰信息的容器
5. 对每个节点，会计算一组像素可见性指标：
   - `visiblePixelRatio`
   - `meanLumaDelta`
   - `meanColorDelta`
   - `textStrokeScore`
6. 文本节点的判定阈值更严格：
   - 如果可见像素比例极低，且颜色/亮度差也很小，就会被认为几乎不可见。
7. 图片和其它节点也有各自的阈值。
8. 一旦判定为不可见，会写入：
   - `pixelInvisible = true`
   - `visualOccluded = true`
   - `visualOcclusionReason = 'pixel-invisible'`
9. 这一步完成后，现有的遮挡判断会继续结合 paint 顺序、后绘制图层和交叠关系来标记 `visualOccluded`。

### 步骤13：OCR 语义可见性检查

1. 仍然是只作用于开发侧。
2. `annotateTextOcrVisibility` 会使用 `arkui.png` 做 OCR。
3. 内部用 Tesseract，语言是中文简体 `chi_sim`，页面分割模式是稀疏文本模式。
4. OCR 输出会被扁平化成统一条目，保留：
   - `text`
   - `rect`
   - `confidence`
   - `kind`
5. OCR 结果支持两种来源：
   - 直接传入 OCR 结果
   - 传入图片缓冲区后自动识别
6. 对每个文本节点，会用以下条件找最优 OCR 命中：
   - 文本相似度至少达到 0.7
   - 节点框和 OCR 框的重叠度至少达到 0.14
   - OCR 置信度至少达到 0.25
7. 最优命中分数由三部分组成：
   - 文本相似度
   - 框重叠度
   - OCR 置信度
8. 找到命中后，会写入 `ocrVisibility.visible = true`；找不到则写入 `ocrVisibility.visible = false`。
9. 这一步的目的不是做匹配，而是确认“这个文本在图上是否真的存在、是否真的可见”。

### 步骤14：进入统一可见性过滤

1. 开发侧节点最终都会经过 `isPipelineVisibleNode`。
2. 这个门禁会按顺序检查：
   - 节点本身必须 `visible`
   - 不能是 `pixelInvisible`
   - 文本节点不能挂在 `hiddenFrameworkAncestor` 下面
   - 不能是 `visualOccluded`
   - 文本节点如果 OCR 明确判定不可见，而且文字笔画分数和可见像素比例又满足特定危险条件，也会被过滤掉
3. 通过这一步后，才会形成 `arkuiVisibleNodes`。
4. 只有这批节点会进入后面的节点匹配。
5. 被这一步过滤掉的节点不会再进入后续 `pairs`、`diffs`、`unmatched`、节点树和前端高亮。
6. 这里的“过滤掉”包括该节点自身和它的后代节点，不只是单个节点被隐藏。

### 步骤15：开发侧最终输出

1. 原始数据经过上述步骤后，最终输出的是：
   - `arkuiResult.nodes`
   - `arkuiVisibleNodes`
2. `arkuiVisibleNodes` 才是节点匹配真正使用的数据源。

## 设计侧（Design / Figma）

设计侧的输入是 `design.json`，如果有截图，还会同时使用 `design.png`。  
这一侧和开发侧不同的地方在于：它是扁平化视觉节点列表，不需要递归树遍历，但会有语义资产折叠、零透明过滤和图标 union 合并。

### 步骤1：读取设计节点与画布尺寸

1. 从 `designJson.data` 里读取所有节点。
2. 先找根节点，也就是 `path.length === 1` 的节点。
3. 根节点的 `rect.w`、`rect.h` 作为画布宽高。
4. 如果根节点不存在，就使用默认画布：
   - 宽度 `360`
   - 高度 `792`
5. 后续所有设计节点都会按这个画布基准做归一化。

### 步骤2：预先收集会影响过滤的路径集合

1. 先收集所有 `BOOLEAN_OPERATION` 节点的 `path`。
2. 这些路径会被当成布尔合成区域，后面其子孙节点会被跳过。
3. 再收集所有 `opacity <= 0` 的节点路径。
4. 这些零透明节点本身以及其子孙都会被跳过。
5. 这一步是设计侧很重要的前置“黑名单”准备。

### 步骤3：识别需要折叠成语义资产的节点

1. 设计侧会先扫描出一批可能要折叠成 `image` 的语义资产根节点。
2. 识别条件主要包括：
   - 节点类型必须是 `FRAME`、`GROUP`、`VECTOR`、`BOOLEAN_OPERATION`
   - 名字不能为空
   - 名字命中插画或图标模式
3. 插画类节点会命中：
   - `#illustration`
   - `illustration`
   - `插画`
4. 图标类节点会命中：
   - `icon`
   - `ic_`
   - `图标`
5. 但是图标类还会额外检查尺寸，必须是“图标大小”才会被认作语义资产。
6. 设计侧也有一层独立的子树剪枝判定函数，目前由 `getDesignSubtreePruneReason(...)` 承接，用来统一管理语义资产、布尔运算和零透明这几类剪枝规则。
7. 这层判定也包含“超出根节点边界”的整棵子树剪枝。

### 步骤4：排除不该折叠的语义资产

1. 如果某个候选语义资产下面还有文本子孙，就不会折叠。
2. 如果某个插画容器面积过大，并且很像完整插画页面，也不会折叠。
3. 这里的“大插画容器”判断会看：
   - 节点名字是否明显是插画
   - 节点面积占画布比例是否足够大
   - 下面是否有足够多文本子孙
4. 这样可以避免把复杂的插画页面误当成单个图标。

### 步骤5：遍历扁平节点并做主过滤

1. 遍历顺序直接使用 `designJson.data` 的顺序，也就是 paint order。
2. 只有这些类型会进入后续处理：
   - `TEXT`
   - `FRAME`
   - `RECTANGLE`
   - `ELLIPSE`
   - `GROUP`
   - `VECTOR`
   - `BOOLEAN_OPERATION`
3. 文本节点如果没有内容，会直接跳过。
4. 位于语义资产子孙中的节点会跳过。
5. 位于布尔运算子孙中的节点会跳过。
6. 自身或祖先透明度为 0 的节点会跳过。
7. 这一层之后，还可以放“只过滤自身节点”的规则：
   - 这类规则只跳过当前节点，不影响后代继续遍历
   - 适合用来处理当前节点不参与输出，但它的子层级仍然要保留的情况
8. 设计侧剪枝规则在实现上统一由 `getDesignSubtreePruneReason(...)` 管理，规则表如下：

| 规则类别 | 判断条件 | 剪枝范围 | 说明 |
| --- | --- | --- | --- |
| 语义资产子孙 | `node.path` 属于 `semanticAssetPaths` 的后代 | 整棵子树 | 语义资产已被整体折叠，不再作为普通视觉节点参与后续流程 |
| 布尔运算子孙 | `node.path` 属于 `booleanOperationPaths` 的后代 | 整棵子树 | `BOOLEAN_OPERATION` 子树不再重复作为普通节点输出 |
| 透明度为 0 | `opacity === 0` 或后代命中 `zeroOpacityPaths` | 整棵子树 | 当前节点及其子孙直接剪掉 |
| 超出边界 | `x > canvasWidth` 或 `y > canvasHeight` 或 `x + w <= 0` 或 `y + h <= 0` | 整棵子树 | 基于 `rect` 的节点边界超出画布即不参与后续处理 |

9. 这些规则只负责“是否还保留这棵子树”，不负责具体的节点类型映射或样式抽取。

### 步骤6：只过滤当前节点，不影响后代

1. 这一层是“只过滤自身节点”的规则层，不做整棵子树剪枝。
2. 这类规则命中时，只会跳过当前节点的统一节点构建。
3. 当前节点的后代仍然会继续递归遍历。
4. 这一层适合放那些“当前节点不参与输出，但它的子节点仍然要保留”的规则。
5. 目前这里的核心边界是：不改遍历，不改子树，只改当前节点是否输出。

### 步骤7：生成统一节点结构

1. 通过主过滤的节点会被转换成 `UnifiedNode`。
2. 字段统一写入：
   - `id`：来自 `guid`
   - `source`：固定为 `design`
   - `type`：统一类型
   - `rawType`：原始类型小写
   - `name`：节点名
   - `path`：树路径
   - `paintIndex`：遍历顺序
   - `rect`：原始几何尺寸
   - `normRect`：归一化几何尺寸
   - `visible`：来自 `state.visible !== false`
   - `style`：设计样式抽取结果
3. 如果是文本节点，还会写入 `textContent = content`。
4. 设计侧不使用 `accessibilityText` 作为兜底。

### 步骤8：把语义资产直接降级为 image

1. 对于被判定为语义资产根节点的节点，统一类型会直接改成 `image`。
2. 同时会补两个标记：
   - `semanticAsset = true`
   - `style.semanticAsset = 'icon'` 或 `'illustration'`
3. 这样做的目的，是把复杂矢量图元提前合并成一个可匹配的视觉单元。
4. 后续匹配时，设计侧图标不会被拆成一堆 path 逐个去比，而是尽量作为整体处理。

### 步骤9：补充设计样式信息

1. `opacity` 会被直接保留。
2. 背景色会从 `background` 里找 `SOLID`。
3. 如果存在渐变，会把线性、径向、角向渐变统一成标准结构，保存：
   - `type`
   - `angle`
   - `stops`
4. 圆角会从 `borderRadius` 解析成标准结构。
5. 描边会从 `border` 列表里取第一项，记录：
   - `color`
   - `width`
   - `style`
6. 模糊会拆成：
   - `backdropBlur`
   - `blur`
7. 阴影会记录：
   - `color`
   - `radius`
   - `offsetX`
   - `offsetY`
8. 文本节点还会记录：
   - `textAlign`
   - `fontSize`
   - `fontWeight`
   - `fontFamily`
   - `lineHeight`
   - `letterSpacing`
   - `fontColor`
9. 容器节点还会记录：
   - `padding`
   - `itemSpacing`

### 步骤10：图标碎片合并成 union 节点

1. 设计侧会额外跑一次图标 union 合并。
2. 先找出所有“path-like layer”，也就是名称像 `path` / `路径` 的图层。
3. 这些图层必须满足：
   - source 是 `design`
   - 类型是 `image`
   - 可见
   - 有有效 rect
   - 宽高都大于 0
4. 再按父路径分组，找同一父节点下面的多个碎片。
5. 只有当碎片数量至少为 2，且它们的 union rect 像一个图标大小时，才会创建 union 节点。
6. 图标大小的判断大致是：
   - 最小边不小于 4
   - 最大边不大于 48
   - 宽高比不能太夸张
7. 生成的 union 节点会：
   - id 追加 `::icon-union`
   - 名字追加 `::icon-union`
   - `paintIndex` 设为子碎片最大值加 `0.1`
   - `style.iconUnion = true`
   - `iconUnion = true`
   - `iconFragmentIds` 记录所有碎片 id
8. 同时，原始碎片节点会被标记：
   - `iconFragment = true`
   - `iconUnionParentId = <union id>`
9. 这样后面做可见性和匹配时，就不会把一个图标拆成多个零件去误判。

### 步骤11：像素级可见性检查

1. 如果提供了 `design.png`，会调用 `annotatePixelVisibility`。
2. 这一步只检查设计侧节点。
3. 它会先把图片解码，再对符合条件的节点做采样。
4. 候选节点包括：
   - 可见节点
   - 没有被标成 `visualOccluded`
   - 具备 rect
   - 尺寸足够大
   - 文本节点必须有文本内容
   - 非文本节点必须是图片、形状，或者带装饰的容器
5. 对每个节点，会计算：
   - `visiblePixelRatio`
   - `meanLumaDelta`
   - `meanColorDelta`
   - `textStrokeScore`
6. 如果节点几乎没有可见像素，就会被标成 `pixelInvisible`，同时写入 `visualOccluded`。

### 步骤12：进入统一可见性过滤

1. 设计侧节点最终也会经过 `isPipelineVisibleNode`。
2. 检查顺序和开发侧一致：
   - 必须 `visible`
   - 不能 `pixelInvisible`
   - 不能 `visualOccluded`
   - 文本节点不需要检查 `hiddenFrameworkAncestor`，因为设计侧没有这类框架树状态
   - 文本节点如果 OCR 不可见的特殊条件满足，也会被拦截，不过设计侧一般不会走 OCR 流程
3. 通过这一步后，才会形成 `designVisibleNodes`。
4. 只有这批节点会进入后面的节点匹配。
5. 被这一步过滤掉的节点不会再进入后续 `pairs`、`diffs`、`unmatched`、节点树和前端高亮。
6. 当前实现里，`isMatchableNode` 还会继续排除 `iconFragment`、`hiddenFrameworkAncestor`、`visualOccluded`、以及接近整张画布大小的根包裹节点。
7. 当前实现里的“结构容器”判断只认背景色、圆角、描边、阴影、背景模糊和前景模糊，不把渐变单独当成装饰项。

### 步骤13：设计侧最终输出

1. 原始数据经过上述步骤后，最终输出的是：
   - `designResult.nodes`
   - `designVisibleNodes`
2. `designVisibleNodes` 才是节点匹配真正使用的数据源。

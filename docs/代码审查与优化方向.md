### 1.3 nodeMatcher.js

**正确实现：**
- 8-Pass 多轮匹配，覆盖文本精确/语义/拓扑/几何等多种策略
- 区域分割（region segmentation）+ 锚点驱动的候选池缩小
- 双向匹配（arkui-first 交换角色后再取并集）
- 视觉遮挡检测（`annotateVisualOcclusion`），排除被覆盖的文本

**发现问题：**

| # | 位置 | 问题描述 |
|---|------|---------|
| M1 | `isAcceptablePair` | 所有 `container` 类型若无视觉装饰（`isStructuralContainer`），既不进入 `comparableDesignCount`（不计入覆盖率分母），也不输出到最终 pairs——这是对的，但**纯布局容器的 padding/itemSpacing 差异就永远无法被检测到** |
| M2 | `segmentRegions` | 区域切割条件是 `current.length >= 4 && gap > 0.075`，稀疏页面（节点少）永远切不出子区域，退化为单区域，Pass 2 区域文本最优匹配失效 |
| M3 | `matchByAnchorTopology` (Pass 3) | 只基于最近锚点做拓扑匹配，当页面有多个相似重复区块（如 Tab 列表每行）时，拓扑距离相同的多个候选无法区分 |
| M4 | `exactAssignmentForSmallGroups` | DP 用 `usedMask` 作位掩码，`aCount` 上限是 18（防止 `1 << 18` 超 Number 精度）。超过 18 个 ArkUI 候选时退化为贪心，大文本区块可能匹配次优 |
| M5 | `textSemanticSimilarity` | 仅做字符级 Jaccard（charJaccard + bigramJaccard），对数字/日期类文本（如 "18°" vs "22°"）虽通过 `textFieldType` 分类，但同类型的两个不同数值仍会获得 `typeScore=0.72` 被当成合理候选 |
| M6 | `candidatePool` | 当 design 节点所在区域无对应 ArkUI 区域时，回退到全量节点池。在节点数多的 case 中（case7: 1000+节点），回退会导致大量候选计算 |
| M7 | `buildVisualFeatures` | 视觉特征提取（imageFeatures）仅在 `options.visualImages` 存在时才运行，而客户端代码（`api/index.js`）从不传图片用于 visual matching（`shouldUseVisualMatching` 默认 false）。**视觉区域匹配功能在实际运行中从未启用** |

### 1.4 styleComparator.js

**正确实现：**
- 逐属性比对，有容差（`TOLERANCE` 配置）
- 颜色用欧氏距离 + 双阈值（warning/error）
- `diffFontScale` 检测系统字体缩放异常（actualFontSize vs fontSize）
- 梯度渐变色标逐个比对

**发现问题：**

| # | 位置 | 问题描述 |
|---|------|---------|
| C1 | `diffShadow` | **只比对 `shadow.radius`**，`color`、`offsetX`、`offsetY` 均未比对；投影偏移和颜色的差异完全漏检 |
| C2 | `diffColor` | `if (!dv && av)` 分支直接 `return`（静默忽略）——设计稿无背景色但实现添加了背景色的情况**不报错** |
| C3 | `diffPadding` | `if (!dv || !av) return`——设计稿有 padding 但实现不报 padding 时，直接跳过不报差异 |
| C4 | `compareStyles` | `lineHeight` 对比条件是 `if (ds.lineHeight && as_.lineHeight)`——设计有行高、ArkUI 无行高（常见情况），静默跳过 |
| C5 | `compareStyles` | `letterSpacing` 对比条件是 `if (ds.letterSpacing !== null && as_.letterSpacing !== null)`，同样单侧缺失则跳过 |
| C6 | `compareStyles` | `margin` 在 arkuiParser 中已提取到 `style.margin`，但 compareStyles 没有任何 `diffMargin` 调用 |
| C7 | `diffGradient` | 只比对渐变色标颜色，`angle`（方向/角度）未比对——线性渐变方向错误无法检测 |
| C8 | `diffBorder` | `if (!dv && av) return`——实现侧多了描边而设计无描边，静默忽略 |
| C9 | `makeDiff` | `confidence` 字段只透传到 diff 对象供展示，但低置信匹配（`confidence: 'low'`）的 diff 和高置信匹配的 diff 使用相同 severity 规则，不降级处理；低置信误报直接混入 error 列表 |
| C10 | `colorDelta` | 欧氏距离把 alpha 通道（透明度）与 RGB 等权叠加；若两个颜色仅 alpha 不同（如 `#FF303133` vs `#80303133`），颜色感知上区别极大，但欧氏距离可能小于阈值 8 |

### 1.5 imageFeatures.js

**正确实现：**
- 自实现 PNG 解码（无第三方依赖）：支持 RGB/RGBA、非隔行 8-bit
- 区域特征：color histogram（64 bin）+ edge histogram（4 方向）+ dhash（8×8）+ meanLuma
- 特征提取一次后对所有区域复用（不重复解码）

**发现问题：**

| # | 位置 | 问题描述 |
|---|------|---------|
| I1 | `decodePng` | 只支持 PNG（colorType 2/6），不支持 JPEG；上传 JPEG 截图会直接抛错并回退为无视觉特征 |
| I2 | `computeRegionFeatures` | `x1 = normRect.x * canvas.w / canvas.w * image.width`，`canvas.w` 分子分母相消，公式等价于 `normRect.x * image.width`；逻辑正确但冗余书写易误导 |
| I3 | `dhash` | 8×8 dhash 仅 64 bits，对大区域（整屏）分辨率不足，相似但不同的 UI 区域（如两个白底卡片）dhash 完全相同 |
| I4 | 整体 | 视觉特征当前在生产中从未被使用（见 M7），这部分代码实际处于死代码状态 |

if (Math.abs(dv.offsetX - av.offsetX) > TOLERANCE.shadowOffset)

### Sprint 2 — 检测质量提升

**6. 低置信差异降级**

在 `makeDiff` 中增加：
```js
// 低置信匹配的 error → 降为 warning，warning → 降为 info
if (ctx.confidence === 'low') {
  severity = severity === 'error' ? 'warning' : 'info'
}
```

**7. 修复 ArkUI 资源引用颜色误判**

在 `normalizeArkuiColor` 中增加校验：
```js
export function normalizeArkuiColor(hex) {
  if (!hex || typeof hex !== 'string') return null
  // 跳过资源引用字符串
  if (hex.startsWith('resource(') || hex.startsWith('$r(')) return null
  return hex.toUpperCase()
}
```

**8. 渐变 angle 比对**

在 `diffGradient` 中增加角度比对（若两侧均为 linear gradient）。

**9. 反向差异补报**

`diffColor`、`diffBorder` 等函数的 `if (!dv && av)` 分支补充报差异（现在是静默忽略）。

**10. SymbolGlyph 图标内容匹配**

在 Pass 1 之前增加 Pass 0：对 design 侧 `BOOLEAN_OPERATION / VECTOR` 类节点，若 ArkUI 侧存在 `SymbolGlyph` 节点且 `iconContent` 可与 design 名称关联（字符串包含），优先做内容匹配。

---

### Sprint 3 — 视觉特征激活与覆盖率提升

**11. 默认启用视觉特征辅助区域匹配**

当前客户端 `checkCase` 调用不传 `visual=1` 参数，服务端也默认 `VISUAL_MATCHING_ENABLED=false`。建议：
- 在 `checkCase` 中**始终上传图片** buffer 用于区域匹配（已有图片文件，只需在路由中读取并传入）
- 将 `shouldUseVisualMatching` 默认改为 `true`（或改为：有图片文件则启用）

**12. JPEG 支持**

在 `imageFeatures.js` 中增加 JPEG 检测，或引入轻量 JPEG 解码库（`jpeg-js` 等）。

**13. 纯布局容器 padding 检测**

当前 `isStructuralContainer`（无视觉装饰容器）排除出 comparableDesignCount 和 pairs。可以单独建立一条"布局属性检测"路径：对已几何匹配的纯布局容器对，只比 padding/itemSpacing，不比颜色/圆角，差异以独立类别输出。

### Sprint 4 — 设计师体验改造

**14. 三态节点覆盖标注（ImagePanel）**

在 `draw()` 中区分三种节点状态：
- 绿色半透明框：已匹配 + 无差异
- 红/黄框：已匹配 + 有 error/warning
- 灰色虚线框：未匹配

需要前端在接收到 `result` 后，预计算每个节点的差异状态 Map（`nodeId → severity`），传给 ImagePanel。

**15. 差异气泡标注**

点击差异条目时，在 ImagePanel 的高亮矩形上叠加显示属性名和值对比（用 Canvas fillText 绘制），而不只是橙色框。

**16. DiffReport 术语本地化**

在 DiffReport 中增加 matchType 和属性名的中文映射：
```js
const MATCH_TYPE_LABELS = {
  'text-content':        '文本精确匹配',
  'anchor-topology':     '锚点拓扑匹配',
  'rescue-iou':          '几何兜底匹配',
  'region-text-optimal': '区域内最优匹配',
  ...
}
```
属性名也做中文映射（fontSize → 字号，fontColor → 文字颜色 等）。

**17. 未匹配节点可点击高亮**

在 DiffReport 的未匹配节点 Tag 上增加 `@click`，emit 一个新事件 `select-unmatched`，App.vue 监听后高亮 design 图对应区域。

**18. NodeTree 显示 diff 指示**

在 NodeTree 节点行右侧增加 error/warning 数量徽标（预计算每个 designNodeId 的差异聚合）。

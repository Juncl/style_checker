# 前端接口调用场景

> 本文以前端视角，按用户操作场景逐一列出实际发生的接口调用。
>
> **两类后端**：
> - **算法(server)** — 端口 3012，baseURL `/devlint/api`，负责节点解析 / 节点匹配 / 样式对比
> - **后台(mock)** — 端口 3001，baseURL `/mock`，负责交付件 / 页面 / 版本数据管理、登录、打点等

---

## 场景一：页面初始化

**入口**：`ConsistencyView.vue` → `onMounted`

```
1. initApp()  — 内网实现，外网空逻辑（见 init/inner/）
   ├─ checkAuth()         — 登录鉴权（外网：空）
   ├─ mountPixso()        — 挂载 Pixso 渲染器（外网：空）
   ├─ detectIframe()      — 检测 iframe 环境（外网：空）
   ├─ restorePlatform()   — 读取本地缓存，返回上次使用的平台
   └─ loadDeliverable()   — URL 含 deliverableId 时加载历史结果（外网：return null）

2. 分支判断：
   ├─ URL 含 deliverableId/pageId/versionId → 【有参分支】见下方
   └─ 否则 → 【无参分支】继续步骤 3
```

---

### 场景一·有参分支：URL 携带 deliverableId / pageId / versionId

**入口**：`ConsistencyView.vue` → `onMounted`，检测到 URL 含三个参数

**1. [后台(mock)] `getConsistencyCheckDeliverables()`**
- 作用：获取交付件列表，初始化交付件下拉框，并找到当前 deliverableId 对应的交付件作为默认选中项
- 调用处：`api.ts`
- 接口详情：[接口文档 → getConsistencyCheckDeliverables](接口文档.md#getconsistencycheckdeliverables--查询用户交付件列表)

**2. [后台(mock)] `getPagesByDeliverableId(deliverableId)`**
- 作用：获取该交付件下的页面列表，初始化页面下拉框，并从列表中取当前 pageId 对应 page 的 `deviceType`
- 调用处：`api.ts`
- 接口详情：[接口文档 → getPagesByDeliverableId](接口文档.md#getpagesbydeliverableid--根据交付件-id-查询页面列表)

**3. [后台(mock)] `getResultsByPageId(pageId, 1, N)`**
- 作用：获取该页面的版本列表，找到 versionId 对应的版本，取出 `devBase64Data`、`designBase64Data`、`devJsonUrl`、`designJsonUrl`、`problems`
- 调用处：`api.ts`
- 接口详情：[接口文档 → getResultsByPageId](接口文档.md#getresultsbypageid--查询页面下的版本列表)

**4. [算法(server)] `parseDevUpload` + `parseDesignUpload`**（并行）
- 作用：fetch devJsonUrl / designJsonUrl 取回原始 JSON，连同 Base64 图片一起调解析接口，得到两侧节点树和 canvas 信息
- 接口路径：`POST /devlint/api/parse/dev` / `POST /devlint/api/parse/design`

**5. 前端重建 result**
- `diffs`：将每个 problem.data JSON.parse 还原为完整 diff 对象
- `pairs`：从 diffs 中提取唯一 (arkuiNodeId, designNodeId) 对，在步骤 4 的节点树中查找对应节点，组装 `{ arkui, design }` 配对
- `canvas`：来自步骤 4 解析结果
- `stats`：从 diffs 重新统计 errorCount / warningCount
- `unmatchedDesignNodes`：回显场景无法还原，给空数组

**6.** 直接渲染报告页，跳过上传流程

### 场景一·无参分支：正常进入上传页

**3. [算法(server)] `fetchCases(platform)`**
- 作用：获取内置案例列表，渲染左侧案例栏
- 调用处：`api/index.ts`
- 接口路径：`GET /devlint/api/cases?platform={platform}`

**同时**，`ConsistencyView.vue` → `onMounted`：

**4. [后台(mock)] `getConsistencyCheckDeliverables()`**
- 作用：获取当前用户的交付件列表，初始化交付件下拉框
- 调用处：`api.ts`
- 接口详情：[接口文档 → getConsistencyCheckDeliverables](接口文档.md#getconsistencycheckdeliverables--查询用户交付件列表)

## 场景二：切换平台（hmPhone / hmWatch / web）

**入口**：`UploadPage` 平台下拉框 → `onPlatformSwitch` → `ConsistencyView.onPlatformSwitch`

**1. [算法(server)] `fetchCases(platform)`**
- 作用：重新加载当前平台的内置案例列表
- 调用处：`api/index.ts`
- 接口路径：`GET /devlint/api/cases?platform={platform}`

---

## 场景三：上传开发侧 JSON 文件

**入口**：`UploadPage` 开发侧文件拖拽/点击上传 → `onStepPicked({ type: 'arkuiJson', file })`

**1.** 前端本地解析 JSON，检测平台类型（web / hmWatch / hmPhone）

**2. [算法(server)] `fetchCases(platform)`**（仅平台发生变化时触发）
- 作用：刷新案例列表
- 调用处：`api/index.ts`
- 接口路径：`GET /devlint/api/cases?platform={platform}`

**3. [算法(server)] `parseDevUpload(arkuiJson, arkuiImage, platform)`**（开发侧 json + 图片均就绪时触发）
- 作用：解析开发侧节点树，返回 `{ nodes, canvas }`，渲染节点预览层
- 调用处：`api/index.ts`
- 接口路径：`POST /devlint/api/parse/dev`（multipart: arkuiJson, arkuiImage, platform）

**4. [算法(server)] `parseDesignUpload(designJson, designImage, platform)`**（设计侧 json + 图片均就绪时触发）
- 作用：解析设计侧节点树，渲染设计侧预览层
- 调用处：`api/index.ts`
- 接口路径：`POST /devlint/api/parse/design`（multipart: designJson, designImage, platform）

> 上传设计侧 JSON（`type: 'designJson'`）同理，仅触发步骤 4。

---

## 场景四：拖拽上传文件

**入口**：拖拽文件到上传区 → `onDrop → assignFiles`

- 两侧四个文件均已就绪 → 直接进入【场景六：对比分析】流程
- 部分就绪：
  - 开发侧有 json + 图片 → 触发 `parseDevUpload`（同场景三步骤 3）
  - 设计侧有 json + 图片 → 触发 `parseDesignUpload`（同场景三步骤 4）

---

## 场景五：点击内置案例

**入口**：案例列表点击 → `selectCase(caseId)`

**1. [算法(server)] `checkCase(caseId, platform)`**
- 作用：执行节点匹配 + 样式对比，返回完整对比结果
- 调用处：`api/index.ts`
- 接口路径：`POST /devlint/api/check/case/{caseId}?platform={platform}`

**2. [算法(server)] `imageUrl(caseId, type, platform)`** × 2（并行）
- 作用：fetch 开发侧、设计侧图片，构造 Blob URL 用于画布展示
- 调用处：`api/index.ts`
- 接口路径：`GET /devlint/api/cases/{caseId}/image/{arkui|design}?platform={platform}`

**3.** → 触发【场景七：对比完成后自动存档】

---

## 场景六：点击"开始对比"按钮

**入口**：`UploadPage` 开始对比按钮 → `runUpload(platform)`

**1. [算法(server)] `checkUpload(designJson, arkuiJson, designImage, arkuiImage, platform)`**
- 作用：执行节点匹配 + 样式对比，返回完整对比结果
- 调用处：`api/index.ts`
- 接口路径：`POST /devlint/api/check/upload`（multipart: designJson, arkuiJson, designImage, arkuiImage, platform）

**2.** → 触发【场景七：对比完成后自动存档】

## 场景七：对比完成后自动存档

**入口**：场景五、场景六对比成功后，后台静默执行 `submitResult()`，不阻塞主流程

**1. [后台(mock)] `getTeamList()`**
- 作用：获取第一层团队，取 `teams[0].teamId`
- 调用处：`api.ts`
- 接口详情：[接口文档 → getTeamList](接口文档.md#getteamlist--获取团队列表)

**2. [后台(mock)] `getSonListByTeamId(teamId)`**
- 作用：获取子项目列表，取 `sonTeams[0].teamId` 作为 subTeamId
- 调用处：`api.ts`
- 接口详情：[接口文档 → getSonListByTeamId](接口文档.md#getsonlistbyteamid--根据团队-id-获取子项目列表)

**3. [后台(mock)] `addConsistencyCheckDeliverable(subTeamId, name)`**
- 作用：新建一个交付件，返回 deliverableId
- 调用处：`api.ts`
- 接口详情：[接口文档 → addConsistencyCheckDeliverable](接口文档.md#addconsistencycheckdeliverable--新增交付件)

**4. [后台(mock)] `addConsistencyCheckPage(params)`**（**新建模式**：不传 `id`）
- 作用：在该交付件下新建页面并创建首个版本，返回 pageId
- 调用处：`api.ts`
- 接口详情：[接口文档 → addConsistencyCheckPage](接口文档.md#addconsistencycheckpage--新增页面--追加版本)

**5. [后台(mock)] `getResultsByPageId(pageId, 1, 1)`**
- 作用：取该页面的第一条版本，获得 versionId
- 调用处：`api.ts`
- 接口详情：[接口文档 → getResultsByPageId](接口文档.md#getresultsbypageid--查询页面下的版本列表)

**6. [后台(mock)] `getPagesByDeliverableId(deliverableId)`**
- 作用：拉取最新页面列表，刷新标签栏页面下拉框
- 调用处：`api.ts`
- 接口详情：[接口文档 → getPagesByDeliverableId](接口文档.md#getpagesbydeliverableid--根据交付件-id-查询页面列表)

**7.** 更新浏览器 URL（不刷新页面）：`window.history.replaceState` → `?deliverableId={id}&pageId={id}&versionId={id}`

> 任意步骤接口失败时静默退出，不影响已渲染的对比报告。

## 场景八：交付件下拉框选中新交付件

**入口**（以下两处触发相同逻辑，改造后共用 `onSelectDeliverable(d)` → `onSelectPage(page)`）：

- **上传页**：页面无报告，首次从标签栏交付件下拉框选中某交付件
- **报告页**：已有报告，从标签栏交付件下拉框切换到另一个交付件

两种入口的接口调用序列完全相同，最终都会切换到报告页模式并同步 URL。

**1. [后台(mock)] `getPagesByDeliverableId(deliverableId)`**
- 作用：获取新交付件下的页面列表，按 `createTime` 降序排列，刷新页面下拉框
- 调用处：`api.ts`
- 接口详情：[接口文档 → getPagesByDeliverableId](接口文档.md#getpagesbydeliverableid--根据交付件-id-查询页面列表)

**2.** 自动取 `pages[0]`（最新页面），更新页面下拉框选中态

**3. [后台(mock)] `getResultsByPageId(pageId, 1, 1)`**
- 作用：获取该页面的最新版本（第 1 条）
- 调用处：`api.ts`
- 接口详情：[接口文档 → getResultsByPageId](接口文档.md#getresultsbypageid--查询页面下的版本列表)

**4.** `fetchVersionJson(devJsonUrl)` + `fetchVersionJson(designJsonUrl)`（并行）
- 作用：通过版本记录中存储的 URL 取回开发侧、设计侧原始 JSON

**5. [算法(server)] `parseDevUpload` + `parseDesignUpload`**（并行）
- 作用：解析两侧节点树，得到节点列表和 canvas 信息
- 接口路径：`POST /devlint/api/parse/dev` / `POST /devlint/api/parse/design`

**6.** 前端重建 result（同【场景一·有参分支 步骤 5】），渲染新报告

**2.** `fetchVersionJson(devJsonUrl)` + `fetchVersionJson(designJsonUrl)`（并行）

**4.** 前端重建 result，渲染新报告

## 场景十：报告页"重新对比"

**入口**：报告页重新对比按钮 → `rerunCheck()`

**1. [算法(server)] `checkUpload(...)`**
- 作用：用当前已有文件重新执行对比，更新 `result`
- 调用处：`api/index.ts`
- 接口路径：`POST /devlint/api/check/upload`

**2. [后台(mock)] `addConsistencyCheckPage({ id, ... })`**（**追加模式**：传 `id`；后台静默，不阻塞主流程）
- 作用：传入当前 `workingPage` 的 `id`，mock 识别后跳过建页、直接在该页面下追加新版本记录，返回该 pageId
- 调用处：`api.ts`
- 接口详情：[接口文档 → addConsistencyCheckPage](接口文档.md#addconsistencycheckpage--新增页面--追加版本)

**3. [后台(mock)] `getResultsByPageId(pageId, 1, 999)`**
- 作用：拉取当前页面的完整版本列表，更新历史面板 `pageVersionList`，`workingVersionId` 选中第一条（最新版本）
- 调用处：`api.ts`
- 接口详情：[接口文档 → getResultsByPageId](接口文档.md#getresultsbypageid--查询页面下的版本列表)

**4.** 更新浏览器 URL：`window.history.replaceState` → `?deliverableId={id}&pageId={id}&versionId={newId}`

> 页面本身未变，不需要刷新 `getPagesByDeliverableId`。

## 场景十一：报告页"开发侧 / 设计侧重新上传"

**入口**：`recheckDev()` / `recheckDesign()` → 清空对应侧文件 → 用户重新上传 → 点击重新对比

1. 用户上传新文件时触发节点预览解析（同场景三）
2. 点击重新对比 → 进入【场景十：重新对比】流程

## 接口分类速查

| 接口函数 | 目标服务 | 调用场景 |
|---|---|---|
| `fetchCases` | 算法(server) | 初始化、切换平台、arkuiJson 变更 |
| `checkCase` | 算法(server) | 点击案例 |
| `checkUpload` | 算法(server) | 开始对比、重新对比 |
| `parseDevUpload` | 算法(server) | 开发侧文件就绪时预览 |
| `parseDesignUpload` | 算法(server) | 设计侧文件就绪时预览 |
| `getConsistencyCheckDeliverables` | 后台(mock) | 初始化（有参/无参分支），首次对比存档 |
| `getPagesByDeliverableId` | 后台(mock) | 初始化有参分支、选中交付件（场景八）、首次/重新对比存档 |
| `getResultsByPageId` | 后台(mock) | 初始化有参分支、选中交付件（场景八）、切换页面（场景九）、首次/重新对比存档 |
| `getTeamList` | 后台(mock) | 首次对比存档（第 1 步，无现有交付件时） |
| `getSonListByTeamId` | 后台(mock) | 首次对比存档（第 2 步，无现有交付件时） |
| `addConsistencyCheckDeliverable` | 后台(mock) | 首次对比存档（第 3 步，无现有交付件时） |
| `addConsistencyCheckPage` | 后台(mock) | 首次/重新对比存档（新建页面并写入版本） |

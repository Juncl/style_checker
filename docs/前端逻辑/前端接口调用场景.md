# 前端接口调用场景

> 本文以前端视角，按用户操作场景逐一列出实际发生的接口调用。
>
> **两类后端**：
> - **算法(server)** — 端口 3000，baseURL `/devlint/api`，负责节点解析 / 节点匹配 / 样式对比
> - **后台(mock)** — 端口 3001，baseURL `/mock`，负责交付件 / 页面 / 版本数据管理、登录、打点等

---

## 场景一：页面初始化

**入口**：`ConsistencyView.vue` → `onMounted`

```
1. initApp()  — 内网实现，外网空逻辑（见 init/inner/）
   ├─ checkAuth()         — 登录鉴权（外网：空）
   ├─ mountPixso()        — 挂载 Pixso 渲染器（外网：空）
   ├─ detectIframe()      — 检测 iframe 环境（外网：空）
   ├─ restorePlatform()   — 读取本地缓存，返回上次使用的平台
   ├─ trackEntry()        — 入口埋点（外网：空）
   └─ loadDeliverable()   — URL 含 deliverableId 时加载历史结果（外网：return null）

2. 分支判断：
   ├─ URL 含 deliverableId → 直接渲染报告页，跳过下方步骤
   └─ 否则 → 继续
```

**3. [算法(server)] `fetchCases(platform)`**
- 作用：获取内置案例列表，渲染左侧案例栏
- 调用处：`api/index.ts`
- 接口路径：`GET /devlint/api/cases?platform={platform}`

**同时**，`UploadPage.vue` → `onMounted`：

**4. [后台(mock)] `getConsistencyCheckDeliverables()`**
- 作用：获取当前用户的交付件列表，渲染开发侧标签栏下拉框
- 调用处：`api.ts`
- 接口详情：[接口文档 → getConsistencyCheckDeliverables](接口文档.md#getconsistencycheckdeliverables--查询用户交付件列表)

## 场景二：切换平台（hmPhone / hmWatch / web）

**入口**：`UploadPage` 平台下拉框 → `onPlatformSwitch` → `ConsistencyView.onPlatformSwitch`

**1. [算法(server)] `fetchCases(platform)`**
- 作用：重新加载当前平台的内置案例列表
- 调用处：`api/index.ts`
- 接口路径：`GET /devlint/api/cases?platform={platform}`

---

## 场景三：上传开发侧 JSON 文件

**入口**：`UploadPage` 开发侧文件拖拽/点击上传 → `onStepPicked({ type: 'arkuiJson', file })`

**1.** 前端本地解析 JSON，检测平台类型（web / hmWatch / hmPhone）

**2. [算法(server)] `fetchCases(platform)`**（仅平台发生变化时触发）
- 作用：刷新案例列表
- 调用处：`api/index.ts`
- 接口路径：`GET /devlint/api/cases?platform={platform}`

**3. [算法(server)] `parseDevUpload(arkuiJson, arkuiImage, platform)`**（开发侧 json + 图片均就绪时触发）
- 作用：解析开发侧节点树，返回 `{ nodes, canvas }`，渲染节点预览层
- 调用处：`api/index.ts`
- 接口路径：`POST /devlint/api/parse/dev`（multipart: arkuiJson, arkuiImage, platform）

**4. [算法(server)] `parseDesignUpload(designJson, designImage, platform)`**（设计侧 json + 图片均就绪时触发）
- 作用：解析设计侧节点树，渲染设计侧预览层
- 调用处：`api/index.ts`
- 接口路径：`POST /devlint/api/parse/design`（multipart: designJson, designImage, platform）

> 上传设计侧 JSON（`type: 'designJson'`）同理，仅触发步骤 4。

---

## 场景四：拖拽上传文件

**入口**：拖拽文件到上传区 → `onDrop → assignFiles`

- 两侧四个文件均已就绪 → 直接进入【场景六：对比分析】流程
- 部分就绪：
  - 开发侧有 json + 图片 → 触发 `parseDevUpload`（同场景三步骤 3）
  - 设计侧有 json + 图片 → 触发 `parseDesignUpload`（同场景三步骤 4）

---

## 场景五：点击内置案例

**入口**：案例列表点击 → `selectCase(caseId)`

**1. [算法(server)] `checkCase(caseId, platform)`**
- 作用：执行节点匹配 + 样式对比，返回完整对比结果
- 调用处：`api/index.ts`
- 接口路径：`POST /devlint/api/check/case/{caseId}?platform={platform}`

**2. [算法(server)] `imageUrl(caseId, type, platform)`** × 2（并行）
- 作用：fetch 开发侧、设计侧图片，构造 Blob URL 用于画布展示
- 调用处：`api/index.ts`
- 接口路径：`GET /devlint/api/cases/{caseId}/image/{arkui|design}?platform={platform}`

**3.** → 触发【场景七：对比完成后自动存档】

---

## 场景六：点击"开始对比"按钮

**入口**：`UploadPage` 开始对比按钮 → `runUpload(platform)`

**1. [算法(server)] `checkUpload(designJson, arkuiJson, designImage, arkuiImage, platform)`**
- 作用：执行节点匹配 + 样式对比，返回完整对比结果
- 调用处：`api/index.ts`
- 接口路径：`POST /devlint/api/check/upload`（multipart: designJson, arkuiJson, designImage, arkuiImage, platform）

**2.** → 触发【场景七：对比完成后自动存档】

---

## 场景七：对比完成后自动存档

**入口**：场景五、场景六对比成功后，后台静默执行 `submitResult()`，不阻塞主流程

**1. [后台(mock)] `getTeamList()`**
- 作用：获取第一层团队，取 `teams[0].teamId`
- 调用处：`api.ts`
- 接口详情：[接口文档 → getTeamList](接口文档.md#getteamlist--获取团队列表)

**2. [后台(mock)] `getSonListByTeamId(teamId)`**
- 作用：获取子项目列表，取 `sonTeams[0].teamId` 作为 subTeamId
- 调用处：`api.ts`
- 接口详情：[接口文档 → getSonListByTeamId](接口文档.md#getsonlistbyteamid--根据团队-id-获取子项目列表)

**3. [后台(mock)] `addConsistencyCheckDeliverable(subTeamId, name)`**
- 作用：新建一个交付件，返回 deliverableId
- 调用处：`api.ts`
- 接口详情：[接口文档 → addConsistencyCheckDeliverable](接口文档.md#addconsistencycheckdeliverable--新增交付件)

**4. [后台(mock)] `addConsistencyCheckPage(params)`**
- 作用：在该交付件下新建页面并创建首个版本，返回 pageId
- 调用处：`api.ts`
- 接口详情：[接口文档 → addConsistencyCheckPage](接口文档.md#addconsistencycheckpage--新增页面含首个版本)

**5. [后台(mock)] `getResultsByPageId(pageId, 1, 1)`**
- 作用：取该页面的第一条版本，获得 versionId
- 调用处：`api.ts`
- 接口详情：[接口文档 → getResultsByPageId](接口文档.md#getresultsbypageid--查询页面下的版本列表)

**6. [后台(mock)] `getPagesByDeliverableId(deliverableId)`**
- 作用：拉取最新页面列表，刷新标签栏页面下拉框
- 调用处：`api.ts`
- 接口详情：[接口文档 → getPagesByDeliverableId](接口文档.md#getpagesbydeliverableid--根据交付件-id-查询页面列表)

**7.** 更新浏览器 URL（不刷新页面）：`window.history.replaceState` → `?deliverableId={id}&pageId={id}&versionId={id}`

> 任意步骤接口失败时静默退出，不影响已渲染的对比报告。

---

## 场景八：报告页"重新对比"

**入口**：报告页重新对比按钮 → `rerunCheck()`

**1. [算法(server)] `checkUpload(...)`**
- 作用：用当前已有文件重新执行对比
- 调用处：`api/index.ts`
- 接口路径：`POST /devlint/api/check/upload`

> `rerunCheck` 完成后不触发 `submitResult`，不产生新的存档记录。

---

## 场景九：报告页"开发侧 / 设计侧重新上传"

**入口**：`recheckDev()` / `recheckDesign()` → 清空对应侧文件 → 用户重新上传 → 点击重新对比

1. 用户上传新文件时触发节点预览解析（同场景三）
2. 点击重新对比 → 进入【场景八：重新对比】流程

---

## 接口分类速查

| 接口函数 | 目标服务 | 调用场景 |

| `fetchCases` | 算法(server) | 初始化、切换平台、arkuiJson 变更 |

| `checkUpload` | 算法(server) | 开始对比、重新对比 |

| `parseDesignUpload` | 算法(server) | 设计侧文件就绪时预览 |
| `getConsistencyCheckDeliverables` | 后台(mock) | UploadPage 初始化 |
| `getTeamList` | 后台(mock) | 对比完成存档（第 1 步） |
| `getSonListByTeamId` | 后台(mock) | 对比完成存档（第 2 步） |
| `addConsistencyCheckDeliverable` | 后台(mock) | 对比完成存档（第 3 步） |
| `addConsistencyCheckPage` | 后台(mock) | 对比完成存档（第 4 步） |
| `getResultsByPageId` | 后台(mock) | 对比完成存档（第 5 步） |
| `getPagesByDeliverableId` | 后台(mock) | 对比完成存档（第 6 步） |

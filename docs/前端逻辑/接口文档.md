# 接口文档

> **说明**：所有接口均通过 `client/src/api/api.ts` 调用，基础路径由 `adminEnv.ts` 中的 `ADMIN_BASE_URL` 控制：外网为 `/mock`，内网为 `window.location.origin`。
>
> **通用请求头**：所有 7 个接口均需携带 `uiplusToken`（值来自 `localStorage.getItem('uiplusToken')`）。mock 环境不校验此字段，但内网后台依赖它进行鉴权，缺失会导致接口报权限错误。

## 通用响应结构

```json
{
  "content": <业务数据>,
  "errorCode": 200,
  "errorMessage": null,
  "success": true
}
```

## 三、页面（Page）与版本（Version）

### addConsistencyCheckPage — 新增页面 / 追加版本

| 项 | 值 |
|---|---|
| **方法** | POST |
| **路径** | `/main/rest.root/reviewTool/consistencyCheck/addConsistencyCheckPage` |
| **Content-Type** | `application/json` |
| **请求头** | `uiplusToken: localStorage.getItem('uiplusToken')` |
| **前端函数** | `addConsistencyCheckPage(params)` |
| **Mock** | ✅ 已实现 |

**双模式说明**

| `id` 字段 | 行为 |
|---|---|
| **不传**（新建模式） | 在指定交付件下新建页面，同时写入首个版本，返回新建页面 ID |
| **传字符串**（追加模式） | 在已有页面下追加新版本，页面本身不变，返回该页面 ID |

**入参（Body）**

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| id | string | 否 | 已有页面 ID；**传此字段时为追加模式** |
| deliverableId | string | 是 | 所属交付件 ID |
| name | string | 是 | 页面名称，格式 `page{N}`，N = 该交付件下当前页面总数 + 1（新建前先查 `getPagesByDeliverableId` 取 `length`）；列表按 createTime 降序展示，编号越大排越前 |
| deviceType | string | 是 | 设备类型，如 `hmPhone` |
| versionName | string | 是 | 版本名称（时间戳字符串） |
| devImageBase64Data | string | 是 | 开发侧图片 Base64 |
| devJson | string | 是 | 开发侧原始 JSON 文本 |
| designImageBase64Data | string | 是 | 设计侧图片 Base64 |
| designJson | string | 是 | 设计侧原始 JSON 文本 |
| problems | array | 是 | 差异列表，见下方结构 |
| nodeMatchs | string | 否 | `JSON.stringify({ matchedPairIds: [[arkuiId, designId], ...] })`，节点匹配结果独立字段 |

**problems 单项结构**

```ts
{
  id:   string  // `${arkuiNodeId}-${property}`
  key:  string  // 节点类型：'text' | 'container'
  type: string  // 属性名，如 'fontSize'
  desc: string  // 差异描述
  data: string  // JSON.stringify(完整 diff 对象)
}
```

**出参**

| 字段 | 类型 | 说明 |
|---|---|---|
| success | boolean | 是否成功 |
| errorCode | number | 200 = 成功 |
| errorMessage | string \| null | 错误信息，成功时为 null |
| content | number | 页面 ID（新建模式为新建 ID，追加模式为传入的 ID） |

> **新建模式调用链**：`addConsistencyCheckDeliverable()` → 得到 `deliverableId` → `addConsistencyCheckPage({ deliverableId, ... })` → 得到 `pageId`
>
> **追加模式调用链**（重新对比）：`addConsistencyCheckPage({ id: pageId, ... })` → `getResultsByPageId(pageId, 1, 999)` → 更新版本列表

| content.list[].createTime | number | 创建时间（Unix ms） |

**problems 单项结构（外层包装）**

```ts
{
  id:   string  // `${arkuiNodeId}-${property}`；特殊项为 'matchedPairIds'
  key:  string  // 节点类型：'text' | 'container'；特殊项为 'pairIds'
  type: string  // 属性名，如 'fontSize'；特殊项为 'obj'
  desc: string  // 差异描述
  data: string  // JSON.stringify(完整 diff 对象)，见下方
}
```

**data 字段（样式差异，JSON.parse 后的 diff 对象）**

| 字段 | 类型 | 说明 |
|---|---|---|
| property | string | 差异属性名，如 `fontSize`、`fontColor`、`backgroundColor`、`borderRadius`、`border`、`opacity`、`blur`、`shadow`、`fontWeight`、`fontFamily`、`spacing.top`、`spacing.left` |
| designValue | string \| number \| object | 设计侧属性值（原始数值、颜色字符串或结构对象） |
| arkuiValue | string \| number \| object | 开发侧属性值 |
| severity | `'error'` \| `'warning'` | 严重程度：`error` = 中/高置信匹配的差异；`warning` = 低置信匹配的差异（前端显示在"模糊比对"tab） |
| description | string | 人类可读的差异描述，如 `字号偏差 2vp（设计 14 / 开发 12）` |
| nodeType | `'text'` \| `'container'` | 节点类型 |
| textContent | string \| null | 节点文本内容（文本节点时有值，容器节点为 null） |
| designName | string | 设计侧节点名称 |
| arkuiName | string | 开发侧节点名称 |
| matchType | string | 匹配类型，如 `textExact`、`topologyAnchor`、`containerIou` 等 |
| confidence | `'high'` \| `'medium'` \| `'low'` | 匹配置信度 |
| iou | number \| null | IoU 交集比（0-1），无时为 null |
| topologyScore | number \| null | 拓扑匹配得分，无时为 null |
| regionScore | number \| null | 区域得分，无时为 null |
| designNodeId | string | 设计侧节点 ID |
| arkuiNodeId | string | 开发侧节点 ID |
| designRect | object | 设计侧节点坐标 `{ x, y, w, h }`（dp 坐标） |
| arkuiRect | object | 开发侧节点坐标 `{ x, y, w, h }`（vp 坐标） |

**data 字段（间距差异，`property` 为 `spacing.top` / `spacing.left` 时的额外字段）**

| 字段 | 类型 | 说明 |
|---|---|---|
| spaceId | string | 开发侧间距关系唯一 ID，格式 `{anchorId}-{nodeId}-{axis}` |
| designSpaceId | string | 设计侧间距关系唯一 ID，同格式 |
| relatedDesignNodeId | string | 参考锚点节点 ID（设计侧） |
| relatedArkuiNodeId | string | 参考锚点节点 ID（开发侧） |
| relatedDesignName | string | 参考锚点节点名称（设计侧） |
| relatedArkuiName | string | 参考锚点节点名称（开发侧） |
| relationKind | `'parent-child'` \| `'sibling'` | 与锚点的空间关系类型 |
| relationAxis | `'vertical'` \| `'horizontal'` | 间距方向 |
| relationRects | object | 锚点和当前节点的 rect 坐标：`{ design: [anchorRect, nodeRect], arkui: [anchorRect, nodeRect] }` |

**旧格式（存量兼容）：problems 末尾特殊项**

存量数据在 problems 数组末尾包含一条特殊记录：

```ts
{
  id:   'matchedPairIds'
  key:  'pairIds'
  type: 'obj'
  desc: ''
  data: JSON.stringify([[arkuiId, designId], ...])
}
```

**新格式：nodeMatchs 独立字段（推荐）**

与 problems 同级，JSON 字符串：

```ts
nodeMatchs = JSON.stringify({ matchedPairIds: [[arkuiId, designId], ...] })
```

新数据仅使用 `nodeMatchs`，不再往 `problems` 中写入特殊项。读取优先级：`nodeMatchs` 存在则优先使用，否则回退到从 `problems` 末尾提取旧格式数据。

> 取最新版本 ID：`content.list[0].id`（调用时 `pageSize=1` 只取第一条）

### 完整调用链（对比完成后自动存档）

```
对比完成（result 拿到）
  → getTeamList()               → teamId
  → getSonListByTeamId(teamId)  → subTeamId
```

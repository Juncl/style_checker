# 节点匹配准确度提升举措汇总

> 目标：把已经落地的节点匹配优化措施整理成一份可回顾、可复用、可继续迭代的清单。

这份文档不是算法原理介绍，而是对“我们为了提升节点匹配准确度，具体做了哪些事”的详细总结。

---

## 1. 总体思路

节点匹配准确度的提升，不是靠单一规则，而是靠一整套分层策略叠加：

1. 先减少错误候选。
2. 再提升文本、几何、视觉、拓扑信息的判别能力。
3. 最后通过一对一全局约束消解冲突。

整体上可以归纳为四个方向：

- 过滤更准
- 特征更全
- 关系更强
- 兜底更稳

---

## 2. 节点过滤层的优化

这一层的目标是把“不该参与匹配的节点”尽量提前剔除掉。

### 2.1 过滤透明节点

已经加入规则：

- `opacity = 0` 的节点不参与比较
- 设计侧和开发侧统一处理

作用：

- 避免完全不可见节点参与文本或几何匹配
- 减少“树里有，但视觉上根本不存在”的误配

### 2.2 过滤空文本节点

已经加入规则：

- 文本内容为空的节点不参与比较
- 例如 case1 顶部 title 这类空内容节点，不再进入候选池

作用：

- 避免空文本占用文本匹配资源
- 减少标题容器、占位节点对结果的干扰

### 2.3 过滤开发侧 `Blank`

已经加入规则：

- 开发侧 `Blank` 节点直接过滤
- 不可选中
- 不参与节点匹配

作用：

- 去掉开发端框架内部的空壳节点
- 避免把“无语义节点”当成正常节点配对

### 2.4 隐藏框架祖先与像素可见性联合处理

已经加入规则：

- 对隐藏框架祖先节点不再机械地一刀切
- 结合像素可见性判断是否真的可见
- 对低可见度文本增加额外门槛

作用：

- 避免因为树结构层级复杂而错杀可见节点
- 同时防止弱可见节点进入宽松救援逻辑

### 2.5 视觉遮挡过滤

已经加入规则：

- 通过视觉遮挡标记 `visualOccluded`
- 被遮挡的节点不参与匹配

作用：

- 处理层级存在但实际上看不见的节点
- 降低“被覆盖节点”和“前景节点”混配

### 2.6 图标 / 插画语义整体化

已经加入规则：

- 设计侧以 `#illustration` 开头的图层按整体处理
- 设计侧 `ic_` 开头的图标节点按整体处理
- 子节点不再拆分参与匹配

作用：

- 插画、图标通常不是靠内部细碎结构匹配
- 整体化后更稳定，能减少碎片化误配

---

## 3. 文本匹配层的优化

文本是节点匹配中最主要、也最容易出错的一层，因此做了很多细分。

### 3.1 文本精确匹配优先

已经保留并增强：

- 完全同文本优先匹配
- 先看位置、样式、可见性，再决定是否落配

作用：

- 让同文节点尽量走高置信通道
- 减少后续兜底逻辑的压力

### 3.2 文本内容空值与弱可见约束

已经加入规则：

- 空文本直接跳过
- 弱可见文本不进入宽松救援通道

作用：

- 防止视觉上几乎不可见的文字被错误匹配
- 例如 case4 中类似“查看详情 / 了解详情”的低可见文本被拉去配别的文本的问题，已经通过门槛控制压住

### 3.3 动态文本槽位

已经加入规则：

- 针对时间、星期、数值、单位、标题槽位等做动态文本匹配
- 不强求文本内容完全一致

作用：

- 解决设计稿 Mock 文案和开发真实数据不一致的问题
- 让“同槽位不同值”的节点仍可被正确识别

典型场景：

- `周三` vs `周六`
- `1 小时 24 分钟`
- `9度`
- 各类日期、时间、数值、单位类文本

### 3.4 文本类型门槛

已经加入规则：

- 时间、日期、星期、单位、数字分别按类型判断
- 低可见文本和弱救援文本必须同类型或严格接近

作用：

- 避免把不同语义类型的文本硬配在一起
- 降低 case7 这类“远距离同数字文本”的误配风险

### 3.5 文本语义匹配

已经加入规则：

- 使用语义相似度判断文本关系
- 不是只看字符串是否相等

作用：

- 解决文案存在同义、近义、业务槽位替换的情况
- 比如同风格标题的语义对齐

### 3.6 语义门槛与类型门槛并行

已经加入规则：

- 语义相似度不是唯一依据
- 需要同时满足类型一致、位置接近、样式接近等约束

作用：

- 避免“同义词”或“同风格词”被无限放宽
- 控制跨文案误配

### 3.7 短文本特殊约束

已经加入规则：

- 短单位文本、短数字文本、重复率高的短标签增加更严格的位置约束
- 对短文本启用更保守的匹配阈值

作用：

- 避免像“9°”、“0”、“周六”这种极短文本在全局范围内乱配
- 降低 case7、case5 中的远距离误配

### 3.8 属性指纹兜底

已经加入规则：

- 使用文本样式指纹作为补充特征
- 重点比较 `fontSize`、`fontWeight`、`fontColor`、`textAlign`、`letterSpacing`

作用：

- 解决内容不同但样式高度一致的文本节点
- 尤其对标题/卡片类文本很有帮助

### 3.9 文本角色匹配

已经加入规则：

- 对“标题 / 副标题 / 标签 / 说明”这类角色槽位做匹配
- 不只看文本本身，也看它在组件里的角色

作用：

- 提升模板化页面中的节点对齐能力
- 避免同区域多文本串位

### 3.10 区域内最优文本分配

已经加入规则：

- 同一个视觉区域内，对文本节点做全局最优分配
- 不是单点贪心匹配

作用：

- 解决列表项、卡片信息块中“多个文本节点互相串位”的问题
- 提升一对一分配的稳定性

### 3.11 强锚点拓扑推断

已经加入规则：

- 先用高置信文本建立强锚点
- 再根据锚点周边节点的拓扑关系推断其它节点

作用：

- 补足弱节点匹配
- 提升 case8 这类局部结构重复页面的识别率

---

## 4. 几何与布局层的优化

几何信息是文本之外的第二大核心特征，尤其对容器、图标、图片、圆点特别重要。

### 4.1 非文本节点引入宽高比规则

已经加入规则：

- 对非文本图层，宽高比可以作为匹配约束
- 宽高比差异过大的节点不配对

作用：

- 避免细长条与接近方块的节点互配
- 对图标、圆点、插画碎片很有效

### 4.2 IoU 与尺寸比例结合

已经加入规则：

- 几何匹配不只看 IoU
- 同时看宽高比例、尺寸比例、中心距离

作用：

- 防止父容器抢占子节点
- 避免大小尺度相差过大的节点误配

### 4.3 容器与图形匹配分档

已经加入规则：

- 容器、shape、image、other 分开处理
- 不同类别采用不同阈值与权重

作用：

- 让布局容器和真实图形不要混在同一规则里
- 提升几何匹配的可控性

### 4.4 圆角视觉优化

已经考虑并部分纳入：

- 圆角可能由其它图层实现
- 全圆角也要作为特殊情形处理

作用：

- 避免只看单一圆角属性，忽略真实渲染方式
- 对组件实现细节更友好

### 4.5 位置回退匹配

已经加入规则：

- 文本内容不一致时，可以在局部位置结构接近的条件下回退匹配

作用：

- 补足 Mock 数据与真实数据不一致导致的漏配
- 让“位置对得上、角色对得上”的文本仍有机会匹配

### 4.6 局部拓扑与兄弟顺序

已经加入规则：

- 兄弟节点顺序
- 父子节点关系
- 局部区域中的相对布局

作用：

- 对列表、表单、卡片等高度结构化页面很有效
- 能减少同区域内节点交换位置导致的误配

---

## 5. 视觉特征层的优化

对于图标、插画、图片、图形，只有几何信息是不够的，所以补了视觉算法。

### 5.1 图像裁剪相似度

已经加入规则：

- 先按组件或局部区域裁剪图像
- 再计算相似度

作用：

- 让图标、插画、局部图形有更可靠的视觉对齐

### 5.2 色彩直方图

已经加入规则：

- 比较图形的颜色分布

作用：

- 补足“形状类似但颜色不同”的区分能力

### 5.3 边缘 / 轮廓特征

已经加入规则：

- 使用边缘和轮廓信息做辅助判断

作用：

- 对图标、圆点、装饰图形特别有效
- 能区分外轮廓相近但视觉结构不同的节点

### 5.4 文本渲染粗细估计

已经加入规则：

- 用视觉特征辅助估计字重

作用：

- 补足 `fontWeight` 解析误差
- 减少 case6 这类样式字段异常导致的误配

### 5.5 图标与插画的整体识别

已经加入规则：

- 图标、插画不只看碎片节点
- 更强调整体视觉特征

作用：

- 避免把插画内部局部 path 当成独立图形乱配
- 让整体语义更稳定

---

## 6. 可见性与遮挡关系的优化

很多误配不是因为节点不相似，而是因为它们根本不该被看见。

### 6.1 像素可见性判定

已经加入规则：

- 用像素采样判断是否真的可见
- 对设计侧和开发侧尽量统一口径

作用：

- 减少“树里有，视觉上没有”的节点被选中

### 6.2 视觉遮挡标记

已经加入规则：

- 对被遮挡节点打上 `visualOccluded`
- 这类节点不进入匹配

作用：

- 处理前景图层覆盖后景内容的情况
- 防止不可见节点参与候选计算

### 6.3 低可见文本的强约束

已经加入规则：

- 低可见文本不能进入宽松救援
- 必须满足更严格的距离和类型条件

作用：

- 解决 case7 中“现在”、“9度”这类不可见文本被远距离错误匹配的问题

---

## 7. 一对一与全局最优约束

准确度提升不只是“多匹配”，还要避免“一对多、多对一”。

### 7.1 全局一对一

已经加入规则：

- 节点匹配只允许一对一
- 不允许多个开发节点同时吃掉同一个设计节点

作用：

- 减少冲突和重复匹配
- 让最终结果更符合人工审查习惯

### 7.2 按置信度消解冲突

已经加入规则：

- 优先保留高置信候选
- 低置信候选让位

作用：

- 防止低质量兜底覆盖高质量主匹配

### 7.3 小组精确分配

已经加入规则：

- 小规模候选集采用精确分配
- 避免局部贪心导致整体次优

作用：

- 提升重复文本场景下的整体最优性

---

## 8. 间距与对齐的专项优化

间距与对齐本质是“两个节点之间的关系”，不是单节点属性。

### 8.1 优先兄弟节点与父子节点关系

已经加入规则：

- 先看同父兄弟节点
- 再看父子关系
- 再看局部拓扑

作用：

- 间距和对齐更容易准确识别
- 减少跨层错误比对

### 8.2 间距问题双节点高亮

已经支持：

- 间距问题不是只高亮一个节点
- 会重点显示两个节点之间的间距关系

作用：

- 更符合人工排查习惯
- 更容易定位布局偏差发生在哪里

---

## 9. 交互与树选择层的优化

准确度提升不仅在算法内部，也在交互入口上做了修正。

### 9.1 节点树选中与图层选中联动

已经加入规则：

- 节点树选中后可以联动图层
- 开发 / 设计树切换后能继续正常定位

作用：

- 避免用户选错节点入口
- 减少“树对了，但图里没对上”的体验问题

### 9.2 选中逻辑下钻

已经加入规则：

- 对容器包裹的文本节点做下钻选择

作用：

- 解决像“我的”这类标题节点被容器层吞掉的问题
- 让可选节点更接近最终参与匹配的节点

---

## 10. 回归验证机制

为了保证准确度提升不是“某个 case 好了、其它 case 崩了”，每轮修改后都做了回归验证。

### 10.1 Golden 验证

已经持续使用：

- `npm run golden`

作用：

- 检查 case 级别的 recall / precision
- 及时发现回归

### 10.2 Stats 验证

已经持续使用：

- `npm run stats`

作用：

- 观察匹配数量、误配数量、弱可见文本命中情况
- 监控一对一约束是否被破坏

### 10.3 定点 case 回归

已经持续使用：

- case4、case7、case8、case1、case2 等典型 case 的定点排查

作用：

- 验证每条规则具体修复了什么
- 防止新增规则带来局部副作用

---

## 11. 当前最有效的提升点总结

如果把这些举措按“对准确度的实际贡献”来归类，最有效的几类大致是：

1. 节点过滤更严格，先排除空文本、透明节点、Blank、弱可见节点。
2. 文本匹配更细分，把时间、星期、数字、单位、标题等拆开处理。
3. 局部拓扑更重要，强锚点 + 周边节点推断明显提升重复页面表现。
4. 非文本节点不能只看文本，要结合宽高比、IoU、视觉特征。
5. 一对一全局约束非常关键，能把误配控制在可接受范围内。
6. 每轮修改都要跑 golden / stats，否则很容易把一个 case 修好、另一个 case 弄坏。

---

## 12. 结语

节点匹配准确度的提升，本质上是把“可见性、语义、几何、视觉、拓扑”一起纳入判断。

我们已经做的事情，不是单点修补，而是把匹配系统从“单规则判断”推进到“分层决策 + 全局约束”的状态。

后续如果继续迭代，最值得继续加深的方向是：

- 更精细的遮挡判定
- 更强的局部拓扑网络
- 更细的文本角色识别
- 更完整的图像视觉特征
- 更稳定的间距与对齐关系输出

## 13. 语义特征的实现逻辑

语义特征主要是给文本节点做“内容理解”，让系统不只看字符串是否相等，还能判断它们是不是同一类信息、同一业务槽位、同一层级角色。

### 13.1 语义特征的入口

语义逻辑主要集中在 `server/src/matchers/textSemantics.js`，并被这些位置复用：

- `server/src/matchers/matchStrategies.js`
- `server/src/matchers/nodeMatcher.js`
- `server/src/matchers/dynamicTextSlots.js`
- `server/src/matchers/nodeVisibility.js`

也就是说，语义不是单独一个函数，而是一组被多处调用的判定器和打分器。

### 13.2 文本先归一化

所有语义判断的第一步，都是先做文本归一化：

- 去掉首尾空格
- 转成小写
- 统一比较口径

对应实现是 `normalizeText(text)`。

作用：

- 避免大小写、空白符导致的假不一致
- 给后续的类型识别和相似度计算提供稳定输入

### 13.3 文本类型识别

系统会先把文本分成不同语义类型，核心入口是 `textFieldType(text)`。

当前主要类型包括：

- `time`
- `weekday`
- `duration`
- `percent`
- `temperature`
- `number`
- `date`
- `label`

这个分类决定了后续是不是允许匹配、能不能做位置回退、要不要加严阈值。

例如：

- `现在 12:00` 会被判成时间类
- `周三` 会被判成星期类
- `9度` 会被判成温度类
- `1 小时 24 分钟` 会被判成时长类

### 13.4 数字与单位的专门处理

数字类文本不是简单地按字符串比对，而是先做数值归一化。

对应实现包括：

- `normalizeNumericText(text)`
- `numericTextCompatible(a, b)`

它会提取：

- 前缀
- 数值
- 单位

再判断是否真正兼容。

作用：

- 避免 `9`、`09`、`9°`、`9分钟` 这类文本被粗暴视为同一个东西
- 对短数字、短单位文本更稳

### 13.5 语义相似度怎么计算

核心入口是 `textSemanticSimilarity(a, b)`。

它不是一个单一模型，而是多个轻量规则的组合：

1. 如果两边完全相同，直接返回 1。
2. 如果属于同类型的非 label 文本，给一个类型分。
3. 再计算字符重叠、双字串重叠、中文前缀重叠。
4. 取多个分数里的较高组合结果作为最终语义相似度。

这意味着语义相似度更偏“规则化近似”，不是纯粹的大模型语义推理。

### 13.6 语义门槛怎么拦误配

语义分数不会单独决定匹配，它还要经过门槛函数：

- `passesTextSemanticGate(a, b, semantic)`

这个门槛会检查：

- 两边是否为空
- 是否完全相同
- 是否属于同一类型
- 如果是 `number`，是否数值兼容
- 如果是短中文标签，语义分是否足够高

作用：

- 防止“看起来像”但其实不是同一类信息的节点被配到一起
- 例如短文本、标题、单位、数值这些高风险文本会被更严格拦截

### 13.7 位置回退允许条件

当文本语义不够强时，系统不会直接放行，而是走 `allowsTextPositionFallback(a, b, semantic)`。

这一步会额外检查：

- 是否是单位文本
- 是否包含数字信号
- 是否仍然属于同类文本
- 如果是较长的说明性文本，才允许用位置关系补救

作用：

- 让语义不完全一致的文本有条件地靠位置补回来
- 但不会让短数字、单位、时间类文本无限放宽

### 13.8 文本角色语义

除了内容类型，系统还会给文本打“角色”标签，主要入口是 `textRoleMatchScore(dn, an)` 和 `inferTextRole(node)`。

文本角色大致分为：

- `page-title`
- `title`
- `subtitle`
- `body`

判断依据主要包括：

- 字号
- 字重
- 位置是否靠页面顶部
- 是否属于标题类文本

作用：

- 让“标题对标题、副标题对副标题”这类结构更稳定
- 避免标题和正文互相乱配

### 13.9 强标题槽位判定

`isStrongTitleSlotMatch(dn, an, score)` 用来处理标题槽位。

它会同时要求：

- 两边都是标题类角色
- 分数足够高
- 位置足够接近
- 高度比例合理
- 样式足够像

作用：

- 提高页面顶部、卡片主标题这类高价值文本的匹配稳定性

### 13.10 语义在各条匹配链路里的作用

语义特征不是只在一个地方用，而是贯穿整个匹配流程：

- 文本精确匹配：同文优先，但会参考语义类型和弱可见门槛。
- 动态文本槽位：依赖文本类型和语义角色判断。
- 区域最优匹配：用语义帮助局部文本做全局分配。
- 文本位置回退：语义较弱时，允许用位置和样式补救。
- 局部拓扑：锚点周边的文本关系也会参考语义门槛。
- 可见性过滤：弱可见文本在救援阶段会更依赖语义和类型约束。

### 13.11 这套语义方案的特点

当前语义实现的特点可以概括为：

1. 不是单一大模型，而是多层规则组合。
2. 先分类，再比较，再过门槛。
3. 对短文本、数字、单位、时间等高风险文本更保守。
4. 对标题、副标题、说明类文本会引入角色概念。
5. 语义永远不是唯一依据，必须和位置、样式、拓扑一起用。

### 13.12 语义特征带来的实际收益

这一套语义逻辑主要提升了几类场景的准确度：

- 动态文案场景下的槽位对齐
- 时间、星期、数字、单位类文本的稳定匹配
- 标题 / 副标题 / 说明文本的角色识别
- 弱可见文本的误救援抑制
- 短标签在重复页面中的串位控制

### 13.13 结论

语义特征的本质，是先把文本从“字符串”变成“类型化的信息单元”，再在这个基础上做相似度、门槛和角色判断。

它不是替代几何和视觉，而是和它们一起组成文本匹配的第一层理解能力。

## 14. 节点匹配的权重机制

节点匹配不是单一分数决定结果，而是“硬门槛 + 软打分 + 全局收口”三层一起工作。

### 14.1 先分清两类权重

当前权重主要分成两类：

1. **候选生成阶段的软权重**
   - 作用：给候选 pair 打分，判断它“像不像”
   - 位置：`matchStrategies.js`、`dynamicTextSlots.js`、`regionContext.js`、`textSemantics.js`

2. **最终选边阶段的排序权重**
   - 作用：当多个 pair 冲突时，决定谁先被保留
   - 位置：`nodeMatcher.js` 的 `pairPriority()`

这两类权重解决的问题不一样：

- 候选阶段解决“能不能进池子”
- 收口阶段解决“最终保留谁”

### 14.2 候选阶段的硬门槛先行

很多规则不是加分，而是直接拦截。

例如：

- `opacity = 0` 直接排除
- 空文本直接排除
- `Blank` 直接排除
- 弱可见文本不能进入宽松救援
- 短数字、单位、时间等必须满足更严格的位置约束
- 语义类型不对时，直接不给候选机会

这些都属于“门槛型规则”，它们比权重更优先。

### 14.3 文本精确匹配的权重

文本精确匹配是最高优先级的文本通道。

它的权重主要来自：

- 同文本命中
- 位置接近
- 样式接近
- 可见性合理

具体在 `bestExactTextCandidate()` 里，大致由这些分量组成：

- `yScore`：纵向位置接近度
- `xScore`：横向位置接近度
- `styleScore`：字体、字重、字色、对齐
- `sizeScore`：高度比例
- `pixelScore`：像素可见性

这类打分的目的不是“找最像的所有文本”，而是“从同文本候选里找最稳的那个”。

### 14.4 动态文本槽位的权重

动态槽位用于时间、星期这类“文案会变，但角色不变”的节点。

`dynamicTextSlots.js` 里的 `slotScore()` 主要看：

- `xScore`
- `yScore`
- `orderScore`
- `style`
- `hRatio`
- `regionScore`
- `lineScore`

对应权重是：

- 横向位置 `0.24`
- 纵向位置 `0.18`
- 顺序关系 `0.24`
- 样式相似 `0.16`
- 高度比例 `0.10`
- 区域亲和力 `0.04`
- 同行组补偿 `0.04`

这个结构说明：动态槽位不只看文本语义，更看“它在布局里的角色位置”。

### 14.5 区域内最优匹配的权重

`matchRegionTextOptimal()` 会先按区域做局部最优分配，再做全局救援。

这里的核心不是某一个单点权重，而是综合：

- 语义相似度
- 文本角色
- 样式
- 位置
- 区域得分

它的特点是：

- 同一区域内优先
- 局部图内做最优分配
- 区域得分会给匹配整体加成

### 14.6 局部拓扑的权重

强锚点周边的拓扑推断，是整套方案里最典型的“多维加权”。

`topologyMatchScore()` 的分量大致是：

- 相对关系分 `relScore * 0.38`
- 尺寸分 `sizeScore * 0.18`
- 类型分 `typeScore * 0.10`
- 文本样式分 `textScore * 0.10`
- 语义/角色分 `semanticScore * 0.14`
- IoU `* 0.06`
- 区域亲和力 `* 0.04`

这说明局部拓扑的核心权重是“相对位置关系”，其他维度都是辅助约束。

### 14.7 图像 / 几何匹配的权重

对于非文本节点，权重主要来自：

- IoU
- 宽高比例
- 中心距离
- 区域加成
- 视觉相似度

其中 `bestVisualIoUMatch()` 会把几何和视觉一起算：

- 几何基础分：`iou * wRatio * hRatio`
- 视觉权重：`visualWeight`
- 区域加成：`regionBonus`

如果目标是 image 节点，视觉权重会更高；如果不是，则几何占比更大。

### 14.8 最终一对一收口的权重

候选都出来以后，还要做全局排序，防止冲突。

`pairPriority()` 里的最终排序权重由这些部分组成：

- `confidenceScore`
- `anchorScore`
- `typeScore`
- `topologyScore`
- `iouScore`
- `visualScore`

其中：

- `confidenceScore`：高 / 中 / 低，分别给更高基础分
- `anchorScore`：是否是锚点
- `typeScore`：匹配类型优先级
- `topologyScore`：局部拓扑分
- `iouScore`：几何重合度
- `visualScore`：视觉相似度

### 14.9 匹配类型优先级

`matchTypePriority()` 里，匹配类型本身也有权重排序。

大致优先级从高到低是：

1. `text-content`
2. `region-text-optimal`
3. `text-role`
4. `anchor-topology`
5. `dynamic-text-slot`
6. `text-row-slot`
7. `text-position`
8. `container-iou`
9. `gradient-iou`
10. `image-geometry`
11. `shape-geometry`
12. `other-geometry`
13. `region-text-global-rescue`
14. `rescue-iou`

这表示系统天然更信任“文本精确 / 区域最优 / 角色匹配”，再到拓扑和几何兜底。

### 14.10 为什么要这样分层

这种权重结构的好处是：

- 高置信规则不会被低置信规则冲掉
- 文本、拓扑、几何、视觉各司其职
- 误配能在早期被门槛拦住
- 低置信补洞不会破坏全局结果

### 14.11 权重机制的实践结论

从实际调优经验看，权重里最重要的不是“数字本身”，而是它们的层级关系：

1. 先有硬门槛，决定能不能进候选池。
2. 再有软权重，决定谁更像。
3. 最后有全局排序，决定谁留下。

这三层合起来，才是当前节点匹配准确度提升的核心。

## 15. 容错兜底举措

容错兜底的目标不是“尽量都配上”，而是“在不破坏主规则的前提下，把可恢复的漏配补回来，把高风险误配压住”。

### 15.1 入口兜底：先保证节点能被正确进入候选池

这类兜底发生在最前面，属于“能不能参与匹配”的修正。

已经做的事包括：

- 过滤空文本，避免空节点占用候选位
- 过滤 `opacity = 0` 节点，避免完全不可见节点参与比较
- 过滤开发侧 `Blank`，既不参与匹配，也不允许被点选
- 对 `#illustration`、`ic_` 图层做整体化处理，避免拆成一堆碎片
- 对设计侧和开发侧都统一可见性口径，避免一边宽松一边严格

这一层兜底的意义是：先把明显不该进池子的节点清出去，减少后续补救的复杂度。

### 15.2 可见性兜底：弱可见节点不直接排死，但要加严

有些节点不是完全不可见，而是“可见性很低”。

当前做法不是直接删掉，而是：

- 先用像素可见性判断是否真的可见
- 再看 `textStrokeScore` 是否还能支持识别
- 弱可见文本进入救援通道时，必须满足更严格的类型和位置条件

例如：

- `visiblePixelRatio < 0.10`
- `textStrokeScore < 0.08`

这类文本就会被视为弱可见文本。

弱可见文本的兜底原则：

- 不允许靠纯语义硬拉
- 不允许远距离匹配
- 不允许跨类型模糊匹配
- 只有非常接近、非常同类的候选才允许保留

这一步主要是为了压住 case7 这类“视觉不可见但仍然被配上”的误配。

### 15.3 语义兜底：允许同槽位不同文案，但不放开到无限宽

语义兜底主要服务这几类场景：

- 时间
- 星期
- 数字
- 单位
- 标题 / 副标题 / 说明的角色互补

已经做的语义兜底包括：

- `textFieldType()` 先把文本分类型
- `textSemanticSimilarity()` 允许同类文本有一定语义相似度
- `passesTextSemanticGate()` 控制短标签、数字、单位等高风险文本
- `allowsTextPositionFallback()` 在语义不够强时，允许用位置回退补救，但会排除单位和数字噪声

这类兜底的核心是：

1. 先同类型。
2. 再看语义分。
3. 语义不够时才考虑位置回退。

它能恢复很多“文案不同但槽位一样”的情况，但不会把完全无关的文本当成同义项。

### 15.4 动态槽位兜底：针对时间、星期这类固定结构补洞

动态槽位是一种非常实用的兜底。

典型场景：

- `周三 -> 周六`
- `现在 -> 中午 12:00`
- `9° -> 0°`

它的做法是：

- 先识别文本类型
- 只在固定槽位类型内做匹配
- 用顺序、位置、样式、区域亲和力一起算

它不是放开语义，而是把“会变的值”限制在固定结构里处理。

### 15.5 区域文本兜底：先局部，再全局

区域文本是另一种重要兜底。

当前策略是：

- 先在区域内做最优匹配
- 如果局部区域没有结果，再做全局救援
- 全局救援也必须通过语义和位置门槛

这样做的好处是：

- 卡片、列表、模块化区域更稳
- 结构重复页面不容易串位
- 漏配补救不会一下子跑到别的模块里去

### 15.6 强锚点兜底：用已确认节点去拉周边节点

这是当前最重要的拓扑兜底之一。

做法是：

- 先用高置信 pair 建立强锚点
- 再根据锚点周边的相对关系去推断弱节点
- 如果拓扑分足够高，就把它提升为可接受 pair

这个兜底特别适合：

- 多行列表
- 同一张卡片里的辅助字段
- 左右对齐、上下对齐的固定模板块

它的优势在于：

- 不用完全依赖文本相似
- 只要局部结构稳定，就能把周边节点补出来

### 15.7 几何兜底：用 IoU、尺寸和宽高比补最后一口气

当文本和语义都不足时，几何兜底会接管一部分工作。

已经做的几何兜底包括：

- 容器 IoU 匹配
- 渐变节点 IoU 匹配
- image / shape / other 的几何匹配
- 非文本节点的宽高比约束
- 中心距离和尺寸比例控制

几何兜底的原则是：

- 小节点不能乱配大节点
- 结构相近但尺寸差太大的节点不能放行
- 图标、圆点、插画片段必须有明显几何相似性

### 15.8 视觉兜底：图像特征补样式缺口

对于图标、插画、图片、圆角形状，单纯几何往往不够，所以加了视觉兜底。

包括：

- 裁剪图像相似度
- 色彩直方图
- 边缘 / 轮廓特征
- 文本渲染粗细估计

它的作用是补足：

- `fontWeight` 解析偏差
- 图标资源差异
- 插画整体外观差异
- 形状轮廓相近但细节不同

### 15.9 最终收口兜底：冲突时优先保高置信

兜底补出来的候选，最后还要回到全局收口。

已经做的收口控制包括：

- `confidenceScore` 高于低
- `isAnchor` 高于普通 pair
- 高优先级 `matchType` 优先保留
- 拓扑分高的 pair 优先保留
- IoU 和视觉分作为补充

这样可以保证：

- 兜底不会压过主匹配
- 补救结果不会把全局结果带偏

### 15.10 容错兜底的边界

容错兜底不是越多越好，边界要很清楚：

- 弱可见不等于可放开
- 同风格不等于同语义
- 位置接近不等于能匹配
- 同槽位不等于同节点

所以兜底一定要和门槛、权重、收口一起用，不能单独启用。

### 15.11 现在这套兜底的整体效果

从调优经验看，当前兜底比较有效的地方是：

- 修复漏配
- 处理动态文案
- 恢复局部重复布局中的弱节点
- 补图标和插画类节点
- 压住不可见文本误配

它的定位很明确：

> 主链路负责准确，兜底链路负责恢复，但恢复必须有边界。

## 16. 算法修改后的回归验证机制

每次改节点匹配算法后，回归验证不是只看“能不能跑”，而是要同时确认：

- 匹配率有没有提升
- 误配有没有下降
- 一对一约束有没有被破坏
- 兜底规则有没有把不该配的节点带进来
- 某个 case 的修复有没有顺手伤到别的 case

### 16.1 Golden 回归验证

这是最基础的一层验证。

当前使用：

- `npm run golden`

它会读取 `case/golden-pairs.json`，逐个 case 对比：

- 期望 pair 是否命中
- forbidden pair 是否被错误配出
- recall / precision 是否达标

这个验证最适合判断：

- 主匹配链路有没有回退
- 新增规则有没有冲掉既有正确配对
- 某些 case 是否出现新的漏配或误配

### 16.2 Stats 回归验证

当前使用：

- `npm run stats`

它会统计每个 case 的：

- 总节点数
- 可比较节点数
- pair 数量
- 文本 pair 数量
- 低置信 pair 数量
- 弱可见文本命中情况
- 不同 `matchType` 的分布

这个验证更适合看整体趋势，例如：

- 兜底是不是过多
- 某类 matchType 是否突然暴涨
- 文本对是不是跑偏成非文本对
- 误配是否被新的规则放大

### 16.3 双方向验证

当前不仅看默认方向，还会看两种匹配方向：

- `arkui-first`
- `design-first`

相关入口在：

- `server/scripts/goldenEval.js`
- `server/scripts/matchStats.js`

这样做的原因是：

- 某些规则在设计侧当主源时更稳
- 某些规则在开发侧当主源时更稳
- 双方向都过，才能证明规则比较通用

### 16.4 定点 case 回归

除了全量 golden 和 stats，还会对典型 case 做定点回归。

常见检查对象包括：

- case4：小圆点、卡片文本、弱可见文本
- case5：短数字、速度值、道路信息
- case7：天气卡片、远距离数字、不可见文本
- case8：生活指数、语义兜底、强锚点拓扑
- case1 / case2：文本精确、标题选中、容器过滤

定点回归的好处是：

- 能快速判断某条规则到底修了什么
- 能看到具体节点层面的变化
- 能避免只看全局指标看不出局部错误

### 16.5 一对一约束检查

算法修改后，必须检查一对一是否仍然成立。

现在的核心检查点包括：

- 一个设计节点不能同时对应多个开发节点
- 一个开发节点不能同时对应多个设计节点
- 低置信兜底不能抢占高置信主匹配
- 局部拓扑结果不能把已经占用的节点再次拉进来

这个检查非常关键，因为很多回归不是“没匹配上”，而是“匹配太多了”。

### 16.6 差异检查

`server/src/comparators/styleComparator.js` 的 `compareAll(pairs)` 会对最终 pair 做属性差异检查。

它能帮助回归验证看清楚：

- 匹配对是不是虽然匹上了，但样式差异过大
- `fontSize / fontWeight / fontColor / borderRadius / padding / opacity` 是否明显异常
- 圆角、背景色、字重等关键维度是不是因为错误匹配而被污染

这一步的价值是：不仅看“配没配对”，还看“配对是否合理”。

### 16.7 图像与可见性回归

因为当前方案里引入了像素可见性和视觉特征，所以回归也要看这类特征有没有失真。

重点检查：

- `annotatePixelVisibility()` 的可见性判断是否正常
- 弱可见文本是否被错误放行
- `visualOccluded` 是否把该保留的节点误杀
- 图标 / 插画 / 圆点类节点是否因为视觉特征调整而跑偏

### 16.8 回归判定顺序

实际回归时，建议按这个顺序看：

1. 先看 golden 是否通过
2. 再看 stats 里是否有异常增长的 matchType 或 low-confidence
3. 再看定点 case 是否修复了目标问题
4. 最后看 compareAll 的差异项是否合理

这样能避免只盯着一个指标，忽略了另一层回归。

### 16.9 回归验证的结论

节点匹配算法的修改，不能只靠人工看一两个例子。

当前比较稳的做法是：

- golden 保底正确性
- stats 看整体趋势
- 定点 case 看局部行为
- compareAll 看配对后果
- 一对一检查看冲突控制
- 双方向验证看通用性

这几层一起上，才算一次比较完整的回归验证。
